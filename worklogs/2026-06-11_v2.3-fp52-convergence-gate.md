## 关键决策

- **判定靠"staged vs HEAD 的未收束完成数比对"**：仅当本次提交把"编号 > last_converged_fp 且 ✅"的功能点数**推过阈值**（staged > threshold 且 staged > head）才拦。→ **非交付提交（不新增完成项）永不被拦，仓库不被冻结**——这是避免"闸门一响全仓瘫痪"的关键设计。
- **机器标记放 STATUS.md**：`last_converged_fp` = 上次收束覆盖到的最大功能点编号；人触发收束后上调它（+收束历史加行）即"释放"闸门。标记从 staged STATUS 读，让收束提交在同次生效。
- **阈值默认 3**（Owner 决策1）；标记从 #46 起（上次 v2.1 收束覆盖 #36-#46）。
- **fail-open**：标记缺失 / 数据不可读 → 放行 + WARN，不因闸门自身问题卡死提交。
- **本交付提交一次性 `[skip-gate]`**：标 #52 ✅ 使未收束完成数达 4>3，闸门恰在同次生效边界；建闸门自身的提交豁免一次，已登记 §三。
- **范围裁剪**：设计阶段C 的 `check_doc_sync.py`（四件套硬同步、分粒度硬/warn）拆为后续，本功能点只交付收束闸门。

## 遇到的问题

- 验证 block 路径时用临时探针改 开发清单 + `git add`，随后 `git checkout -- file` 把工作区从 index 还原成了"带探针的暂存版"，残留了临时编辑 → 用 `git checkout HEAD -- 开发清单.md` 从 HEAD 干净还原（该文件本无合法未提交改动）。教训：探针式 `git add` 后应 `git checkout HEAD -- <file>` 而非 `git checkout -- <file>`。

# ADR-0008: ECC 全项目对标与事务化初始化

> 日期: 2026-07-11
> 状态: 已采纳

## 背景

DevGuard 已有可测治理与显式脚手架，但 ECC 借鉴局限于全局 Hook 兼容，Codex 缺 canonical skill/AGENTS/config，强制覆盖也没有失败回滚。

## 决策

采用“机制对标而非资产镜像”：以 10 个可执行能力域覆盖 ECC 的 skills、rules、hooks、installer/profile、Codex、安全、CI/TDD、provenance 与量化治理。core profile 同时服务 Codex/Claude；写入使用唯一临时文件、fsync、原子替换和事务回滚；多代理和 MCP 保持 opt-in。

## 考虑的替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 复制 ECC 全部 agents/skills/commands | 能力数量最多 | 上下文膨胀、规则重复、维护成本不可控 |
| 只继续兼容 ECC Hook | 改动最小 | 无 Codex 工作流入口、无安装预演/回滚 |
| 十域机制对标 + 最小 skill | 可测、跨 Harness、保持单一真源 | 需要维护对标脚本与 E2E |

## 后果

- 正面：一条命令生成可独立运行的 Codex/Claude 基线，密钥扫描不依赖外部环境，失败不会留下半写文件。
- 负面：core manifest 从 18 增至 22 个文件；全量 pre-commit 首次运行需要下载 gitleaks 环境。
- 约束：不生成子代理或 legacy commands；MCP/凭据不写入项目模板；外部远程分支删除仍需真实 Git 状态验证。

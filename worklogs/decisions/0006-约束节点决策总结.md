# ADR 0006: 仅在约束节点通过 worklog 整理并进行决策总结

> 日期: 2026-05-27
> 状态: 已采纳

## 背景

此前 `docs/decisions/` 作为 ADR 的独立存放目录，与 `worklogs/` 分属两条文档流。实践中发现：

1. **决策碎片化**：开发过程中随时记录 ADR 导致决策分散，缺乏阶段性审视
2. **与工作流脱节**：ADR 独立于 worklog 存在，阅读者难以追溯"这个决策是在什么背景下做出的"
3. **过早形式化**：单个功能点内的取舍被过早提升为 ADR，但实际影响范围有限，不值得永久存档
4. **重复记录**：worklog 的"关键决策"段与 ADR 内容大量重叠

## 决策

**决策记录统一纳入 worklog 体系**：

1. **`decisions/` 移入 `worklogs/decisions/`**：ADR 文件不再是独立文档流，而是 worklog 体系的子集
2. **仅在约束节点（收束节点）时整理决策**：日常开发中，技术取舍记录在对应功能点 worklog 的"关键决策"段即可。收束节点执行时，回顾本批次所有 worklog，将值得长期保留的决策提升为 ADR（放入 `worklogs/decisions/`）
3. **ADR 编号与收束节点对齐**：新 ADR 的编号紧随上次收束节点的最后一个 ADR，确保决策历史可线性追溯

**理由**：
- **决策质量更高**：经过一批功能点的验证后再判断"这真的值得成为 ADR 吗"，比即时判断更准确
- **减少噪音**：大量微决策留在 worklog 即可，只有跨批次、影响架构方向的决策才需 ADR
- **可追溯性增强**：读者可从 worklog → decisions 完整还原决策链：why + when + what context
- **工作流统一**：worklog 和 ADR 不再是两条并行的文档流，减少维护负担

## 后果

### 操作变更
- 日常开发：技术取舍只记录在 worklog 的"关键决策"段（可选段）
- 收束节点：回顾本批次 worklog → 判断哪些值得提升为 ADR → 写入 `worklogs/decisions/`
- ADR 不再接受"即时创建"，必须是收束节点的产出物

### 文件变更
- `decisions/` 从 `docs/` 移入 `worklogs/`
- `FILE_GRAPH.md` 决策树更新：ADR 落点从 `docs/decisions/` → `worklogs/decisions/`
- `CLAUDE.md` 目录结构同步更新
- 收束节点流程（`06-第三步_收束节点.md`）补充"决策整理"环节

### 历史 ADR
- 已有 ADR 0001-0005 随文件夹一并迁移，内容不变，仅路径变更

# 测试规范

---

## 一、测试金字塔

```
         /\
        /E2E\       少量，验证核心流程
       /------\
      /  集成  \     中量，验证模块协作
     /----------\
    /   单元测试  \   大量，验证单一功能
   /--------------\
```

### 理想比例
- 单元测试：70%
- 集成测试：20%
- E2E 测试：10%

---

## 二、测试原则

### 2.1 FIRST 原则
- **F**ast：快速执行，秒级
- **I**ndependent：独立运行，不依赖顺序
- **R**epeatable：可重复，不受环境影响
- **S**elf-validating：自动判断通过/失败
- **T**imely：及时编写，最晚与代码同步

### 2.2 什么需要测
- 核心业务逻辑（必须）
- 复杂算法（必须）
- 边界条件（必须）
- 异常路径（建议）
- 简单 getter/setter（可选）

### 2.3 什么不需要测
- 第三方库的功能
- 框架的基础能力
- 纯配置文件
- 自动生成的代码

## 三、单元测试

### 3.1 结构 AAA 模式
```
Arrange   # 准备数据
Act       # 执行操作
Assert    # 验证结果
```

### 3.2 示例
```python
def test_calculate_discount_vip_user():
    # Arrange
    user = User(role="vip", level=3)
    order = Order(amount=1000)

    # Act
    result = calculate_discount(user, order)

    # Assert
    assert result == 200  # VIP 3级打8折
```

### 3.3 命名规范
```
test_<被测方法>_<场景>_<期望>
test_calculate_discount_vip_user_returns_20_percent_off
```

## 四、Mock 策略

### 4.1 需要 Mock 的
- 外部 API 调用
- 数据库操作（单元测试中）
- 文件系统操作
- 时间依赖（`datetime.now()`）
- 随机函数

### 4.2 不需要 Mock 的
- 纯函数
- 简单的数据结构
- 被测函数本身

### 4.3 Mock 原则
- 只 Mock 自己的依赖，不 Mock 别人的
- Mock 边界明确，不过度 Mock
- 验证 Mock 的调用次数和参数

## 五、测试覆盖率

### 5.1 目标
| 项目阶段 | 最低覆盖率 |
|---------|-----------|
| 快速原型 | 不作要求 |
| MVP | 60% |
| 正式产品 | 80% |
| 核心模块 | 90% |

### 5.2 覆盖率含义
- 高覆盖率不代表高质量
- 关注未覆盖的代码是否重要
- 覆盖率是下限，不是上限

## 六、常用断言

```python
# 相等
assert result == expected

# 异常
with pytest.raises(ValueError):
    divide(1, 0)

# 近似（浮点数）
assert abs(result - 3.14) < 0.01

# True/False
assert user.is_active
assert not order.is_expired()

# 包含
assert "error" not in response
assert user_id in user_ids

# 类型
assert isinstance(result, User)
```

## 七、CI 集成

### 7.1 提交时（pre-commit）
- 运行单元测试（< 30 秒）
- 不通过则阻止提交

### 7.2 PR 时
- 运行全部单元测试 + 集成测试
- 覆盖率检查
- 不通过则阻止合并

### 7.3 发布前
- 运行全部测试（含 E2E）
- 性能测试
- 安全扫描

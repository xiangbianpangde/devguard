# 07 · 验收交付

> 定义三级验收标准（功能点级 / 收束节点级 / 项目级）+ 交付物清单 + 文件放置速查 + 红线速查。
> 功能点完成、收束节点触发、项目交付时必读本篇。

---

## 一、三级验收体系

```
功能点级验收（每功能点）
    │
    ▼
收束节点级验收（每 3 功能点）
    │
    ▼
项目级验收（项目交付）
```

---

## 二、功能点级验收

### 2.1 Checklist

每完成一个功能点，必须逐项确认：

- [ ] **代码通过所有层级测试**：单元 + 集成 + E2E（如适用）
- [ ] **真浏览器截图验证通过**：不接受 HTTP 200 / tsc 零错误作为证据
- [ ] **代码位置标注清晰**：每个改动点标注 `文件路径:行号`
- [ ] **设计.md 验收标准逐项勾验**：对照 §七验收标准逐条确认
- [ ] **commit 已推送**：Conventional Commits 格式，引用 worklog 文件
- [ ] **worklog 已更新**：`worklogs/YYYY-MM-DD_<主题>.md`
- [ ] **STATUS.md 进度已同步**：功能点状态更新
- [ ] **验收报告已产出**：对话内联汇报（功能点级默认不落盘）

### 2.2 验收报告格式

按 [04-长程开发 §五](04-长程开发.md) 的 8 要素（截图 / 代码位置 / 可视化 / Done&NotDone / Design清单 / Commit&Worklog / STATUS / 交接）。

---

## 三、收束节点级验收

### 3.1 触发条件

- 预设闸门到达（默认每 3 个功能点）
- 思考任务完成（一轮完整设计结束）
- 人手动触发

### 3.2 五件交付物

| # | 交付物 | 位置 | 格式 |
|---|--------|------|------|
| 1 | 收束报告 | `docs/reports/` | HTML（含 Mermaid 图表、折叠区、锚点导航） |
| 2 | ADR（如有架构决策） | `worklogs/decisions/NNNN-<标题>.md` | Markdown |
| 3 | 更新后的 STATUS.md | 项目根目录 | Markdown |
| 4 | 更新后的开发清单 | `docs/plan/开发清单.md` | Markdown |
| 5 | HTML 设计文档（如有设计变更） | `docs/design/<模块名>/` | HTML |

### 3.3 收束节点四阶段

详见 [08-汇报收束 §收束节点](08-汇报收束.md)。

## 四、项目级验收

### 4.1 交付物清单

| # | 交付物 | 要求 |
|---|--------|------|
| 1 | **产品设计文档 + 技术文档** | 飞书文档 + 仓库 `docs/design/` |
| 2 | **可运行 Demo** | GitHub 仓库地址，含 README 快速开始 |
| 3 | **AI 协作开发记录** | 仓库内：`docs/plan/背景.md`（总纲）+ `worklogs/` + `docs/reports/`；通过飞书文档介绍协作流程和产物地址 |
| 4 | **Demo 视频** | 3-10 分钟，串流程，讲解产品主要功能 |

### 4.2 项目级验收 Checklist

- [ ] 所有 P0 功能点完成并验收通过
- [ ] 所有收束节点已执行
- [ ] 所有 ADR 已归档
- [ ] 代码仓库 README 含快速开始指南
- [ ] CHANGELOG 已更新
- [ ] 产品设计文档（飞书）指向仓库中的设计文档
- [ ] AI 协作开发记录完整（背景.md + worklogs/ + reports/）
- [ ] Demo 视频录制完成

## 五、文件放置速查

| 内容 | 位置 | 说明 |
|------|------|------|
| 产品需求文档 | `docs/plan/PRD_<项目名>.md` | PRD |
| 开发清单 | `docs/plan/开发清单.md` | 全量功能点 + 收束节点预设 |
| 任务分配 | `docs/plan/task-assignment_任务分配.md` | 模块→负责人映射（可选） |
| 设计文档 | `docs/design/<模块名>/` | 每个模块一个文件夹（7 件套） |
| BDD 规格 | `docs/specs/` | Gherkin 格式 |
| 调研笔记 | `docs/explore/` 或 `docs/research/` | 探索性/调研性内容 |
| 验收报告 | `docs/reports/` | 仅收束报告或人要求留档时落盘 |
| 工作日志 | `worklogs/YYYY-MM-DD_<主题>.md` | 每功能点更新 |
| 架构决策 | `worklogs/decisions/NNNN-<标题>.md` | 仅架构级取舍 |
| 进度状态 | `STATUS.md` | 项目根目录 |
| 仪表盘 | `dashboard.html` | 项目根目录 |
| 自动化脚本 | `scripts/` | 检查/部署/生成脚本 |
| AI 上下文 | `CLAUDE.md` | 项目根目录（AI Agent 入口） |

> 详情见 `meta/FILE_GRAPH.md`

## 六、红线速查

以下红线来自 `conventions/` 各规范。违反 = 打回，不可绕过。

| 编号 | 红线 | 来源 |

| CR-09 | 禁 force push 共享分支 | 03-Git |
| CR-10 | API 路径 kebab-case 名词复数，无动词 | 04-API |

## 七、反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| 功能点完成了不更新 STATUS | 进度不可见，后续计划失准 | 每功能点完成即更新 |
| 收束节点到了装作没看见 | 积压问题在后期集中爆发 | AI 必须拦截，拒绝开始新功能点 |
| 验收报告只有文字没有截图 | 无法证明功能真的在浏览器里跑通 | 截图 + 代码位置标注 |
| 红线违反了说"后面再修" | 技术债累积 | 违反即打回 |

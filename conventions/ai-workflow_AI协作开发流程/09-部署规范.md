# 09 · 部署规范

> 定义统一服务器（max）的部署全流程：推送 GitHub → 服务器插入模块 → 验证 → 监控。
> 部署是**第四层也是最终层**——代码写完测试通过后，只有部署上线才算真正交付。
> 关联：[02-模块分类](02-模块分类.md)（模块插入机制的依据）
> 更新: 2026-07-11

## 一、统一服务器架构

### 1.1 核心概念

**统一服务器 是所有模块的**运行时装配工厂**。它与 [02-模块分类](02-模块分类.md) 中的「基础模块 + 功能模块」体系直接对应：

```
统一服务器
    │
    ├── 基础模块（装配工厂）         ← 服务器原生运行，定义插槽
    │   ├── Orchestrator（任务分发）
    │   ├── AdapterRegistry（适配器注册中心）
    │   ├── AgentFactory（Agent 创建流程）
    │   ├── SessionManager（会话管理）
    │   └── ...
    │
    └── 功能模块（插入模块）         ← 通过注册机制插入
        ├── ClaudeCodeAdapter
        ├── CodexAdapter
        ├── OpenCodeAdapter
        ├── ConversationalAgentCreator
        └── ...
```

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **基础模块先部署** | 装配工厂（基础模块）必须先于功能模块部署，否则功能模块无处插入 |
| **模块热插入** | 新增功能模块不应要求重启整个服务器 |
| **统一凭证管理** | 所有模块的密钥/Token 走统一环境变量，不硬编码 |
| **部署即验证** | 每次部署后自动运行冒烟测试 |
| **可回滚** | 每次部署保留上一个稳定版本，出问题可一键回滚 |

### 1.3 部署拓扑

## 二、部署前检查（门禁）

### 2.1 必须全部通过

任何一项不通过 → **拒绝部署**。

| # | 检查项 | 通过标准 | 自动化 |
|---|--------|---------|:---:|
| 1 | 所有层级测试通过 | 单元 + 集成 + E2E 全部绿 | ✅ CI |
| 2 | 覆盖率门禁 | ≥ 80% | ✅ CI |
| 3 | Lint 检查 | ruff + gitleaks + markdownlint 零错误 | ✅ pre-commit |
| 4 | 规范红线检查 | 19 条红线逐条确认 | 🔧 审查 |
| 5 | PR 审查通过 | ≥ 1 Approve | 🔧 GitHub |
| 6 | 收束节点已执行 | 当前区间收束报告已签核 | 🔧 人工 |
| 7 | 环境变量完备 | 所有必需密钥/凭证已配置 | 🔧 检查 |
| 8 | 数据迁移就绪 | migration 已测试且可回滚 | 🔧 检查 |

### 2.2 收束节点与部署的关系

**未通过收束节点 → 禁止部署**。收束节点是部署前的最后一道质量闸门（详见 [08-汇报收束](08-汇报收束.md)）。

## 三、部署流程

### 3.1 总览

```
代码开发完成（第三层）
    │
    ▼
GitHub 推送 + PR 合并 ──→ 打 Tag ──→ 触发部署
    │                                    │
    ▼                                    ▼
收束节点已执行                     服务器拉取新版本
    │                                    │
    ▼                                    ▼
部署前检查全部通过                  模块插入 / 更新
    │                                    │
    ▼                                    ▼
进入部署流程                        冒烟测试自动运行
                                         │
                                         ▼
                                    健康检查通过 → 部署完成 ✅
                                    │
                                    失败 → 自动回滚 ⚠️
```

### 3.2 阶段一：GitHub 推送

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 功能分支开发完成 | 所有测试通过 + worklog 已更新 |
| 2 | 推送到 GitHub | `git push origin feature/<分支名>` |
| 3 | 创建 PR | 目标分支 `main`，标题符合 Conventional Commits |
| 4 | CI 自动运行 | lint + test + 覆盖率门禁自动触发 |
| 5 | 代码审查 | ≥ 1 Approve（按 `03-git_Git协作规范.md`） |
| 6 | 合并到 main | Squash merge 或 Merge commit |
| 7 | 打 Tag | `git tag -a vX.Y.Z -m "..."` |

### 3.3 阶段二：服务器插入模块

**核心概念**：功能模块通过**注册机制**插入统一服务器（max），不需要修改服务器核心代码。

#### 插入方式 A：代码级插入（新功能模块）

```
1. 功能模块实现基础模块定义的抽象接口
    │  如：ClaudeCodeAdapter extends BaseAdapter
    │
    ▼
2. 在模块注册中心注册
    │  adapter_registry.register("claude-code", ClaudeCodeAdapter)
    │
    ▼
3. 服务器启动时自动发现并加载
    │  通过 import 或配置文件触发注册
    │
    ▼
4. 模块在运行时可用
      用户通过统一接口调用 → 注册中心路由到对应模块
```

#### 插入方式 B：配置级插入（已部署模块的配置更新）

```
1. 修改模块配置（环境变量 / 配置文件）
    │  如：新增 LLM 凭证、调整超时时间
    │
    ▼
2. 配置热加载（无需重启）
    │  或：重启服务（如需）
    │
    ▼
3. 验证配置生效
```

#### 插入方式 C：容器级插入（独立服务）

```
1. 功能模块打包为独立容器
    │  Dockerfile → image
    │
    ▼
2. 容器注册到服务发现
    │  统一服务器通过服务发现找到新模块
    │
    ▼
3. API Gateway 路由更新
    │  新模块的端点加入路由表
```

### 3.4 阶段三：部署执行

| 步骤 | 操作 | 负责 |
|------|------|------|
| 1 | 拉取最新代码 / 镜像 | 服务器 |
| 2 | 执行数据库 migration | 服务器（自动或手动） |
| 3 | 安装/更新依赖 | 服务器 |
| 4 | 构建前端资源 | 服务器 |
| 5 | 重启/热加载服务 | 服务器 |
| 6 | 运行冒烟测试 | CI / 服务器 |
| 7 | 健康检查 | 监控系统 |

### 3.5 阶段四：部署后验证

| # | 验证项 | 方式 | 通过标准 |
|---|--------|------|---------|
| 1 | 冒烟测试 | 自动运行核心 API 调用 | 全部 200 + 数据正确 |
| 2 | 健康检查 | `GET /health` | 200 + 所有依赖绿 |
| 3 | 真浏览器验证 | Playwright 核心链路 | 截图对比通过 |
| 4 | 新功能验证 | 手动 / 自动走通新功能链路 | 功能可用 |
| 5 | 回滚验证 | 确认回滚路径可用 | 上一版本可恢复 |

## 四、环境配置

### 4.1 环境分层

| 环境 | 用途 | 部署方式 | 数据 |
|------|------|---------|------|
| **dev** | 本地开发 | 手动 / docker-compose | 测试数据，可任意重置 |
| **staging** | 预发布验证 | 自动（PR 合并后） | 仿真数据 |
| **prod** | 生产环境 | 手动触发（Tag 后） | 真实数据，不可丢失 |

### 4.2 凭证管理

| 原则 | 说明 |
|------|------|
| **禁硬编码** | 所有密钥/Token/密码走环境变量（`.env`） |
| **`.env` 不入库** | `.gitignore` 已配置，`.env.example` 提供模板 |
| **分级管理** | dev/staging/prod 三套独立凭证 |
| **展示脱敏** | 展示密钥时用 inline code + 末 4 位明文（`02-coding` 规范） |

### 4.3 环境变量模板

```bash
# .env.example（入库）
# 统一服务器 max 环境变量

# 数据库
DATABASE_URL=postgresql://user:password@localhost:5432/dbname

# LLM 凭证
CLAUDE_API_KEY=sk-xxxx           # 仅末 4 位明文
CODEX_API_KEY=sk-xxxx
OPENCODE_API_KEY=sk-xxxx

# 服务器
HOST=0.0.0.0
PORT=8000
WS_PORT=8001

# 部署
DEPLOY_ENV=dev                   # dev | staging | prod
ROLLBACK_ENABLED=true
HEALTH_CHECK_URL=/health
```

## 五、回滚机制

### 5.1 回滚触发条件

| 条件 | 动作 |
|------|------|
| 冒烟测试失败 | 自动回滚 |
| 健康检查失败 | 自动回滚 |
| 错误率 > 阈值（5 分钟内） | 告警 → 人工决定 |
| 人发现严重 Bug | 人工触发回滚 |

### 5.2 回滚流程

```
检测到需要回滚
    │
    ▼
停止当前版本服务
    │
    ▼
切换到上一个 Tag / Release
    │
    ▼
执行 migration 回滚（如有）
    │
    ▼
启动服务
    │
    ▼
冒烟测试通过 → 回滚完成 ✅
    │
    ▼
记录回滚原因 + 通知团队
```

### 5.3 回滚要求

| 要求 | 说明 |
|------|------|
| 保留上一版本 | 每次部署保留上一个稳定版本的代码/镜像 |
| 数据兼容 | migration 必须有 down 操作（可回滚） |
| 回滚演练 | 每次部署后验证回滚路径可用 |
| 回滚记录 | 记录在 worklog 中 |

## 六、监控与告警

### 6.1 监控指标

| 类别 | 指标 | 告警阈值 |
|------|------|---------|
| 可用性 | 服务存活 | 连续 3 次健康检查失败 |
| 延迟 | API 响应时间 P95 | > 2s |
| 错误率 | 5xx 占比 | > 5%（5 分钟窗口） |
| 资源 | CPU / 内存 / 磁盘 | > 80% |
| 业务 | 消息发送成功率 | < 95% |

## 七、与模块分类的关系

部署方式由模块类型决定（回顾 [02-模块分类](02-模块分类.md)）：

| 模块类型 | 部署方式 | 插入时机 |
|---------|---------|---------|
| **基础模块**（装配工厂） | 随服务器核心一起部署 | 服务器启动时 |
| **功能模块**（插件） | 通过注册机制插入 | 基础模块就绪后，可独立部署 |
| **工程模块** | 作为独立服务或随核心部署 | 不依赖注册机制 |

### 7.1 基础模块部署原则

- 基础模块是服务器的**骨架**——必须先部署
- 基础模块变更 = 服务器核心变更 = 需要完整部署流程
- 基础模块的接口变更 = **破坏性变更**——需要所有功能模块同步更新

### 7.2 功能模块部署原则

- 功能模块可**独立部署**——不依赖其他功能模块
- 新增功能模块 = 新增一个插件 = 注册 + 配置更新
- 功能模块更新 = 替换插件实现 = 热加载或重启

## 八、部署 Checklist

每次部署前核对：

### 代码侧
- [ ] 所有测试通过（单元 + 集成 + E2E）
- [ ] 覆盖率 ≥ 80%
- [ ] PR 已审查并 Approve
- [ ] commit message 符合 Conventional Commits
- [ ] worklog 已更新
- [ ] STATUS.md 已同步
- [ ] 收束节点已执行（如到达）

### 配置侧
- [ ] 环境变量完备（dev/staging/prod 三套）
- [ ] 密钥不包含在代码中（gitleaks 通过）
- [ ] migration 有 up 和 down
- [ ] `.env.example` 已更新（如有新增变量）

### 部署侧
- [ ] 冒烟测试脚本就绪
- [ ] 健康检查端点可用
- [ ] 回滚路径已验证
- [ ] 监控告警已配置

### 部署后
- [ ] 冒烟测试通过
- [ ] 健康检查通过
- [ ] 真浏览器核心链路走通
- [ ] 新功能可用
- [ ] 上一版本可回滚（已验证）

## 九、反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| 跳过 staging 直接上 prod | 没有预发布验证，问题直接暴露给用户 | dev → staging → prod 逐级推进 |
| 功能模块部署时重启整个服务器 | 影响所有用户 | 热加载或分批重启 |
| 密钥硬编码在代码里 | 泄露风险 + 换密钥要改代码 | 环境变量 + `.env` |
| migration 没有 down 操作 | 出问题无法回滚 | 每个 migration 配 down |
| 部署后不验证 | "应该没问题"不是证据 | 冒烟测试 + 真浏览器验证 |
| 基础模块和功能模块同时部署 | 基础模块接口变更可能导致所有功能模块挂 | 基础模块先部署验证，功能模块再跟进 |
| 回滚路径从不演练 | 真需要回滚时才发现回不去 | 每次部署后验证回滚可用 |

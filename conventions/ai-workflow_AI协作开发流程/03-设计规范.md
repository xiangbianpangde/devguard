# 03 · 设计规范

> 定义每个模块/功能组的**设计文件夹标准结构**，以及核心文档的章节模板。
> 开始设计任何模块前必读本篇。
> 参考模板：`docs/plan/design/M1-im-chat-v2.0-参考模板/`

---

## 一、设计文件夹结构（7 件套）

每个模块或功能组的设计文档按以下标准结构组织：

```
docs/design/<模块名>/
├── README.md           # 文件夹索引：这个模块是什么 / 包含哪些文档 / 依赖关系
├── 简报.md              # ≤ 80 行快速概览（详见 §二）
├── 设计.md              # 宏观设计 8 节（详见 §三）
├── 实现计划.md          # 实施路径 8 节（详见 §四）
├── 阅读笔记.md          # AI 阅读导航（详见 §五）
├── deliverable.md      # 交付物清单（详见 §六）
├── 设计.html           # 设计.md 的 HTML 渲染版（人机交互评审用）
└── 实现计划.html        # 实现计划.md 的 HTML 渲染版（人机交互评审用）
```

> **HTML 文件**：由 Markdown 渲染生成，含 Mermaid 图表、折叠区、锚点导航。用于人机交互评审——人在浏览器中查看并提出修改意见。

## 二、简报.md 结构

**定位**：1 分钟速读。任何人（含非技术干系人）打开这个文件就能理解本模块是什么。

| 节 | 内容 | 行数建议 |
|----|------|:---:|
| 标题 | `<模块名> · 简报` | 1 |
| 一句话 | 3 秒读懂：本模块要解决什么问题 | 1 |
| 功能速览 | 4 列表：功能点 / 优先级 / 用户怎么做 / 系统做什么 | ≤ 10 |
| 关键数字 | 功能点数 / 总工时 / 硬阻塞数 / 零后端改动功能数 | ≤ 6 |
| 接口拓扑 | Mermaid 图：用户 → 前端 → API → 后端 → 数据库 | 1 图 |
| 推荐顺序 | 阶段顺序 + 阻塞标注 | ≤ 5 |
| 核心决策 | 决策 / 选择 / 原因 三列表 | ≤ 8 |
| 风险 | 风险 / 缓解 两列表 | ≤ 6 |

## 三、设计.md 结构（8 节）

**定位**：宏观设计——讲 WHAT + WHY，不讲 HOW（HOW 在实现计划.md）。

### §一 背景：现在能做什么，缺什么

| 内容 | 说明 |
|------|------|
| 已有基建 | 现有系统能做什么 |
| 差距分析 | 现状 vs 目标的四列表（功能点 / 现在 / 目标） |
| 为什么做 | 业务价值 / 用户痛点 |

### §二 用户故事

每个功能点写一个完整的用户故事：

- **用户故事**：一段话描述用户是谁、他在什么场景下、他想做什么
- **交互流程**：ASCII 流程图或用例步骤
- **根本原因**（如适用）：为什么现在不行、表象和根因的区别

### §三 架构决策

每个关键决策一段：

- **选择**：选了方案 A 还是方案 B
- **原因**：为什么选这个
- **代价**：这个选择带来的成本 / 限制

### §四 数据流

每个功能点一条数据流链路：

```
触发点 → 前端层 → API 层 → Service 层 → Repository 层 → 数据库 → 返回路径
```

### §五 组件关系（概念层）

ASCII 框图展示涉及的组件及其连接关系，标注哪些是新增、哪些是修改、哪些不动。

### §六 边界与降级

| 内容 | 说明 |
|------|------|
| 降级场景 | 每项依赖不可用时的降级行为 |
| 明确不做 | 列出本模块明确不做的事情（防止范围蔓延） |
| 已知未知 | "我现在不知道但是有的"——待确认的技术细节 + 责任人 + 截止日期 |

### §七 验收标准

| 功能点 | 验收方式 | 通过标准 |
|--------|---------|---------|

### §八 与其他模块的关系

| 关系 | 模块列表 |
|------|---------|
| 依赖谁 | 本模块依赖的外部模块/服务 |
| 被谁依赖 | 哪些模块依赖本模块 |
| 与谁并行 | 可以并行开发的模块 |
| 与谁无关 | 没有契约冲突的模块（注明以消除顾虑） |

## 四、实现计划.md 结构（8 节）

**定位**：实施路径——讲 HOW（阶段·里程碑·依赖·阻塞）。

### §一 总览

六列表：功能点 / 阶段 / 优先级 / 工时 / 后端改动 / 前端改动 / 阻塞

### §二~§五 各阶段实现路径

每个阶段一节，包含：

- **目标**：本阶段要完成什么
- **实现路径**：ASCII 流程图（先做什么 → 再做什么）
- **改动模块清单**：表格（层 / 模块 / 改什么 / 估时）
- **阻塞项**：本阶段的硬阻塞及缓解方案

### §六 里程碑

时间线视图，标注可并行段：

```
阶段 A ────┴────
阶段 B ────┴────
阶段 C         ────┴────
阶段 D               ────┴────
```

## 八、HTML 渲染规范

### 8.1 设计要求

- **Mermaid 图表**：所有流程图、架构图、拓扑图必须可渲染
- **折叠区**：长代码块、详细表格默认折叠
- **锚点导航**：左侧/顶部固定导航栏
- **截图灯箱**：点击缩略图放大
- **高信息密度**：一屏内展示尽可能多的关键信息

### 8.2 人机交互评审流程

```
AI 生成 HTML → 人浏览器查看 → 人提出修改意见
                                    │
                    ┌───────────────┘
                    ▼
              AI 修改 Markdown → 重新生成 HTML → 人再审
                                    │
                              循环至人确认 ✅
```

## 九、与 conventions/ 规范的关系

本设计规范是以下规范的**流程级细化**：

- `01-architecture_架构设计规范.md` — 分层 / 依赖方向

# 01 · 流程全景

> 定义 AI 协作的两类任务 + 它们的关系 + 触发条件。
> 开工前必读本篇，判断当前任务属于哪一类。
> 更新: 2026-06-10

---

## 一、两类任务

本项目将 AI 协作任务分为两大类，按 **产出物性质** 区分：

### 1.1 长程任务（Long-Running Task）

**定义**：需要多轮迭代、产生代码/测试/部署等**可执行产物**的任务。

**典型场景**：
- 功能实现（新功能开发）
- Bug 修复
- 代码重构
- 性能优化
- 部署与运维

**特征**：
- 产出是**可运行的代码**
- 需要 TDD 循环（红 → 绿 → 重构）
- 需要真机 E2E 验证（不接受 HTTP 200 / tsc 零错误作为证据）
- 每功能点完成为最小交付单元

**AI 角色**：执行者 — 按规范写代码、跑测试、提交、汇报。

### 1.2 思考任务（Thinking Task）

**定义**：以方案讨论、需求分析、技术调研等**思考/设计**为主的任务。

**典型场景**：
- 需求讨论与 PRD 编写
- 技术方案设计
- 架构评审
- 技术调研（调研报告）
- 代码审查（仅审查，不修改）

**特征**：
- 产出是**设计文档 / 决策记录 / 调研报告**
- AI 是**讨论引导者**而非执行者
- 核心方法是五步法（详见 [05-思考设计](05-思考设计.md)）
- 完成后形成收束节点

**AI 角色**：讨论 Agent — 笔记整理 + 正向引导 + 选项呈现 + 规范设计。

### 1.3 对比总结

| 维度 | 长程任务 | 思考任务 |
|------|---------|---------|
| 产出物 | 可运行代码 + 测试 + 报告 | 设计文档 + 决策记录 + 调研 |
| AI 角色 | 执行者 | 讨论引导者 |
| 工作单元 | 功能点 | 设计议题 |
| 验证方式 | TDD + E2E 真浏览器截图 | 人审阅 + HTML 交互 |
| 流程文档 | [04-长程开发](04-长程开发.md) | [05-思考设计](05-思考设计.md) |
| 典型时长 | 数小时 ~ 数天 | 数十分钟 ~ 数小时 |

## 二、任务判定

### 2.1 判定流程

```
接到任务
    │
    ▼
它需要写代码吗？
    │
    ├── 是 → 长程任务 → 走 [04-长程开发](04-长程开发.md)
    │
    └── 否 → 它在讨论/设计/调研吗？
                │
                ├── 是 → 思考任务 → 走 [05-思考设计](05-思考设计.md)
                │
                └── 否 → 回到任务描述，找用户澄清
```

### 2.2 边界案例

| 场景 | 判定 | 理由 |
|------|:---:|------|
| 写一个技术方案文档 | 思考 | 产出是文档，不写代码 |
| 根据已审批的设计写代码 | 长程 | 设计已完成，现在是执行 |
| 边讨论边写原型代码 | 思考 → 长程 | 讨论阶段是思考；原型确认后转入长程 |
| 代码审查（只读不写） | 思考 | 审查结论是文档/评论 |
| 代码审查（发现 Bug 并修复） | 长程 | 修复 = 写代码 |
| 调研一个新技术栈并写 Demo | 思考 → 长程 | 调研报告=思考；Demo=长程 |

## 三、两类任务的关系

```
思考任务                          长程任务
   │                                │
   ▼                                ▼
讨论 + 设计                          实现 + 测试
   │                                │
   ├── 产出：设计文件夹               ├── 输入：设计文件夹（来自思考任务）
   ├── 产出：BDD 规格                 ├── 依据：BDD 规格（来自思考任务）
   └── 产出：技术方案                 └── 依据：技术方案（来自思考任务）
   │                                │
   ▼                                ▼
收束节点 ──────── 闸门 ────────→ 预先计划 ──→ 迭代开发 ──→ 验收报告
                                        │
                              设计完成后转入长程开发
```

**关键规则**：
- 思考任务**必须先完成**，产生设计文档和技术方案，才能启动长程开发
- 收束节点是两类任务的**交接闸门**：思考任务以收束节点结束，长程任务从收束节点后的预先计划开始
- 思考任务和长程任务的收束节点是**同一个概念**（详见 [08-汇报收束](08-汇报收束.md)）

## 四、完整端到端流程（总览）

以下是从项目启动到交付的端到端流程（详见 [06-端到端流程](06-端到端流程.md)）：

```
PRD ──→ BDD+TDD ──→ 技术转换 ──→ 模块划分 ──→ 并行+层级开发 ──→ 测试 ──→ 交付
 │         │            │            │              │              │
 ▼         ▼            ▼            ▼              ▼              ▼
思考      思考+长程     思考         思考           长程           长程
```

> 前四个阶段以**思考任务**为主（产出设计文档）；后两个阶段以**长程任务**为主（产出可运行代码）。

## 五、收束节点（闸门机制）

收束节点是流程中的**硬闸门**，默认每 **3 个功能点**触发一次。

### 5.1 作用

- **防止积压**：强制定期整理、测试、审计、验证
- **质量门禁**：未通过收束 → 不能开始下一个功能点
- **交接点**：思考任务 → 长程任务的切换点

### 5.2 触发时机

| 触发条件 | 说明 |
|---------|------|
| 预设闸门 | `docs/plan/开发清单.md` 头部预设（默认每 3 功能点） |
| 思考任务完成 | 每次完成一轮完整设计 = 一次收束节点 |
| 人手动触发 | 人判断需要收束时主动触发 |

### 5.3 执行内容

详见 [08-汇报收束 §收束节点](08-汇报收束.md)。

# 05 · 思考设计

> 定义思考任务的五步法：讨论 Agent 引导用户完成从需求到规范化设计文件夹的全过程。
> 适用于方案讨论、需求分析、技术调研等以思考/设计为主的任务。
> 更新: 2026-07-11

---

## 一、思考任务总览

```
Step 1               Step 2               Step 3               Step 4               Step 5
笔记                  引导                  规范设计              HTML 汇报             记录整理
  │                    │                     │                     │                     │
  ▼                    ▼                     ▼                     ▼                     ▼
边讨论边整理        正向激发+心流引导      根据初稿+对话          HTML 人机交互         收束报告+worklog
Markdown 实时记录    +选项引导              +开发/设计规范        人提出修改意见         +更新STATUS
                    帮用户完成设计初稿     生成规范化设计文件夹   迭代至确认             +更新开发清单
```

## 二、Step 1：笔记

### 2.1 做什么

边讨论边整理 Markdown 笔记，实时记录讨论过程。

### 2.2 记录内容

| 记录项 | 说明 |
|--------|------|
| 讨论要点 | 每个议题的核心观点 |
| 分歧点 | 不同方案的对立观点 |
| 结论 | 达成的共识和决策 |
| 待办事项 | 需要后续跟进的事项 |
| 开放问题 | 本次讨论未解决、需要下次讨论的问题 |

### 2.3 存储位置

- 与某个模块强相关 → `docs/design/<模块名>/` 或该模块设计文件夹
- 探索性/调研性 → `docs/explore/<主题>/`
- 技术调研 → `docs/research/<主题>/`

### 2.4 格式要求

- Markdown 格式
- 时间戳标注（至少标注日期）
- 参与者标注（如适用）

## 三、Step 2：引导

### 3.1 目标

通过正向激发、心流引导、选项引导，帮助用户完成**计划设计初稿**。

### 3.2 三项引导技术

#### 正向激发

- **肯定用户的合理想法**："这个方向很好，我们可以进一步…"
- **扩展思路**："基于你的想法，还可以考虑…"
- **避免否定**：不说"这样不对"，说"这个方案有个潜在问题…我们看看有没有办法规避"

#### 心流引导

- **控制讨论节奏**：避免过早陷入细节
- **从宏观到微观**：背景 → 用户故事 → 架构 → 数据流 → 细节
- **识别心流中断**：用户卡住时主动提供选项打破僵局
- **节奏信号**：
  - 太快（用户跟不上）→ 放慢，总结当前进度
  - 太慢（用户失去兴趣）→ 抛出下一个关键决策点

#### 选项引导

- **遇到决策点**：给出 2-4 个选项（含利弊分析）
- **让用户拍板**：不自作主张
- **选项格式**：

```markdown
## 决策点：<问题>

| 选项 | 优点 | 缺点 | 推荐度 |
|------|------|------|:---:|
| A: ... | ... | ... | ⭐⭐⭐ |
| B: ... | ... | ... | ⭐⭐ |
| C: ... | ... | ... | ⭐ |

我的推荐是 A，因为... 你觉得呢？
```

### 3.3 产出

设计初稿——可以是聊天记录 + 要点总结的 Markdown 文档。

## 四、Step 3：规范设计

### 4.1 输入

- Step 2 产出的设计初稿
- 原始对话文件（聊天记录）
- `conventions/` 下的开发规范（01-06, 08）
- [03-设计规范](03-设计规范.md)（本文件夹）

### 4.2 输出

规范化的设计文件夹，按 [03-设计规范 §一](03-设计规范.md) 的 7 件套结构：

```
docs/design/<模块名>/
├── README.md
├── 简报.md
├── 设计.md          # 按 8 节结构
├── 实现计划.md      # 按 8 节结构
├── 阅读笔记.md
├── deliverable.md
├── 设计.html        # 后续 Step 4 生成
└── 实现计划.html    # 后续 Step 4 生成
```

### 4.3 AI 的职责

1. 从初稿中提取关键信息，填入对应章节
2. 发现初稿中的遗漏和模糊点，标注为「待确认」
3. 应用开发规范的红线检查设计（如分层/依赖方向/API 命名）
4. 识别模块类型（工程模块 vs 基础+功能模块），按 [02-模块分类](02-模块分类.md) 标注
5. 生成 deliverable.md 中的「Owner 决策点」

### 4.4 质量自检

| 检查项 | 说明 |
|--------|------|
| 8 节齐全 | 设计.md 的 8 节每节都有实质内容 |
| 每个功能点有用户故事 | §二覆盖了所有功能点 |
| 每个架构决策有选择+原因+代价 | §三每个决策三项齐全 |
| 数据流可追溯 | §四从触发点到数据库完整 |
| 降级方案存在 | §六每个外部依赖有降级行为 |
| Owner 决策点明确 | deliverable.md 中有待拍板问题 |

## 五、Step 4：HTML 汇报

### 5.1 目标

将设计文档生成为 HTML 格式，通过**人机交互界面**展示，人审阅后提出修改意见。

### 5.2 交互流程

```
AI 生成 HTML（设计.html + 实现计划.html）
    │
    ▼
人在浏览器中查看
    │
    ├── 满意 → 确认 ✅ → 进入 Step 5
    │
    └── 不满意 → 提出修改意见
                    │
                    ▼
              AI 修改 Markdown → 重新生成 HTML → 人再审
                    │
                    └── 循环至人确认
```

### 5.3 HTML 要求

| 要素 | 说明 |
|------|------|
| Mermaid 图表渲染 | 所有流程图/架构图/拓扑图可渲染 |
| 折叠区 | 长代码块、详细表格默认折叠 |
| 锚点导航 | 固定侧边栏/顶部导航 |
| 响应式 | 桌面端为主，兼顾移动端查看 |
| 打印友好 | 导出 PDF 不乱 |

## 六、Step 5：记录整理

### 6.1 触发条件

完成一次完整设计（设计.md + 实现计划.md 经人确认） = 一次 **收束节点**。

### 6.2 产出

| # | 产出 | 位置 |
|---|------|------|
| 1 | 收束报告（HTML） | `docs/reports/` |
| 2 | worklog | `worklogs/YYYY-MM-DD_<主题>.md` |
| 3 | STATUS 更新 | `STATUS.md` |
| 4 | 开发清单更新 | `docs/plan/开发清单.md` |
| 5 | ADR（如有架构决策） | `worklogs/decisions/NNNN-<标题>.md` |

### 6.3 执行细则

详见 [08-汇报收束](08-汇报收束.md)。

## 七、思考任务 → 长程任务的切换

```
思考任务完成（Step 5 收束）
    │
    ▼
设计文件夹就绪（设计.md + 实现计划.md + BDD 规格）
    │
    ▼
切换到长程任务模式
    │
    ▼
按 [04-长程开发](04-长程开发.md) 执行：
    预先计划（确认实现计划.md）→ 迭代开发 → 验收报告
```

**关键规则**：思考任务必须先完成，设计文档经人确认后，才能启动长程开发。不允许"边设计边开发"——设计阶段的不确定性会污染实现阶段。

## 八、反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| AI 替人做产品决策 | 越界——产品决策需要人的判断 | 给选项 + 利弊分析，让人拍板 |
| 讨论跳跃过早陷入细节 | 打断心流，宏观方向未定就讨论实现细节 | 从宏观到微观递进 |
| 跳过规范设计直接写代码 | 缺乏设计约束，实现方向可能偏离 | 完成设计 → 人确认 → 再写代码 |
| 选项过多（>4 个） | 选择瘫痪 | 控制在 2-4 个 |
| 设计文档不给人审阅 | 设计偏差发现太晚 | Step 4 HTML 汇报必须做 |

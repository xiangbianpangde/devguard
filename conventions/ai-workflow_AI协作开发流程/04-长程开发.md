# 04 · 长程开发

> 定义长程任务的完整开发流程：AI决策 → 预先计划 → 验收报告。
> 适用于功能实现、Bug 修复、代码重构等需要多轮迭代的任务。
> 更新: 2026-06-10

---

## 一、长程任务总览

```
收到长程任务
    │
    ▼
阶段一：AI决策 ──→ 阶段二：预先计划 ──→ 阶段三：迭代开发（每功能点循环）
    │                   │                      │
    ▼                   ▼                      ▼
ADR 落盘            实现计划.md             TDD 红→绿→重构
(worklogs/decisions) (docs/design/)         真机 E2E 验证
                                             │
                                    ┌───────┘
                                    ▼
                              验收报告（每功能点）
                              │
                              ▼
                         收束节点（每 3 功能点）
```

---

## 二、阶段一：AI 决策

### 2.1 什么需要记录为 ADR

**触发条件**：任何涉及"选 A 还是选 B"的架构/设计层面的取舍。

| 需要 ADR | 不需要 ADR |
|---------|-----------|
| 技术栈选型（React vs Vue） | 变量命名 |
| 架构模式选择（微服务 vs 单体） | 代码风格 |
| 数据存储方案（PostgreSQL vs MongoDB） | 函数拆分方式 |
| 第三方服务选型 | 日常 bug 修复方案 |
| API 设计范式选择 | 重构中的局部实现细节 |

### 2.2 ADR 格式

存放位置：`worklogs/decisions/NNNN-<简短标题>.md`

编号：从 0001 开始递增。

```markdown
# ADR-NNNN: <标题>

> 日期: YYYY-MM-DD
> 状态: 提议 / 已采纳 / 已废弃 / 已替代
> 替代: ADR-xxxx（如适用）

## 背景
<为什么需要做这个决策>

## 决策
<选择了什么方案>

## 考虑的替代方案
| 方案 | 优点 | 缺点 |
|------|------|------|

## 后果
<这个决策带来的正面/负面影响>

## 相关
<关联的 ADR / 设计文档 / PR>
```

### 2.3 不需要 ADR 的决策

日常取舍记 worklog 的「关键决策」段即可，不落 ADR。

## 三、阶段二：预先计划

### 3.1 产出物

`docs/design/<模块名>/实现计划.md`，按 [03-设计规范 §四](03-设计规范.md) 的 8 节结构编写。

### 3.2 必须包含

| # | 要素 | 说明 |
|---|------|------|
| 1 | 阶段划分 | 每个阶段：目标 + 实现路径 + 改动模块清单（含估时） |
| 2 | 阻塞项 | 每阶段的硬阻塞及缓解方案 |
| 3 | 里程碑 | 时间线视图，标注可并行段 |
| 4 | 验收标准 | 每阶段的通过条件（可测试、可量化） |
| 5 | 红线触发 | 每个功能点触发了哪些规范红线 |

### 3.3 推荐顺序原则

1. **无阻塞的先做**：不依赖外部条件的优先
2. **有阻塞的后做**：等阻塞解除再开工
3. **零后端改动的优先**：纯前端/纯测试改动排前面
4. **高风险的后做**：改动面大的排在后面（有更多时间测试）

### 3.4 参考模板

`docs/plan/design/M1-im-chat-v2.0-参考模板/实现计划.md`

## 四、阶段三：迭代开发

### 4.1 每功能点循环

```
实现计划中取下一个功能点
    │
    ▼
TDD：红（写失败测试）→ 绿（最小实现）→ 重构
    │
    ▼
真机 E2E 验证（不接受 HTTP 200 / tsc 零错误作为证据）
    │
    ▼
审查 diff（自查规范红线）
    │
    ▼
人确认（浏览器查看 / 接口调用确认）
    │
    ▼
git commit（Conventional Commits，引用 worklog）
    │
    ▼
更新 worklog + STATUS.md
    │
    ▼
功能点验收报告（内联汇报，不落盘）
    │
    ▼
检查是否到达收束节点 → 是 → 触发收束；否 → 下一功能点
```

### 4.2 TDD 纪律

按 `05-testing_测试规范.md` 执行：

- 先写测试 → 测试失败（红）
- 最小实现 → 测试通过（绿）
- 重构优化 → 测试仍通过
- 覆盖：正常路径 + 边界条件 + 异常路径

### 4.3 真机 E2E 验证（不可妥协）

| 前端改动 | 接口改动 |
|---------|---------|
| 真浏览器渲染并亲眼确认 | 真实调用并核对返回数据 |
| Playwright 截图存档 | curl / Postman 实际调用 |
| 不接受 demo 录屏/截图自我证明 | 不接受"代码看起来没问题" |

## 五、验收报告（8 要素）

每完成一个功能点，生成验收报告。**功能点级验收报告默认对话内联汇报（不落盘）**；仅收束报告或人要求留档时才落盘 `docs/reports/`。

### 5.1 八要素

| # | 要素 | 要求 |
|---|------|------|
| 1 | **截图测试** | 真浏览器截图（Playwright / 手动），不接受 HTTP 200 或 tsc 零错误作为证据 |
| 2 | **具体代码 + 位置** | 每个改动点标注 `文件路径:行号` |
| 3 | **可视化 + 可交互 + 高信息密度** | HTML 报告含 Mermaid 图、折叠区、锚点导航、截图灯箱（落盘时） |
| 4 | **完成任务 + 未完成任务** | 明确列出 Done / Not Done / Blocked，每个带一句话说明 |
| 5 | **Design 清单** | 对照 `设计.md` §七验收标准逐项勾验 |
| 6 | **Commit + Worklog 清单** | 列出本次涉及的 commit hash + worklog 文件路径 |
| 7 | **STATUS 完成汇报** | 更新 `STATUS.md` 进度 |
| 8 | **交接细则 + 后续方向** | 下一个接手的人需要知道什么 / 下一步建议 |

### 5.2 验收报告 HTML 格式（落盘时）

```html
<!DOCTYPE html>
<html>
<head><title>验收报告 · <模块名> · <日期></title></head>
<body>
  <h1>验收报告</h1>
  <section id="screenshots"><h2>1. 截图测试</h2>...</section>
  <section id="code"><h2>2. 具体代码+位置</h2>...</section>
  <section id="status"><h2>3. 完成/未完成</h2>...</section>
  ...
  <section id="handoff"><h2>8. 交接细则+后续方向</h2>...</section>
</body>
</html>
```

## 六、收束节点触发

默认每 **3 个功能点**触发一次收束节点（闸门设在 `docs/plan/开发清单.md` 头部）。

**到达收束节点时**：
1. AI **拒绝开始下一个功能点**
2. AI 提醒人触发收束
3. 人确认后执行收束节点四阶段（详见 [08-汇报收束 §收束节点](08-汇报收束.md)）

## 七、反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| 跳过设计直接写代码 | 没有预先计划，实现方向可能偏离 | 先完成实现计划.md |
| 所有决策都写 ADR | ADR 泛滥，重要决策被淹没 | 只有架构级取舍才写 ADR |
| 跳过 TDD 直接写实现 | 测试后补通常覆盖不足 | 红→绿→重构 |
| HTTP 200 当验收证据 | 接口可能返回 200 但是错误的数据 | 真实调用+核对返回数据 |
| 积压到收束节点一次性汇报 | worklog 断档，无法追溯 | 每功能点即时更新 |

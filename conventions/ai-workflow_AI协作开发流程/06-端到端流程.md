# 06 · 端到端流程

> 定义从 PRD 到交付的完整开发流程，共 6 个阶段。
> 新项目启动或大模块开发时，按此流程端到端执行。
> 更新: 2026-07-11

---

## 一、流程总览

```
PRD ──→ BDD+TDD ──→ 技术转换 ──→ 模块划分 ──→ 并行+层级开发 ──→ 测试 ──→ 部署 ──→ 交付
 │         │            │            │              │              │        │
 ▼         ▼            ▼            ▼              ▼              ▼        ▼
思考      思考+长程     思考         思考           长程           长程     长程
(讨论Agent)                                        (四层)         (三层)   (统一服务器max)
```

| 阶段 | 任务类型 | 主要产出 | 参考 |
|------|:---:|------|------|
| 1. PRD | 思考 | 产品需求文档 | 本章 §二 |
| 2. BDD+TDD | 思考 → 长程 | BDD 规格 + 测试用例 | 本章 §三 |
| 3. 技术转换 | 思考 | 9 项技术文档 | 本章 §四 |
| 4. 模块划分 | 思考 | 模块设计 + 任务分配 | 本章 §五 |
| 5. 并行+层级开发 | 长程 | 可运行代码（第四层含部署） | 本章 §六 |
| 6. 测试 | 长程 | 测试报告 | 本章 §七 |
| 7. 部署 | 长程 | 线上可访问 | 本章 §八 + [09-部署规范](../09-部署规范.md) |

## 二、阶段一：PRD（产品需求文档）

### 2.1 目标

明确做什么、为谁做、为什么做。

### 2.2 内容

| 要素 | 说明 |
|------|------|
| 用户故事 | 用户是谁 / 在什么场景下 / 想完成什么 |
| 功能列表 | 按 P0 / P1 / P2 排优先级 |
| 验收标准 | 每个功能的通过条件（可测试、可量化） |
| 非功能需求 | 性能 / 安全 / 可用性 / 兼容性 |
| 范围边界 | 明确不做的事情 |

### 2.3 存放位置

`docs/plan/PRD_<项目名>.md`

### 2.4 AI 角色

- 帮助整理用户需求为结构化文档
- 问尖锐问题发现遗漏（"如果用户断网了怎么办？"）
- 标注模糊表述（"你说的'实时'是延迟 < 100ms 还是 < 1s？"）
- **不替人做产品决策** — 优先级、功能取舍由人决定

### 2.5 完成标准

- [ ] 用户故事覆盖所有核心场景
- [ ] 功能按 P0/P1/P2 分级
- [ ] 每个功能有可测试的验收标准
- [ ] 人已确认

## 三、阶段二：BDD + TDD

### 3.1 BDD（行为驱动开发）

**定义**：把 PRD 的用户故事转为**可执行的规格**。

**格式**：Gherkin

```gherkin
Feature: <功能名>
  作为 <角色>
  我想要 <功能>
  以便 <价值>

  Scenario: <场景名>
    Given <前置条件>
    When <操作>
    Then <预期结果>
```

**存放位置**：`docs/specs/` 目录

| 规格类型 | 文件 | 内容 |
|---------|------|------|
| 用户故事规格 | `01-user-stories_用户故事.md` | BDD Feature + Scenario |
| 数据模型 | `03-data-model_数据模型.md` | 实体定义、字段、约束 |
| 命令接口 | `04-commands_命令接口.md` | API 端点 × 请求/响应结构 |

### 3.2 TDD（测试驱动开发）

**循环**：红（写失败测试）→ 绿（最小实现）→ 重构

**测试分层**（详见 `05-testing_测试规范.md`）：

| 层 | 框架 | 覆盖范围 |
|----|------|---------|
| 单元测试 | pytest / vitest | 函数、类、组件 |
| 集成测试 | pytest + TestClient | API 端点 + 数据库 |
| E2E 测试 | Playwright | 完整用户链路 |

### 3.3 BDD ↔ TDD 的关系

- **BDD 定义"做什么"**（从用户视角）
- **TDD 保证"做对了"**（从代码视角）
- 一个 BDD Scenario 可能对应多个 TDD 测试用例（正常+边界+异常）

## 四、阶段三：技术转换

### 4.1 目标

将 PRD 的"用户语言"转换为"技术语言"，消除歧义，形成技术方案。

### 4.2 九项产出

| # | 产出 | 说明 | 存放位置 |
|---|------|------|---------|
| 1 | **技术方案** | 总体架构选型、技术栈、关键设计模式 | `docs/design/<模块名>/设计.md` |
| 2 | **技术术语表** | PRD 业务术语 → 技术对应物（如"聊天对象" → `Agent` 实体） | 设计.md §附录 |
| 3 | **业务流程图/架构图** | Mermaid 格式，含数据流和模块交互 | 设计.md §四/§五 |
| 4 | **歧义消除表** | PRD 中模糊表述的明确化（如"实时" → 延迟 < 500ms） | 设计.md §一 |
| 5 | **业务规则表** | 条件 → 动作 → 异常 的三列决策表 | 设计.md §六 |
| 6 | **数据字典** | 实体字段定义：字段名 / 类型 / 约束 / 说明 | `docs/specs/03-data-model_数据模型.md` |
| 7 | **契约表** | API 端点 × 请求/响应结构（与 `04-commands_命令接口.md` 对齐） | `docs/specs/04-commands_命令接口.md` |
| 8 | **依赖表** | 本模块依赖的外部服务/模块/凭证 | 设计.md §八 |
| 9 | **"我现在不知道但是有的"表** | 已知的未知：待确认的技术细节 + 责任人 + 截止日期 | 设计.md §六 |

### 4.3 歧义消除表格式

| PRD 原表述 | 歧义点 | 技术明确化 | 确认人 | 确认日期 |
|-----------|--------|-----------|--------|---------|
| "实时更新" | 延迟多少算实时？ | WebSocket 推送，延迟 < 500ms | — | — |
| "大量数据" | 多大算大？ | 单次查询 > 1000 行需分页 | — | — |

### 4.4 业务规则表格式

| 条件 | 动作 | 异常处理 |
|------|------|---------|
| 用户发送消息 | 消息入队 → Agent 处理 → 流式返回 | Agent 超时 → 返回"正在处理中"提示 |
| 用户归档会话 | 标记 archived=true → 从主列表移除 | 会话不存在 → 返回 404 |

## 五、阶段四：模块划分

### 5.1 划分原则

1. **先判定类型**：每个模块按 [02-模块分类](02-模块分类.md) 分为工程模块或基础+功能模块
2. **一个模块 = 一个设计文件夹**：`docs/design/<模块名>/`
3. **模块边界 = 接口契约**：数据契约（工程模块）或抽象类/接口（基础+功能模块）
4. **依赖无环**：模块间依赖必须是 DAG（有向无环图）

### 5.2 任务分配

| 维度 | 说明 |
|------|------|
| 分配方式 | 按模块分配，1 个功能点 = 1 个开发任务 |
| 优先级 | P0 → P1 → P2，同一优先级内按依赖拓扑排序 |
| 并行策略 | 无依赖关系的模块可并行开发 |
| 跟踪 | `docs/plan/开发清单.md` + `STATUS.md` |

### 5.3 产出物

```
docs/design/<模块名>/    # 每个模块一个文件夹（按 03-设计规范）
docs/plan/开发清单.md     # 全量功能点清单 + 收束节点预设
STATUS.md                # 进度状态
```

## 六、阶段五：并行 + 层级开发

### 6.1 四层架构

按以下顺序启动，前一层就绪后，后续层可并行推进：

```
第一层：装配工厂等脚手架
    │
    ├── 基础模块的抽象接口/注册机制
    ├── 数据库 migration / 数据模型
    ├── API 路由骨架
    └── 前端路由 + 布局骨架
    │ 验收：脚手架可运行，接口定义清晰
    │
    ▼
第二层：思考设计（可并行）
    │
    ├── 功能模块的详细设计（设计.md + 实现计划.md）
    └── 复杂交互的 BDD 规格
    │ 验收：设计文档经人确认
    │
    ▼
第三层：长程开发（可并行）
    │
    ├── 功能模块实现（TDD 循环）
    ├── 前端组件实现
    └── E2E 测试编写
    │ 验收：功能点验收报告 × N
    │
    ▼
第四层：推送 GitHub + 服务器插入模块（统一服务器 max）
    │
    ├── 代码推送 + PR 审查 + Tag
    ├── 基础模块部署（装配工厂）
    ├── 功能模块插入（通过注册机制）
    ├── 冒烟测试 + 健康检查
    └── 部署验证 + 回滚路径确认
    │ 验收：线上可访问 + 核心链路通过
    │
    │ 详见 [09-部署规范](09-部署规范.md)
```

### 6.2 关键规则

| 规则 | 说明 |
|------|------|
| 层间串行 | 下一层必须等上一层验收通过 |
| 层内并行 | 无依赖的模块可在同一层内并行开发 |
| 收束闸门 | 默认每 3 个功能点触发一次收束节点 |
| 阻塞管理 | 有硬阻塞的模块排到阻塞解除后再进入当前层 |
| 部署门禁 | 第四层部署前必须通过：全部测试 + 收束节点 + PR审查 + 部署前检查（详见 [09-部署规范 §二](09-部署规范.md)） |
| 模块插入顺序 | 基础模块先部署 → 功能模块后插入（详见 [09-部署规范 §七](09-部署规范.md)） |

### 6.3 基础模块优先原则

基础模块（装配工厂）必须在功能模块之前完成：

- 基础模块定义了功能模块的**接口契约**
- 功能模块依赖基础模块的抽象类/接口
- 基础模块就绪 → 多个功能模块可并行实现

## 七、阶段六：测试

### 7.1 三层测试

| 层 | 框架 | 覆盖范围 | 运行方式 | 门禁 |
|----|------|---------|---------|:---:|
| 单元测试 | pytest / vitest | 函数、类、组件 | `pytest` / `npx vitest run` | 覆盖率 ≥ 80% |
| 集成测试 | pytest + TestClient | API 端点 + 数据库 | `pytest -m integration` | 全部通过 |
| E2E 测试 | Playwright | 完整用户链路 | `npx playwright test` | 核心链路通过 |

### 7.2 真机 E2E 验证原则（不可妥协）

| 原则 | 说明 |
|------|------|
| 不接受 demo 录屏 | 录屏可以剪辑，不能证明真实运行 |
| 不接受 HTTP 200 | 200 可能返回错误数据 |
| 不接受 tsc 零错误 | 类型正确 ≠ 逻辑正确 |
| 前端必须真浏览器渲染 | Playwright 截图或手动浏览器确认 |
| 接口必须真实调用 | curl / Postman 实际调用并核对返回数据 |

### 7.3 测试在流程中的位置

- **BDD 规格**：阶段二产出，PRD 之后
- **TDD 单元测试**：阶段五（第三层）与实现同步进行
- **集成测试**：阶段五（第三层）实现完成后
- **E2E 测试**：阶段五（第三层）核心链路打通后
- **收束节点测试**：收束时全量回归

## 八、完整流程与两类任务的关系

```
思考任务区域                    长程任务区域
══════════════                  ══════════════
PRD (思考)                       BDD→TDD 测试编写 (长程)
BDD 规格 (思考)                  并行+层级开发 (长程)
技术转换 (思考)                  测试执行 (长程)
模块划分 (思考)                  部署上线 (长程) ← 统一服务器 max
        │                              │
        └──── 收束节点 ────────────────┘
              (闸门：设计确认 → 开始实现)
```

## 九、反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| PRD 未确认就开始技术转换 | 需求不清 → 技术方案反复改 | PRD 经人确认后再进入阶段三 |
| 跳过技术转换直接写代码 | 业务术语歧义 → 实现偏差 | 阶段三 9 项产出至少完成 1-5 |
| 模块划分跳过模块分类 | 混用工程模块和插件模式 → 架构混乱 | 先按 02-模块分类 判定每个模块类型 |
| 所有模块串行开发 | 周期过长 | 无依赖的模块层内并行 |
| 基础模块和功能模块同时开发 | 功能模块依赖的接口还没定义 | 基础模块先建（第一层） |
| 部署跳过 staging 直接上 prod | 没有预发布验证，问题直接暴露给用户 | dev → staging → prod 逐级推进 |
| 部署后不验证就认为完成 | "应该没问题"不是证据 | 冒烟测试 + 真浏览器验证（详见 [09-部署规范](09-部署规范.md)） |

# 08 · 汇报收束

> 更新: 2026-07-11
> 定义 AI 汇报的类型、格式、时机，以及收束节点的完整执行流程。
> 每功能点完成或收束节点触发时必读本篇。
> 更新: 2026-06-10

---

## 一、汇报类型

AI 协作中有三种汇报，按 **落盘 vs 对话** 区分：

| 类型 | 触发时机 | 落盘？ | 位置 |
|------|---------|:---:|------|
| **功能点级汇报** | 每功能点完成 | ❌ 对话内联 | 直接在对话中说，不落盘 |
| **收束报告** | 收束节点触发 | ✅ 落盘 | `docs/reports/` |
| **人要求留档** | 人明确说"写一份报告" | ✅ 落盘 | `docs/reports/` |

### 1.1 功能点级汇报（对话内联）

内容：
- 完成了什么功能点
- 代码位置（文件路径:行号）
- 测试结果（通过/失败）
- 截图（如适用）
- 下一步做什么

**不落盘** — 功能点级汇报直接在对话中说清楚即可，不写 `docs/reports/` 文件。

### 1.2 收束报告（落盘 HTML）

内容按收束节点四阶段（详见 §四），包含：
- 整理阶段的代码/文档审计结果
- 测试阶段的全量回归结果
- 审计阶段的规范红线检查结果
- 验证阶段的真机 E2E 验证结果
- 汇总：完成任务 / 未完成任务 / 阻塞项 / 后续方向

### 1.3 人要求留档

当人明确说"写一份报告"、"生成验收报告"时，按 [04-长程开发 §五](04-长程开发.md) 的 8 要素格式落盘。

## 二、Worklog（工作日志）

### 2.1 格式

文件命名：`worklogs/YYYY-MM-DD_<主题>.md`

```markdown
# <日期> · <主题>

> 关联：<功能点编号 / 模块名>
> Commit: <hash>（如适用）

## 完成了什么
- [x] ...
- [x] ...

## 关键决策
- 决策 1：<选择>（原因：<一句话>）
- 决策 2：...

## 遇到的问题
- 问题 1：<描述> → <解决方案>
- 问题 2：...

## 下一步
- [ ] ...
- [ ] ...
```

### 2.2 更新时机

**每功能点完成时即时更新**，不积压到收束节点一次性补。

### 2.3 红线

- ❌ 不能跳过一个功能点的 worklog
- ❌ 不能收束节点时一次性补多个功能点的 worklog
- ❌ commit message 必须引用 worklog 文件名（`06-documentation` 红线 + pre-commit 钩子自动检查）

## 三、ADR（架构决策记录）

### 3.1 触发条件

任何涉及"选 A 还是选 B"的**架构/设计层面**的取舍。详见 [04-长程开发 §二](04-长程开发.md)。

### 3.2 格式

文件命名：`worklogs/decisions/NNNN-<简短标题>.md`

```markdown
# ADR-NNNN: <标题>

> 日期: YYYY-MM-DD
> 状态: 提议 / 已采纳 / 已废弃 / 已替代

## 背景
<为什么需要做这个决策>

## 决策
<选择了什么方案>

## 考虑的替代方案
| 方案 | 优点 | 缺点 |

## 后果
<正面/负面影响>

## 相关
<关联的 ADR / 设计文档 / PR>
```

### 3.3 ADR 状态流转

```
提议 ──→ 已采纳
  │
  └──→ 已废弃（决策被推翻）
         │
         └──→ 已替代（被新 ADR 替代，标注替代者编号）
```

## 四、STATUS.md 更新

### 4.1 格式

```markdown
# 项目进度状态

> 最后更新: YYYY-MM-DD
> 总功能点: N / 已完成: N

## 功能点进度

| # | 功能点 | 优先级 | 状态 | 完成日期 | 备注 |
|---|--------|:---:|:---:|---------|------|
| 1 | ... | P0 | ✅ | YYYY-MM-DD | ... |
| 2 | ... | P0 | 🔄 | — | ... |
| 3 | ... | P1 | ⬜ | — | ... |

## 收束节点历史

| 节点 | 触发条件 | 执行日期 | 状态 | 报告 |
|------|---------|---------|:---:|------|
| v0.1 | 功能点 1-3 完成 | YYYY-MM-DD | ✅ 已审计 | [报告](docs/reports/...) |
| v0.2 | 功能点 4-6 完成 | — | ⏳ 待执行 | — |

## 当前阻塞

| 阻塞项 | 影响功能点 | 预计解除 |
|--------|-----------|---------|
```

### 4.2 更新时机

- 每功能点完成时：更新对应功能点状态
- 收束节点执行后：更新收束节点历史
- 阻塞出现/解除时：更新当前阻塞

## 五、开发清单更新

### 5.1 文件位置

`docs/plan/开发清单.md`

### 5.2 内容

```markdown
# 开发清单

> 版本: vX.X · 最后更新: YYYY-MM-DD
> 收束节点预设: 每 3 个功能点

## 功能点清单

| # | 功能点 | 模块 | 优先级 | 阶段 | 状态 | 估时 | 阻塞 |
|---|--------|------|:---:|:---:|:---:|:---:|------|

## 收束节点预设

| 节点 | 触发条件 | 状态 |
|------|---------|:---:|
| v0.1 | 功能点 1-3 | ⏳ |
| v0.2 | 功能点 4-6 | ⬜ |
```

### 5.3 更新时机

- 新增功能点时：加行
- 功能点完成时：更新状态
- 收束节点执行后：更新节点状态

## 六、收束节点（完整执行流程）

收束节点是流程中的**硬闸门**。到达预设节点时，AI **拒绝开始下一个功能点**，并提醒人触发收束。

### 6.1 四阶段

```
阶段 1：整理              阶段 2：测试              阶段 3：审计              阶段 4：验证
     │                        │                        │                        │
     ▼                        ▼                        ▼                        ▼
代码 diff 汇总          全量单元测试回归         逐条规范红线检查          真浏览器 E2E 验证
worklog 完整性检查      集成测试回归             安全扫描                  核心链路手动走通
文档同步检查            E2E 核心链路             依赖漏洞检查              截图存档
STATUS 一致性检查       覆盖率报告               API 契约检查
ADR 归档检查
     │                        │                        │                        │
     └────────────────────────┴────────────────────────┴────────────────────────┘
                                          │
                                          ▼
                                    收束报告（HTML）
                                    ADR（如有新决策）
                                    更新 STATUS.md
                                    更新开发清单
```

### 6.2 阶段一：整理

| 检查项 | 说明 |
|--------|------|
| 代码 diff 汇总 | 本次收束区间内所有改动的汇总 |
| worklog 完整性 | 每个功能点是否有对应的 worklog |
| 文档同步 | `docs/specs/` 和 `docs/design/` 是否与代码一致 |
| STATUS 一致性 | `STATUS.md` 进度与实际是否一致 |
| ADR 归档 | 本区间内的架构决策是否已落 ADR |

### 6.3 阶段二：测试

| 检查项 | 命令 / 方式 |
|--------|-----------|
| 单元测试全量回归 | `pytest` / `npx vitest run` |
| 集成测试回归 | `pytest -m integration` |
| E2E 核心链路 | `npx playwright test` |
| 覆盖率报告 | 有业务代码时 `pytest --cov` 且按项目门槛；纯文档/契约仓不得用无意义 `src` 覆盖率伪装质量 |

### 6.4 阶段三：审计

| 检查项 | 依据 |
|--------|------|
| 规范红线逐条检查 | `conventions/CLAUDE-规范导航.md` 红线总表 |
| 一致性事实矩阵 | `python scripts/check_consistency.py --threshold 80`，逐维真实通过数/总数，低于 80% 拒绝收束 |
| 强制性故障注入 | `python scripts/check_enforcement.py --threshold 90`，至少 10 项隔离 Git 违规，低于 90% 拒绝收束 |
| 安全扫描 | gitleaks（密钥泄露） |
| 依赖漏洞 | `pip audit` / `npm audit` |
| API 契约 | spectral OpenAPI lint |

### 6.5 阶段四：验证

| 检查项 | 说明 |
|--------|------|
| 真浏览器 E2E | 核心用户链路在真实浏览器中手动走通 |
| 接口真实调用 | 核心 API 端点用 curl/Postman 调用并核对返回数据 |
| 初始化真实执行 | 脚手架类交付必须在仓库外空目录运行完整 `--install`，再从目标自身执行 hooks、verify 与测试 |
| 截图存档 | 关键页面/操作截图存入收束报告 |

### 6.6 阶段五：汇总产出

收束节点完成后产出：

| # | 产出 | 说明 |
|---|------|------|
| 1 | 收束报告 HTML | 含四阶段结果 + 截图 + Mermaid 图表 |
| 2 | ADR | 本区间内的新架构决策 |
| 3 | STATUS.md 更新 | 收束节点历史 + 功能点状态 |
| 4 | 开发清单更新 | 收束节点状态 |

### 6.7 收束节点与人的关系

| 角色 | 职责 |
|------|------|
| **人** | 触发收束、审核收束报告、签核通过 |
| **AI** | 执行四阶段、生成报告、到闸门拦截、提醒人触发 |

> **AI 不替人做收束** — 收束是人的质量门禁，AI 负责执行和汇报，人负责审核和签核。

## 七、汇报频率速查

| 事件 | 做什么 |
|------|--------|
| 功能点完成 | 功能点级汇报（对话内联）+ 更新 worklog + 更新 STATUS |
| 收束节点到达 | AI 拦截 → 人触发 → 四阶段执行 → 收束报告落盘 |
| 人要求报告 | 按 8 要素格式落盘 `docs/reports/` |
| 设计完成 | 思考任务 Step 5 记录整理（= 一次收束节点） |
| 项目交付 | 项目级验收 Checklist（[07-验收交付 §四](07-验收交付.md)） |

## 八、反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| 多个功能点共用一个 worklog | 无法追溯每个功能点的决策和问题 | 每功能点独立 worklog |
| 收束节点跳过直接继续 | 积压问题在项目后期集中爆发 | AI 拦截，必须收束 |
| 收束报告只有测试结果没有审计 | 规范红线可能被违反但未发现 | 四阶段缺一不可 |
| AI 替人签核收束 | 越界——质量门禁必须人来把关 | AI 执行 + 汇报，人审核 |
| 功能点级汇报也落盘 | `docs/reports/` 堆积大量小报告 | 功能点级对话说，收束级才落盘 |

# 02 · 模块分类

> 定义两类模块（工程模块 / 基础+功能模块）的判定标准、特征、示例和设计影响。
> 做模块划分或架构设计前必读本篇。
> 更新: 2026-06-10

---

## 一、为什么需要模块分类

不同的模块有不同的**接口形态**和**拼接方式**。错误分类会导致：

- 把插件当服务调 → 循环依赖
- 把服务当插件注册 → 过度抽象
- 模块边界模糊 → 改动面失控

**核心区分维度**：模块之间是**数据契约**（对等拼接）还是**插件契约**（插槽-插件）。

---

## 二、工程模块（Engineering Module）

### 2.1 定义

具有标准化输入/输出接口的独立模块，**可以直接拼接在一起**形成完整系统。

### 2.2 特征

| 维度 | 说明 |
|------|------|
| 接口形态 | **数据契约**：固定的输入类型 → 固定的输出类型 |
| 连接方式 | 函数调用 / REST API / 消息队列 |
| 拼接方式 | 直接调用，不需要中间层或适配器 |
| 耦合程度 | 低——接口相同即可替换 |
| 替换成本 | 低——换一个实现相同接口的模块即可 |
| 典型场景 | REST API 路由、UI 组件、工具函数库 |

### 2.3 判定标准

> **去掉模块 A，模块 B 是否只需要换一个实现了相同接口的模块 C 就能继续工作？**
> 是 → 工程模块。否 → 检查是否为基础+功能模块。

### 2.4 示例（AgentHub）

| 模块 | 接口 | 为什么是工程模块 |
|------|------|-----------------|
| `sessions.py`（Session 路由） | `GET/POST/PATCH /api/sessions` | 标准 REST CRUD，换一个 ORM 不影响调用方 |
| `MessageBubble.tsx` | `Props: { message: Message }` | Props 接口固定，换一个消息渲染组件直接替换 |
| `auth.py`（认证中间件） | `depends → User` | 输入 request → 输出 User，任何认证方案都可替换 |
| `s3_storage.py` | `upload(file) → url` | 换云存储只需实现相同接口 |

## 三、基础模块 + 功能模块（Base + Functional Module）

### 3.1 定义

**基础模块**提供装配工厂（Assembly Factory），**功能模块**插入装配工厂后获得能力。本质是**插件系统**或**策略模式**在模块级别的体现。

### 3.2 特征

| 维度 | 说明 |
|------|------|
| 接口形态 | **插件契约**：功能模块实现基础模块定义的抽象类/接口/协议 |
| 连接方式 | 注册 / 依赖注入 / 发现机制 |
| 拼接方式 | 基础模块不关心具体实现，只定义插槽 |
| 耦合程度 | 中——功能模块必须实现完整协议 |
| 替换成本 | 中——需实现完整接口，但基础模块不受影响 |
| 典型场景 | 适配器层、策略模式、插件系统、中间件链 |

### 3.3 判定标准

> **模块 B 是否实现了模块 A 定义的抽象接口/协议，并通过注册机制接入？**
> 是 → 功能模块（B）插入基础模块（A）。否 → 检查是否为工程模块。

### 3.4 示例（AgentHub）

| 基础模块 | 定义的插槽 | 功能模块（插入实现） |
|---------|-----------|-------------------|
| `Orchestrator` | `schedule` / `dispatch` / `aggregate` | —（自身是协调器） |
| `BaseAdapter`（抽象类） | `send_message(input) → stream_response` | `ClaudeCodeAdapter` / `CodexAdapter` / `OpenCodeAdapter` |
| `AdapterRegistry` | `register(adapter)` / `get(name)` | 各 Adapter 注册时调用 |
| `AgentFactory` | Agent 创建流程插槽 | `ConversationalAgentCreator` / `FormWizardAgentCreator` |

### 3.5 装配工厂示意

```
┌─────────────────────────────────────┐
│         基础模块：AdapterRegistry    │
│                                     │
│  register(adapter: BaseAdapter)     │  ← 插槽定义
│  get(name: str) → BaseAdapter       │
│  list() → BaseAdapter[]             │
└─────────────────────────────────────┘
              ▲        ▲        ▲
              │ 实现    │ 实现    │ 实现
              │        │        │
    ┌─────────┴──┐ ┌──┴──────┐ ┌┴──────────┐
    │ ClaudeCode  │ │ Codex   │ │ OpenCode   │
    │ Adapter     │ │ Adapter │ │ Adapter    │
    └────────────┘ └─────────┘ └────────────┘
        功能模块       功能模块      功能模块
```

## 四、对比总结

| 维度 | 工程模块 | 基础模块 + 功能模块 |
|------|---------|-------------------|
| 关系 | 对等拼接 | 插槽-插件 |
| 耦合方式 | 数据契约（输入/输出类型） | 接口/抽象类继承 + 注册 |
| 接入方式 | 直接调用 / HTTP | 注册 / 依赖注入 |
| 基础模块的独立性 | — | 可以独立存在，定义插槽即可 |
| 功能模块的独立性 | 完全独立 | 必须依附基础模块的协议 |
| 替换成本 | 低（接口相同即可） | 中（需实现完整协议） |
| 新增成本 | 低（新增调用方即可） | 中（需实现协议 + 注册） |
| 典型场景 | REST API 路由、UI 组件 | 适配器层、策略模式、插件系统 |

---

## 五、设计决策表

| 场景 | 选择 | 原因 |
|------|:---:|------|
| 多个外部平台需要统一接入 | 基础+功能 | 适配器模式天然适合多平台统一接口 |
| 一个模块只被一个调用方使用 | 工程模块 | 过度抽象增加复杂度 |
| 预期未来会新增同类实现 | 基础+功能 | 预留插槽降低未来改动成本 |
| 不预期扩展，接口稳定 | 工程模块 | 简单直接 |
| 模块需要在运行时动态发现/加载 | 基础+功能 | 注册机制支持运行时发现 |
| 模块是编译时确定的依赖 | 工程模块 | 不需要运行时灵活性 |

## 六、与设计规范的关系

模块分类影响设计文档的写法（详见 [03-设计规范](03-设计规范.md)）：

| 模块类型 | 设计.md 重点 | 实现计划.md 重点 |
|---------|-------------|----------------|
| 工程模块 | 数据契约（输入/输出类型定义） | API 端点 + 数据模型 |
| 基础模块（装配工厂） | 抽象接口定义 + 注册机制 | 基础模块先建（脚手架层） |
| 功能模块（插件） | 协议实现 + 注册方式 | 依赖基础模块，排在后面 |

## 七、反模式

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| 把 REST API 路由做成插件系统 | REST 路由本身就是标准化接口，不需要再套一层抽象 | 工程模块，直接调用 |
| 多个 Adapter 之间互相调用 | 功能模块之间不应有直接依赖——它们通过基础模块协作 | 基础模块作为中介 |
| 基础模块包含具体实现逻辑 | 基础模块只定义插槽，不关心实现 | 具体逻辑下沉到功能模块 |
| 把所有模块都做成插件 | 过度抽象 | 只对"会有多个实现"的模块用插件模式 |

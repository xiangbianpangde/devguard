# 架构设计规范

> 定义软件系统的分层结构、模块边界和依赖方向，确保系统可维护、可扩展。

---

## 适用范围

- **项目类型**：所有软件项目（前端/后端/全栈/移动端）
- **技术栈**：语言无关，适用于任何面向对象/函数式编程语言
- **团队规模**：1 人以上均适用

---

## 一、核心原则

### 原则 1：单一职责

每个模块/类/函数只做一件事，做好一件事。

**为什么**：职责混杂的模块修改一个功能可能破坏另一个，且难以测试。单一职责让代码意图清晰，修改影响面可控。

### 原则 2：开闭原则

对扩展开放，对修改关闭。新增功能通过扩展实现，而非修改已有代码。

**为什么**：每修改一次已有代码，都有引入回归 bug 的风险。扩展新代码可以隔离风险，同时保持旧功能的稳定性。

### 原则 3：依赖倒置

高层模块不依赖低层模块，两者都依赖抽象。抽象不依赖细节，细节依赖抽象。

**为什么**：如果不倒置依赖，底层数据库/缓存变更会直接冲击上层业务逻辑。面向接口编程让底层可以被替换而不影响业务。

### 原则 4：接口隔离

不应强迫使用者依赖他们不使用的接口。大接口拆分为多个小接口。

**为什么**：一个类实现一个"万能接口"时，修改任一方法都会触发所有依赖者重新编译/测试。小接口降低耦合面。

## 二、具体规则

### 2.1 分层架构

```
表现层 (UI/API)
    |
应用层 (业务编排)
    |
领域层 (核心业务逻辑)
    |
基础设施层 (数据库/缓存/外部服务)
```

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 上层依赖下层 | 表现层 → 应用层 → 领域层，依赖单向 | 必须 |
| 下层不依赖上层 | 基础设施层只提供实现，不反向引用上层类型 | 必须 |
| 领域层不依赖框架 | 领域层导入应限于标准库 + 领域接口，不 import Web 框架/ORM | 必须 |
| 跨层调用走应用层 | 表现层不直接调用基础设施层（需经过应用层） | 建议 |

### 2.2 各层职责

| 层 | 职责 | 不应包含 |
|----|------|---------|
| 表现层 | 接收请求、参数校验、返回响应 | 业务逻辑 |
| 应用层 | 用例编排、事务管理、权限校验 | 领域规则 |
| 领域层 | 实体定义、业务规则、领域服务 | 数据库操作细节 |
| 基础设施层 | 数据库访问、缓存、外部 API 调用 | 业务判断 |

### 2.3 模块划分

**按业务领域划分（推荐大项目）**

```
src/
├── user/          # 用户模块
│   ├── api/       # 接口层
│   ├── service/   # 业务层
│   └── model/     # 数据模型
├── order/         # 订单模块
├── product/       # 商品模块
└── common/        # 公共模块
```

**按技术层次划分（传统/小项目）**

```
src/
├── controllers/   # 控制器
├── services/      # 服务层
├── models/        # 数据模型
├── utils/         # 工具函数
└── config/        # 配置
```

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 项目大/业务复杂 → 按业务领域划分 | 模块自带内聚，新人容易定位代码 | 建议 |
| 项目小/功能简单 → 按技术层次划分 | 减少目录层级，快速上手 | 建议 |
| 公共代码放 common/ 或 shared/ | 避免跨业务模块的直接引用 | 必须 |

### 2.4 依赖管理

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 禁止循环依赖 | A → B → A 不允许；如出现，提取公共部分到独立模块 | 必须 |
| 依赖方向：高内聚 → 低内聚 → 通用工具 | 稳定的在上层，多变的在下层 | 必须 |
| 第三方库：标准库 > 社区主流 > 自研 | stars > 1000, 持续维护的社区库优先 | 建议 |

### 2.5 配置管理

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 密钥/密码通过环境变量注入 | 不在代码或配置文件中硬编码 | 必须 |
| 配置文件提供 `.example` 模板 | 真实配置文件不入版本库 | 必须 |
| 区分开发/测试/预发布/生产环境 | 每个环境使用独立配置 | 必须 |
| 非敏感配置可放入配置文件 | 如日志级别、超时时间 | 建议 |

### 2.6 技术选型

| 规则 | 说明 |
|------|------|
| 成熟稳定 > 新奇酷炫 | 已被大量项目验证的方案优先 |
| 社区活跃 > 功能强大 | 遇到问题有地方问 > 功能多但没人用 |
| 团队熟悉 > 性能最优 | 能维护的 > 理论上更快的 |
| 够用就好 > 过度准备 | 当前需求不需要的技术栈不引入 |
| 选型结果写入 ADR | 决策理由可追溯，方便后来者理解 |

## 三、反模式

### ❌ 反模式 1：循环依赖

```python
# user/service.py
from order.service import OrderService

# order/service.py
from user.service import UserService  # 循环依赖！
```

**问题**：两个模块互相引用，导致无法独立测试、无法独立部署，且初始化顺序不确定容易出错。

### ✅ 正确做法：提取公共接口

```python
# common/interfaces.py
class INotificationService:
    def notify(self, user_id: str, message: str): ...

# user/service.py
from common.interfaces import INotificationService

# order/service.py
from common.interfaces import INotificationService
```

**理由**：通过接口抽象，两个模块都只依赖公共抽象层，解除了直接循环引用。

---

### ❌ 反模式 2：跨层调用

```python
# controllers/user_controller.py
def create_user(request):
    db.execute("INSERT INTO users ...")  # 表现层直接操作数据库！
```

**问题**：控制器跳过了应用层和领域层，业务规则无处安放。数据库变更会直接冲击表现层。

### ✅ 正确做法：逐层调用

```python
# controllers/user_controller.py
def create_user(request):
    service = UserService()
    return service.register(request.data)

# application/user_service.py
class UserService:
    def register(self, data):
        user = User.create(data)        # 领域层
        self.user_repo.save(user)       # 基础设施层
        return user
```

**理由**：每层只做自己职责内的事，表现层不知道数据库存在。数据库换成 MongoDB 只需改基础设施层。

---

### ❌ 反模式 3：配置硬编码

```python
# config.py
DATABASE_URL = "postgresql://admin:secret123@prod-db:5432"
API_KEY = "sk-abc123def456"
```

**问题**：密钥泄露到版本历史，更换密钥需改代码重新部署。

### ✅ 正确做法：环境变量 + 模板

```python
# config.py
import os
DATABASE_URL = os.environ["DATABASE_URL"]
API_KEY = os.environ["API_KEY"]

# config.example.py （提交到版本库）
DATABASE_URL = "postgresql://user:password@localhost:5432/db"
API_KEY = "your-api-key-here"
```

**理由**：密钥与代码分离，不同环境注入不同值，`.example` 文件告诉新人需要配置什么。

## 四、示例

### 场景 1：新项目模块划分

**需求**：一个电商系统，包含用户、商品、订单、支付四个业务域。

| | 反例 | 正例 |
|---|------|------|
| 结构 | `src/controllers/user_controller.py`, `src/models/user.py` 等全部平铺 | `src/user/api/`, `src/user/service/`, `src/user/model/` 等按业务域划分 |
| 说明 | 功能多了之后 controllers 和 models 下各有几十个文件，找代码困难 | 每个业务域自包含，关闭一个业务域只需删除一个目录 |

### 场景 2：分层后新增功能

**需求**：订单模块新增"满减优惠"规则。

| | 反例 | 正例 |
|---|------|------|
| 做法 | 在订单 controller 的 `create_order` 方法里直接加 if-else 判断打折 | 在领域层的 `Order` 实体中实现 `apply_discount()` 方法，controller 调用应用层 `OrderService.place_order()` |
| 说明 | 优惠逻辑散落在表现层，换一个前端框架就得重写 | 领域层独立于框架，Web/CLI/API 三种入口共享同一套业务规则 |

## 五、工具推荐

| 工具 | 用途 | 推荐度 |
|------|------|--------|
| `pydeps` / `dependency-cruiser` | 检查模块依赖图，发现循环依赖 | 推荐 |
| `ArchUnit` (Java) / `pytest-arch` (Python) | 架构规则自动化测试 | 推荐 |
| `mermaid.js` | 架构图即代码，可放在 Markdown 中 | 可选 |
| `ADR Tools` | 管理架构决策记录 (docs/decisions/) | 可选 |

---

## 六、检查清单

- [ ] 是否有循环依赖？（层与层之间、模块与模块之间）
- [ ] 领域层是否依赖了框架/ORM？（只能依赖标准库 + 接口抽象）
- [ ] 是否有跨层调用？（表现层直接调基础设施层）
- [ ] 敏感信息是否从代码中移除？（密钥仅通过环境变量注入）
- [ ] 新模块的划分方式是否符合项目规模？
- [ ] 技术选型是否记录了决策理由（ADR）？
- [ ] 配置文件是否提供了 `.example` 模板？

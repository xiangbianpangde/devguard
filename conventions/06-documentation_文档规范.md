# 文档规范

> 定义项目文档的类型、结构和维护策略，确保文档与代码同步、新人可快速上手。

---

## 适用范围

- **项目类型**：所有软件项目
- **文档类型**：README、CHANGELOG、API 文档、设计文档、代码注释
- **团队规模**：1 人以上均适用（文档是跨时间的沟通）

---

## 一、核心原则

### 原则 1：文档是产品的一部分

文档不是"写完代码后的补充"，而是交付物的组成部分。没有文档的功能是不完整的功能。

**为什么**：没有 README 的仓库，新人拿到后连怎么跑起来都不知道。没有 CHANGELOG 的发布，用户不知道更新了什么。文档缺失会让项目看起来不专业，降低采用率和贡献率。

### 原则 2：贴近代码，而非远离代码

文档应尽可能放在代码仓库中（Markdown 文件），与代码一起版本管理、一起 Review、一起发布。避免文档和代码分离（如单独的 Wiki）。

**为什么**：与代码分离的文档很快就会过时——改了代码但忘了去 Wiki 更新。放在仓库里的文档可以在 PR 中同步修改，Review 者能检查文档是否与变更匹配。

### 原则 3：结构化优先于篇幅长

短而结构清晰的文档比长篇大论更有用。使用标题层级、表格、代码块、列表等让读者能快速扫描定位。

**为什么**：没有人会从头到尾读一份 50 页的文档。结构化的文档让读者能 30 秒内定位到需要的章节，提升了文档的"命中率"。

### 原则 4：定期修剪，防止腐烂

过时的文档比没有文档更危险。每季度检查文档与代码的一致性，及时更新或归档。

**为什么**：过时的 API 文档会让调用方按错误的方式对接，过时的架构文档会让新人理解错误的系统结构。定期修剪是文档质量的唯一保障。

## 二、具体规则

### 2.1 README.md

每个项目根目录必须有 README.md，读完前 30 秒让读者理解项目是什么、怎么用。

```markdown
# 项目名称
一句话描述项目（< 30 字，说明做什么、给谁用）

## 快速开始
# 克隆 → 安装依赖 → 运行
git clone ...
npm install
npm start

## 功能特性
- 特性 1
- 特性 2

## 技术栈
- 前端: React 18 + TypeScript
- 后端: Python 3.12 + FastAPI
- 数据库: PostgreSQL 16

## 目录结构
src/
├── components/   # UI 组件
├── services/     # 业务逻辑
└── utils/        # 工具函数

## 贡献指南
见 CONTRIBUTING.md

## 许可
MIT
```

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 必须有 README.md | 无 README 的项目视为未完成 | 必须 |
| 必须有"快速开始" | 拷贝粘贴就能跑起来，不超过 5 条命令 | 必须 |
| 使用 Markdown | 纯文本，Git 友好 | 必须 |

### 2.2 CHANGELOG.md

每次发布必须更新，遵循 Keep a Changelog 格式。

```markdown
## v1.2.0 (2026-05-20)

### Added
- 用户头像上传功能 (#45)
- 订单导出 CSV 功能 (#48)

### Changed
- 登录页 UI 重新设计
- 默认分页大小从 10 调整为 20

### Fixed
- 修复 token 过期后无限循环刷新的问题 (#42)
- 修复移动端日期选择器样式错乱 (#44)

### Deprecated
- `GET /api/v1/users/export` 将于 v1.4.0 移除，请迁移到 `POST /api/v2/exports`

### Security
- 升级依赖 `express` 到 4.19.0 修复 CVE-2024-XXX
```

| 分类 | 说明 |
|------|------|
| Added | 新增功能 |
| Changed | 已有功能的变更 |
| Fixed | Bug 修复 |
| Deprecated | 即将移除的功能 |
| Removed | 已移除的功能 |
| Security | 安全修复 |

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 每次发布必须更新 CHANGELOG | 版本号 + 日期 + 变更分类 | 必须 |
| 按 Keep a Changelog 6 分类 | Added/Changed/Fixed/Deprecated/Removed/Security | 必须 |
| 关联 Issue/PR 编号 | `(#42)` 方便追溯 | 建议 |

### 2.3 CONTRIBUTING.md

```markdown
## 开发环境搭建
# 必需的依赖和工具版本

## 开发流程
1. Fork 仓库
2. 创建功能分支
3. 编写代码 + 测试
4. 提交 PR

## 代码规范
见 conventions/02-coding_代码编写规范.md

## PR 规范
- 标题格式: feat(scope): description
- 描述包含: 做了什么 / 为什么 / 怎么验证
- 确保 CI 通过
```

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 开源项目必须 | 降低外部贡献者的入门成本 | 必须 |
| 内部项目建议 | 新成员入职时的操作手册 | 建议 |

### 2.4 代码注释

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 公共 API 必须有注释 | 参数类型、返回值、异常、使用示例 | 必须 |
| 复杂算法用自然语言解释 | 说明为什么选这个算法，而非重复代码 | 必须 |
| 注释与代码同步维护 | 改代码时必须同步更新注释 | 必须 |

```python
def create_order(user_id: str, product_id: str, quantity: int) -> Order:
    """创建订单。
    
    Args:
        user_id: 用户唯一标识
        product_id: 商品唯一标识
        quantity: 购买数量，必须大于 0
    
    Returns:
        创建的订单对象，状态为 pending
    
    Raises:
        ValueError: 库存不足
        UserNotFoundError: 用户不存在
        ProductNotFoundError: 商品不存在
    
    使用示例:
        order = create_order("user_123", "prod_456", 2)
    """
```

### 2.5 设计文档

| 何时写 | 何时不写 |
|--------|---------|
| 需求不清晰，需要多方对齐 | 简单 CRUD 功能 |
| 复杂功能（涉及 3+ 模块） | 纯前端 UI 调整 |
| 重大架构变更 | 已有明确方案的小修改 |

**设计文档模板**：

```markdown
# 设计文档：{功能名称}

## 背景
为什么要做这个功能？当前有什么问题？

## 目标
做完后达到什么效果？（可量化）

## 方案
技术方案描述 + 架构图（Mermaid）

## 影响范围
涉及哪些模块/服务/数据库表？

## 风险
已知风险和缓解措施
```

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 设计文档放 `docs/design/` | 或 `docs/decisions/` (ADR) | 建议 |
| 使用 Mermaid 画图 | Markdown 内嵌，无需外部工具 | 建议 |

### 2.6 文件命名

| 类型 | 格式 | 示例 |
|------|------|------|
| 普通文档 | 描述性名称，kebab-case | `架构设计.md`、`deployment-guide.md` |
| 日期相关 | `YYYY-MM-DD-描述` | `2026-05-26-会议纪要.md` |
| 版本相关 | `v<版本号>-描述` | `v2.0-迁移指南.md` |
| 工作日志 | `YYYY-MM-DD_描述.md` | `2026-05-26_功能A开发.md` |

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 禁止空格，用 `-` 或 `_` 连接 | 避免 URL 编码和 shell 转义问题 | 必须 |
| 禁止特殊字符 | 只允许字母、数字、`-`、`_`、`.` | 必须 |

### 2.7 文档维护

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 功能变更时同步更新文档 | 代码和文档在同一个 PR 中修改 | 必须 |
| 每季度检查文档一致性 | 逐份文档对照代码现状 | 建议 |
| 过时文档移入 `archive/` | 不直接删除，保留历史参考 | 建议 |
| 标注废弃日期 | 过时文档头部标注 `> ⚠️ 已废弃: YYYY-MM-DD，替代文档: xxx` | 建议 |

## 三、反模式

### ❌ 反模式 1：README 是空的或有名无实

```markdown
# my-project

A cool project.

## Install

npm install

## Usage

See code.
```

**问题**：读者连这个项目做什么都不知道。"A cool project" 比空白好不了多少。外部开发者直接关掉标签页，内部新人只能去翻代码。

### ✅ 正确做法：告诉读者 5 个问题

```markdown
# 订单管理系统

电商平台的订单处理微服务，支持创建、支付、退款、物流追踪。

## 快速开始
git clone ... && docker-compose up -d
打开 http://localhost:8080/docs 查看 API 文档

## 功能特性
- 订单 CRUD + 状态机（待支付→已支付→已发货→已完成）
- 支付宝/微信支付集成
- 30 分钟未支付自动取消
- 物流单号实时查询

## 技术栈
Python 3.12 + FastAPI + PostgreSQL + Redis + RabbitMQ

## 目录结构
...（略）

## API 文档
http://localhost:8080/docs
```

**理由**：30 秒内回答：做什么、怎么跑、有什么功能、用什么技术栈、API 在哪。读者能立即决定这个项目是否相关。

---

### ❌ 反模式 2：CHANGELOG 是 Git Log 的复制

```markdown
## v1.2.0
- feat: add user avatar
- fix: fix token refresh
- chore: update deps
- refactor: clean up code
- docs: update readme
```

**问题**：Git log 是针对开发者的，CHANGELOG 是针对用户的。用户不关心"重构了代码"或"更新了依赖"，他们想知道"这次更新对我有什么影响"。

### ✅ 正确做法：用用户的视角写

```markdown
## v1.2.0 (2026-05-20)

### Added
- 用户可上传自定义头像（支持 JPG/PNG，最大 2MB）
- 订单列表支持导出 CSV 文件

### Fixed
- 修复登录后一段时间自动退出的问题
- 修复手机端日期选择器显示异常

### Security
- 更新底层框架依赖，修复已知安全漏洞（CVE-2024-XXX）
```

**理由**：用户能直接看到哪些新功能可用、哪些 bug 已修复。技术细节（如重构、依赖更新）只在影响用户时提及。

---

### ❌ 反模式 3：代码与注释不一致

```python
def get_user(user_id: str) -> User:
    """通过用户名查询用户"""  # 实际是 user_id 不是 username
    return db.query("SELECT * FROM users WHERE username = ?", user_id)
```

**问题**：注释说"通过用户名查询"，但变量名叫 `user_id`，实际查询的也是 `username` 字段。读者不知道该信注释还是信代码。过时的注释比没有注释更危险。

### ✅ 正确做法：注释与代码同步，说明意图

```python
def get_user(user_id: str) -> User:
    """通过用户 ID 查询用户信息。
    
    Args:
        user_id: 用户唯一标识（UUID 格式）
    
    Returns:
        用户对象
    
    Raises:
        UserNotFoundError: ID 对应的用户不存在
    """
    return db.query("SELECT * FROM users WHERE id = ?", user_id)
```

**理由**：代码改了，注释同步更新。参数名、字段名、行为描述三者一致，读者无需猜测。

## 四、示例

### 场景 1：新项目创建 README

**需求**：一个新启动的后端项目，需要 README 让同事 5 分钟跑起来。

| | 反例 | 正例 |
|---|------|------|
| README | 只有项目名 + "TBD" | 包含：项目定位、5 条命令跑起来、功能列表、技术栈、目录结构、API 文档链接 |
| 结果 | 同事无法自行启动，每来一个人就要口述一遍 | 同事按 README 操作即可启动，减少重复沟通 |

### 场景 2：发布新版本时维护 CHANGELOG

**需求**：v1.3.0 包含 2 个新功能、1 个破坏性变更、3 个 bug 修复。

| | 反例 | 正例 |
|---|------|------|
| 做法 | 发行说明写"修复若干问题，优化体验" | 按 Added/Changed/Fixed 分类列出每项变更，破坏性变更单独标注迁移步骤 |
| 结果 | 升级的用户不知道这次更新有哪些风险，出问题后无法回溯 | 用户根据 CHANGELOG 决定是否升级，破坏性变更提前准备迁移 |

## 五、工具推荐

| 工具 | 用途 | 推荐度 |
|------|------|--------|
| Markdown + Mermaid | 文档编写 + 流程图/时序图/架构图 | 推荐 |
| OpenAPI 3.x (Swagger) | API 文档自动生成，优于手写 | 推荐 |
| Storybook | 前端组件文档 + 交互式预览 | 推荐 |
| conventional-changelog | 从 Conventional Commits 自动生成 CHANGELOG | 可选 |
| VitePress / Docusaurus | 文档站点生成（> 5 份文档时推荐） | 可选 |

## 六、检查清单

- [ ] 项目根目录是否有 README.md？（30 秒内能理解项目做什么）
- [ ] README 是否包含"快速开始"章节？（拷贝粘贴就能跑）
- [ ] CHANGELOG 是否按 Keep a Changelog 格式维护？
- [ ] 每次发布是否更新了 CHANGELOG？
- [ ] 公共 API 函数是否有完整的 docstring？
- [ ] 过时的文档是否已更新或归档（而非放任不管）？
- [ ] 代码注释是否与代码行为一致？
- [ ] 文件命名是否遵循 kebab-case（无空格、无特殊字符）？
- [ ] 复杂功能（3+ 模块）是否有设计文档？
- [ ] 设计文档是否有背景/目标/方案/影响/风险五段？

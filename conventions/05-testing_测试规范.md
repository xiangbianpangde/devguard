# 测试规范

> **本规范是 [ai-workflow 04-长程开发](ai-workflow_AI协作开发流程/04-长程开发.md) 两个环节的细化**：
> - 细化 **§2.1 TDD 自检**（先写测试 → 实现 → 重构）—— Opus 保证代码按 BDD 预期运行的手段。
> - 细化 **§2.2 可观测验证** —— 测试断言 / 覆盖率全绿是「重构、算法、配置」类功能点的合格证据。
>
> **档位裁剪**：轻量项目不强制覆盖率门禁，但红线（独立 / 不只 happy path）仍守；标准 / 团队守覆盖率目标。
> 更新: 2026-07-11

---

## 一、红线（必守）

| # | 红线 | 怎么自动抓 |
|---|------|-----------|
| 1 | 测试独立，不依赖执行顺序、不共享全局状态 | `pytest -p no:randomly` 乱序跑仍全绿 |
| 2 | 只 Mock 外部边界（DB/HTTP/文件/时间），禁止 Mock 项目内部模块 | 审查 |
| 3 | 覆盖正常 + 边界 + 异常路径，不只测 happy path | 审查 + 分支覆盖率 |
| 4 | 无 flaky test（时过时不过），出现立即修复，禁止「重跑就行」 | CI 重试检测 |
| 5 | **开发按 BDD+TDD 流程**：先 BDD 规格（docs/specs/）→ 先写失败测试（红）→ 最小实现（绿）→ 重构。每个功能点的 BDD 场景必须逐条转成测试用例 | 审查 + 测试用例 ↔ BDD 场景一一对应 |

---

## 二、落地：TDD 节奏 + 覆盖率门禁（复制即用）

**TDD 循环**（对应 §2.1）：红（写失败测试）→ 绿（最小实现）→ 重构。每个功能点的 BDD 场景逐条转成测试用例。

**AAA 模板 + 命名 `test_<方法>_<场景>_<期望>`：**

```python
def test_calculate_discount_vip_user_gets_20_percent_off():
    user = User(role="vip"); order = Order(total=100.0)   # Arrange
    result = calculate_discount(user, order)              # Act
    assert result == 80.0                                 # Assert
```

**`pyproject.toml` + CI 覆盖率门禁：**

```toml
[tool.pytest.ini_options]
addopts = "--cov=myapp --cov-fail-under=80 --cov-branch"
```

```yaml
# CI: 单元测试 < 30s 必过；低于阈值阻止合入
- run: pytest -q
```

## 三、决策表 / 速查

### 测试金字塔（比例≈70/20/10）

| 层 | 占比 | 速度 | 反馈粒度 |
|----|------|------|----------|
| 单元 | ~70% | 毫秒 | 函数级 |
| 集成 | ~20% | 秒 | 模块级 |
| E2E | ~10% | 分钟 | 流程级 |

### 什么该测

| 优先测 | 不必测 |
|--------|--------|
| 核心业务逻辑、复杂算法 | 第三方库内部、框架能力 |
| 边界（空/零/极大值） | 简单 getter/setter |
| 异常路径（超时/宕机/非法输入） | 纯配置、自动生成代码 |
| 安全相关代码 | |

### 覆盖率目标（按档位/阶段）

| 阶段 | 行 | 分支 |
|------|----|----|
| 快速原型 / 轻量 | 不要求 | 不要求 |
| MVP | ≥60% | ≥50% |
| 正式产品 | ≥80% | ≥70% |
| 核心模块（支付/认证/数据安全） | ≥90% | ≥85% |

> 覆盖率是下限不是目标——关注未覆盖分支的业务意义。

### Mock 边界

| 该 Mock | 不该 Mock |
|---------|-----------|
| 外部 API、DB、文件系统、`datetime.now()`、随机数 | 纯函数、值对象、被测函数本身、项目内部模块 |

## 四、反模式（保留测试独有）

### ❌ 依赖执行顺序
`test_update_user` 靠 `test_create_user` 写的全局 `test_user_id` → 单独跑就崩。
✅ 每个测试在 Arrange 阶段自建数据，乱序 / 并发都能跑。

### ❌ Mock 了不该 Mock 的
把 `DiscountCalculator` Mock 成 `return 20.0` → 测的只是 Mock，真实折扣逻辑从未执行。
✅ 只 Mock DB，折扣计算走真实代码，验证「VIP 确实打 8 折」这个行为。

### ❌ 只测 Happy Path
只有 `test_transfer_success`，缺余额不足 / 账户不存在 / 负数金额。
✅ 每条异常路径一个测试，输入域被完整覆盖，重构时漏校验会立即红。

# 测试规范

---

## 一、测试金字塔
```
         /\
        /E2E\       少量，验证核心流程
       /------\
      /  集成  \     中量，验证模块协作
     /----------\
    /   单元测试  \   大量，验证单一功能
   /--------------\
```
理想比例: 单元70% / 集成20% / E2E 10%

---

## 二、测试原则

### 2.1 FIRST 原则
- **F**ast 快速执行 | **I**ndependent 独立运行
- **R**epeatable 可重复 | **S**elf-validating 自动判断
- **T**imely 及时编写

### 2.2 什么需要测
- 核心业务逻辑（必须） | 复杂算法（必须）
- 边界条件（必须） | 异常路径（建议）
- 简单 getter/setter（可选）

### 2.3 什么不需要测
- 第三方库 | 框架基础能力 | 纯配置文件 | 自动生成代码

---

## 三、单元测试

### 3.1 AAA 模式
```
Arrange → Act → Assert
```

### 3.2 命名规范
`test_<方法>_<场景>_<期望>`

---

## 四、Mock 策略

### 需要 Mock: 外部API、数据库（单元测试中）、文件系统、时间依赖、随机函数
### 不需要 Mock: 纯函数、简单数据结构、被测函数本身
### 原则: 只Mock自己的依赖，不过度Mock，验证Mock调用次数和参数

---

## 五、覆盖率目标
| 阶段 | 覆盖率 |
|------|--------|
| 快速原型 | 不作要求 |
| MVP | 60% |
| 正式产品 | 80% |
| 核心模块 | 90% |

---

## 六、CI 集成
- **pre-commit**: 单元测试 < 30s，不通过阻止提交
- **PR**: 全部单元+集成测试 + 覆盖率检查
- **发布前**: 全部测试 + 性能测试 + 安全扫描

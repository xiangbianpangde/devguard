# 代码编写规范

> 定义代码的命名、结构、注释、错误处理和安全标准，确保团队代码风格一致、可读可维护。

---

## 适用范围

- **项目类型**：所有软件项目（前端/后端/全栈/移动端）
- **技术栈**：语言无关，示例以 Python/TypeScript 为主
- **团队规模**：1 人以上均适用

---

## 一、核心原则

### 原则 1：可读性优先于简洁

代码被阅读的次数远多于被编写的次数。优先选择能让读者一眼看懂的写法，即使多写几行。

**为什么**：过于"聪明"的代码在 3 个月后连作者自己都看不懂。可读的代码降低 Review 成本，减少因误解引入的 bug。

### 原则 2：显式优于隐式

行为要能从代码表面直接推断，不依赖隐藏的副作用或魔法。

**为什么**：隐式行为（如全局状态变更、猴子补丁）让调试变成猜谜。显式代码让每个函数/方法的行为自包含，测试和重构都更安全。

### 原则 3：尽早失败

错误在发生处立即暴露，而非传递到远处才崩溃。输入校验放在函数入口，不信任任何外部数据。

**为什么**：延迟暴露的错误堆栈信息丢失了原始上下文，排查成本指数级上升。入口处校验能给出精确的错误信息，也保护了下游代码。

### 原则 4：一致性高于个人偏好

项目内统一风格优先级最高。哪怕你更喜欢另一种写法，在现有项目中遵循既定风格。

**为什么**：不一致的代码风格让读者的大脑不断切换模式，消耗认知资源。统一的代码库像一本书，跨文件阅读时无缝衔接。

## 二、具体规则

### 2.1 命名规范

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 见名知意 | 名字应准确描述用途，禁止 `data`、`info`、`temp` 等模糊命名 | 必须 |
| 长度适中 | 不过短（`a`、`b`）也不过长（超过 50 字符） | 建议 |
| 项目内风格一致 | 选定 camelCase 或 snake_case 后全项目统一 | 必须 |

| 元素 | 风格 | 示例 |
|------|------|------|
| 变量/函数 | camelCase / snake_case | `userName` / `user_name` |
| 类/接口/组件 | PascalCase | `UserService`、`UserCard` |
| 常量 | UPPER_SNAKE | `MAX_RETRY_COUNT` |
| 文件名 | kebab-case / snake_case | `user-service.ts` / `user_service.py` |
| 布尔变量 | is/has/can 前缀 | `isActive`、`hasPermission`、`canEdit` |

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 文件名用描述性全名 | 禁止缩写，如 `userProfile.ts` 而非 `usrPrf.ts` | 必须 |
| 目录入口用 `index` | `index.ts` / `__init__.py` 作为模块入口 | 建议 |

### 2.2 代码结构

#### 函数

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 单一职责 | 一个函数只做一件事 | 必须 |
| 长度 ≤ 50 行 | 复杂逻辑除外（需注释说明） | 建议 |
| 参数 ≤ 4 个 | 超过用对象/结构体传参 | 建议 |
| 优先纯函数 | 无副作用，相同输入 → 相同输出 | 建议 |

#### 类

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 单一职责 | 一个类只负责一个概念 | 必须 |
| 成员变量 ≤ 7 个 | 超过需考虑拆分为多个类 | 建议 |
| 公开方法在前 | 公开方法放在私有方法之前 | 建议 |

#### 文件

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 单文件 ≤ 500 行 | 超过拆分为多个模块 | 建议 |
| 相关功能放一起 | 高内聚，一个文件解决一个问题域 | 必须 |

### 2.3 注释规范

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 复杂算法必须注释 | 用自然语言解释算法思路和选择理由 | 必须 |
| 业务规则必须注释 | 说明为什么这样处理，而非重复代码 | 必须 |
| 临时方案标注标记 | TODO（待做）/ HACK（临时绕过）/ FIXME（已知缺陷）需标注责任人和期限 | 必须 |
| 公共接口建议注释 | 提供参数说明、返回值、异常 | 建议 |
| 不写显而易见的注释 | `i++ // 自增` 这类注释纯属噪音 | 必须 |
| 过时注释及时删除 | 与代码行为不一致的注释比没有注释更危险 | 必须 |

**TODO 标记格式**：

```python
# TODO(张三): 2026-06-01 前替换为异步实现
# FIXME: 并发场景下有 data race，需要加锁
# HACK: 临时绕过第三方库 bug，等 2.0 版本修复后移除
```

### 2.4 错误处理

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 禁止静默失败 | 所有异常必须有处理或向上传播，不能 `except: pass` | 必须 |
| 最外层统一兜底 | 全局异常处理器捕获未处理异常，返回标准化错误响应 | 必须 |
| 错误信息含上下文 | 包含：什么操作、涉及什么数据、失败原因 | 必须 |
| 使用具体异常类型 | 捕获 `UserNotFoundError` 而非 `Exception` | 必须 |
| 资源用上下文管理器 | 文件/连接/锁使用 `with` 语句自动释放 | 必须 |

### 2.5 安全规范

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 密钥/Token/密码禁止硬编码 | 通过环境变量注入，`.env` 加入 `.gitignore` | 必须 |
| 所有外部输入必须校验 | API 参数、文件上传、URL 参数均须校验，使用白名单优先 | 必须 |
| SQL 使用参数化查询 | 禁止字符串拼接，防 SQL 注入 | 必须 |
| 日志禁止输出敏感信息 | 密码、Token、身份证号须脱敏：`phone=138****1234` | 必须 |
| 禁止信任客户端数据 | 服务端独立校验所有数据，前端校验仅用于 UX | 必须 |

### 2.6 日志规范

| 级别 | 用途 | 生产环境 |
|------|------|---------|
| DEBUG | 开发调试信息 | 关闭 |
| INFO | 关键业务流程节点 | 开启 |
| WARN | 可恢复的异常 | 开启 |
| ERROR | 需要关注的错误 | 开启 |

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 禁止 `print()` 替代日志 | 统一使用 logging 模块 | 必须 |
| 禁止循环中大量日志 | 高频循环中的日志会淹没关键信息 | 必须 |
| 日志含关键上下文 | 如 user_id、order_id、操作类型 | 建议 |

### 2.7 代码审查

| 规则 | 说明 | 强制程度 |
|------|------|---------|
| 审查逻辑正确性 | 正常路径 + 边界条件 + 异常路径 | 必须 |
| 审查安全性 | SQL 注入、XSS、敏感信息泄露 | 必须 |
| 审查命名清晰度 | 是否能从名字推断用途 | 必须 |
| 禁止注释掉的代码块 | 直接删除，历史可从 Git 找回 | 必须 |
| 禁止遗留调试代码 | `print`、`console.log`、调试断点 | 必须 |
| 禁止硬编码配置值 | 超时、URL、阈值等应提取为配置 | 必须 |
| 评估是否有过度设计 | 解决当前问题，不为"可能的需求"提前抽象 | 建议 |

## 三、反模式

### ❌ 反模式 1：模糊命名

```python
# 差：看完不知道是什么
def process(d):
    return d[0] + d[1]

# 差：缩写到不可读
def calc_usr_sc(u, s):
    return u * s / 100
```

**问题**：读者需要通读函数体才能推测意图，Review 和调试效率极低。

### ✅ 正确做法：见名知意

```python
# 好：名字即文档
def calculate_total_price(item_prices: list[float]) -> float:
    return sum(item_prices)

def calculate_user_score(user_activity: float, base_weight: float) -> float:
    """计算用户活跃度得分，基重为百分比"""
    return user_activity * base_weight / 100
```

**理由**：函数名描述做什么，参数名描述输入什么。调用方不需要阅读实现就能理解。

---

### ❌ 反模式 2：静默吞异常

```python
# 危险：异常被吞掉，程序继续运行但状态已错
def fetch_user(user_id):
    try:
        return db.query("SELECT * FROM users WHERE id = ?", user_id)
    except Exception:
        pass  # 返回 None，调用方不知道出错了
```

**问题**：数据库连接失败、表不存在、权限不足等情况全部静默。调用方拿到 `None` 后可能引发更隐蔽的 NPE 或逻辑错误，排查时日志里什么线索都没有。

### ✅ 正确做法：明确处理或传播

```python
import logging

def fetch_user(user_id: str) -> User:
    try:
        return db.query("SELECT * FROM users WHERE id = ?", user_id)
    except UserNotFoundError:
        raise  # 明确传播，让上层决定如何处理
    except DatabaseError as e:
        logging.error("查询用户失败 | user_id=%s | error=%s", user_id, e)
        raise ServiceUnavailableError("数据库服务不可用") from e
```

**理由**：每类异常有明确处理或传播路径。日志记录了完整上下文（user_id + 原始错误），排查时一目了然。

---

### ❌ 反模式 3：SQL 字符串拼接

```python
# 危险：SQL 注入
user_id = request.params["id"]
result = db.execute(f"SELECT * FROM users WHERE id = {user_id}")
```

**问题**：攻击者传入 `1 OR 1=1` 即可绕过条件，传入 `1; DROP TABLE users` 可删表。

### ✅ 正确做法：参数化查询

```python
# 安全：参数与 SQL 分离
user_id = request.params["id"]
result = db.execute("SELECT * FROM users WHERE id = ?", [user_id])
```

**理由**：数据库驱动将参数与 SQL 结构分离处理，参数中的任何特殊字符都不会被解析为 SQL 语句。

---

### ❌ 反模式 4：密钥硬编码 + 遗留调试代码

```python
# config.py
API_KEY = "sk-abc123def456"
DEBUG = True

# user_service.py
def create_user(data):
    print("DEBUG:", data)  # 遗留调试代码
    password = "admin123"  # 默认密码硬编码
    return db.insert(data)
```

**问题**：密钥泄露到版本历史，调试日志污染生产输出，默认密码成为安全漏洞。

### ✅ 正确做法：环境变量 + 日志 + 强制修改

```python
# config.py
import os
API_KEY = os.environ["API_KEY"]
DEBUG = os.environ.get("DEBUG", "false").lower() == "true"

# user_service.py
import logging
logger = logging.getLogger(__name__)

def create_user(data):
    logger.info("创建用户 | username=%s", data.get("username"))
    # 初始密码必须由调用方传入，不允许默认值
    return db.insert(data)
```

**理由**：密钥与代码分离，日志用标准框架且不含敏感信息，不允许默认密码消除安全隐患。

## 四、示例

### 场景 1：函数拆分的决策

**需求**：一个函数负责"验证用户输入 → 查询数据库 → 发送通知 → 记录日志"。

| | 反例 | 正例 |
|---|------|------|
| 代码 | 所有逻辑写在一个 80 行的 `register_user()` 函数中 | 拆分为 `validate_input()`、`create_user_record()`、`send_welcome_email()`、`log_registration()` 四个函数 |
| 说明 | 修改通知逻辑时要通读整个函数；测试只能测"全流程"，无法单独测试校验 | 每个函数可独立测试、独立修改；函数名就是文档，调用方看主函数一眼就知道流程 |

### 场景 2：错误处理的层次

**需求**：用户请求创建订单，但库存不足。

| | 反例 | 正例 |
|---|------|------|
| 做法 | 在 controller 层 catch 所有异常，返回 `{"error": "出错了"}` | 领域层抛 `InsufficientStockError(product_id, requested, available)`；应用层捕获后记录日志并转换为用户可读的 `{"code": 40004, "message": "库存不足：商品 XXX 仅剩 3 件"}` |
| 说明 | 前端和用户都不知道具体原因；排查只看日志（可能还没记） | 前端可以根据错误码做精细化提示；排查时有完整上下文 |

## 五、工具推荐

| 工具 | 用途 | 推荐度 |
|------|------|--------|
| ESLint / Pylint / Clippy | 代码风格自动检查，CI 中强制通过 | 推荐 |
| Prettier / Black / rustfmt | 代码格式化，保存时自动执行 | 推荐 |
| SonarQube / CodeQL | 静态分析，发现安全漏洞和代码异味 | 可选 |
| pre-commit hooks | 提交前自动运行 lint + format | 推荐 |
| Commitlint | 校验 commit message 格式 | 可选 |

## 六、检查清单

- [ ] 变量/函数名是否见名知意？（没有 `data`、`temp`、`process` 等模糊命名）
- [ ] 函数是否单一职责且 ≤ 50 行？
- [ ] 是否所有外部输入都已校验？
- [ ] 是否有 SQL 字符串拼接？（必须全部用参数化查询）
- [ ] 密钥/Token/密码是否全部通过环境变量注入？
- [ ] 是否有遗留的 `print()` / `console.log()` 调试代码？
- [ ] 日志中是否含有密码/Token/身份证号等敏感信息？
- [ ] 是否有被注释掉的代码块？（应直接删除）
- [ ] 异常是否被具体捕获和处理，而非 `except Exception: pass`？
- [ ] 命名风格是否与项目已有代码一致？

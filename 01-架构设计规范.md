# 架构设计规范

---

## 一、核心原则

### 1.1 单一职责
每个模块/类/函数只做一件事，做好一件事。

### 1.2 开闭原则
对扩展开放，对修改关闭。新增功能通过扩展实现，而非修改已有代码。

### 1.3 依赖倒置
高层模块不依赖低层模块，两者都依赖抽象。抽象不依赖细节，细节依赖抽象。

### 1.4 接口隔离
不应强迫使用者依赖他们不使用的接口。大接口拆分为多个小接口。

---

## 二、分层架构（通用）

```
表现层 (UI/API)
    |
应用层 (业务编排)
    |
领域层 (核心业务逻辑)
    |
基础设施层 (数据库/缓存/外部服务)
```

### 依赖方向
- **上层依赖下层**：表现层 -> 应用层 -> 领域层
- **下层不依赖上层**：基础设施层只提供实现，不反向依赖
- **领域层是核心**：不依赖任何框架或外部库

### 各层职责

| 层 | 职责 | 不应包含 |
|----|------|---------|
| 表现层 | 接收请求、参数校验、返回响应 | 业务逻辑 |
| 应用层 | 用例编排、事务管理、权限校验 | 领域规则 |
| 领域层 | 实体定义、业务规则、领域服务 | 数据库操作细节 |
| 基础设施层 | 数据库访问、缓存、外部API调用 | 业务判断 |

## 三、模块划分

### 3.1 按业务领域划分（推荐）

```
src/
├── user/          # 用户模块
│   ├── api/       # 接口层
│   ├── service/   # 业务层
│   └── model/     # 数据模型
├── order/         # 订单模块
├── product/       # 商品模块
└── common/        # 公共模块
```

### 3.2 按技术层次划分（传统）

```
src/
├── controllers/   # 控制器
├── services/      # 服务层
├── models/        # 数据模型
├── utils/         # 工具函数
└── config/        # 配置
```

### 3.3 选择建议
- 项目规模大、业务复杂 -> 按业务领域划分
- 项目规模小、功能简单 -> 按技术层次划分

---

## 四、依赖管理

### 4.1 循环依赖
**绝对禁止**模块间循环依赖。如出现，提取公共部分到独立模块。

### 4.2 依赖方向
```
高内聚模块 -> 低内聚模块 -> 通用工具
```

### 4.3 第三方依赖
- 优先标准库
- 其次社区主流库（stars > 1000, 持续维护）
- 最后自研

---

## 五、配置管理

### 5.1 环境隔离
```
开发环境 (development)
测试环境 (testing)
预发布环境 (staging)
生产环境 (production)
```

### 5.2 配置原则
- 敏感信息（密钥、密码）通过环境变量注入
- 非敏感配置可放在配置文件
- 配置文件不入版本库（提供 `.example` 模板）
- 不同环境使用不同配置文件

---

## 六、技术选型

### 6.1 选型原则
1. 成熟稳定 > 新奇酷炫
2. 社区活跃 > 功能强大
3. 团队熟悉 > 性能最优
4. 够用就好 > 过度准备

### 6.2 选型流程
1. 明确需求
2. 列出 2-3 个候选方案
3. 对比优缺点
4. 记录决策理由（ADR）

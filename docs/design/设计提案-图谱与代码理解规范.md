# 设计提案：新增「代码理解与知识图谱」规范维度

> 版本: v0.1 | 状态: 提案 | 来源调研: `docs/research/图谱与代码理解工具调研.md`

---

## 一、背景与动机

### 1.1 为什么需要这个维度

当前开发规范覆盖了"怎么写代码"（架构/编码/API/测试）和"怎么协作"（Git/文档/AI流程），但缺少一个关键维度——**怎么让代码库本身变得可理解**。

随着项目规模增长，新成员 onboarding、跨团队贡献、AI 辅助开发都面临同一个瓶颈：**理解现有代码的成本远高于写新代码的成本**。

### 1.2 调研启示

对 5 个代码理解/知识图谱项目的调研（CodeGraph、CodeGraphContext、Understand-Anything、Karpathy LLM Wiki、LLM Wiki）揭示：

- **代码图技术已成熟**：tree-sitter AST 解析 + 图数据库 + MCP 工具接口，可将代码理解成本降低 35-57%（Token/工具调用）
- **确定性关系优于语义相似度**：代码理解的核心是"谁调用了谁""谁继承了谁"，而非语义相似度。图结构比向量搜索更适合
- **混合精度提取是最佳实践**：tree-sitter 做确定性结构提取 + LLM 做语义摘要，两者互补

## 二、提议的新规范结构

### 2.1 方案 A：新增第 08 号规范「代码理解与图谱规范」

```
conventions/
├── 08-code-understanding_代码理解与图谱规范.md   # 新规范正文
```

覆盖内容：

| 章节 | 内容 | 参考 |
|------|------|------|
| 1. 核心原则 | 可理解性优先、图优先于向量、增量维护、工具辅助 | 调研总结 3.1 |
| 2. 代码图谱构建 | 节点定义（函数/类/模块/文件）、边类型标准化（CALLS/IMPORTS/EXTENDS/IMPLEMENTS）、混合精度提取 | CodeGraph、CodeGraphContext |
| 3. 图存储与查询 | SQLite+FTS5 做最小方案、图数据库做进阶方案、MCP 工具接口封装 | CodeGraph、Karpathy |
| 4. 可视化规范 | 力导向图 + 分层树双视图、按架构层颜色编码、角色自适应 | Understand-Anything |
| 5. 理解辅助设施 | 自动生成导览（tour）、入口点标注、影响分析报告 | Understand-Anything |
| 6. 文档知识图谱 | 从文档提取互链 wiki、增量维护、矛盾/lint 检查 | Karpathy Gist、LLM Wiki |
| 7. 维护与质量 | 图 lint、死代码检测、版本控制、社区检测 | CodeGraphContext、LLM Wiki |
| 8. 反模式 | 依赖黑洞、循环依赖、孤儿模块、巨型函数 | 通用 |
| 9. 检查清单 | 可操作的自查项 | 通用 |

### 2.2 方案 B：分散增强到现有规范

| 现有规范 | 增强内容 |
|---------|---------|
| 01-架构设计规范 | 增加"依赖关系图谱"作为架构文档的必备输出 |
| 02-代码编写规范 | 增加"函数间调用关系可视化"要求 |
| 06-文档规范 | 增加"文档间互链图谱"要求（wiki 模式） |
| 07-AI协作开发流程 | 增加"MCP 代码理解工具"作为开发环境标配 |

### 2.3 方案对比

| 维度 | 方案 A：独立规范 | 方案 B：分散增强 |
|------|----------------|----------------|
| **学习成本** | 一条新规范，一次性学习 | 需要改 4 个规范，散落各处 |
| **执行成本** | 独立检查清单 | 多个检查清单中零散项 |
| **系统性强** | ✅ 完整覆盖图构建→存储→查询→可视化 | ❌ 碎片化 |
| **渐进采用** | ❌ 全有或全无 | ✅ 可逐个规范推进 |
| **维护难度** | ✅ 一处更新 | ❌ 多文件同步 |

**推荐方案 A**（独立规范）+ **方案 B 的部分交叉引用**，在 01-架构和 06-文档中加简短引用指向第 08 号规范。

## 四、实施建议

### 4.1 渐进路径

```
第 1 步：规范初稿（本提案+调研报告 → 正式规范文档）
第 2 步：工具选型（评估 CodeGraph / CodeGraphContext / Understand-Anything 哪个适合团队）
第 3 步：项目试点（选一个中等项目做代码图谱）
第 4 步：总结经验（更新规范 + 添加示例）
```

### 4.2 推荐工具链

| 角色 | 推荐工具 | 理由 |
|------|---------|------|
| CI 代码图谱 | CodeGraph | 零配置、单二进制、Token 节省显著 |
| 深度分析 | CodeGraphContext | 支持图数据库、跨语言、死代码检测 |
| 团队可视化 | Understand-Anything | 多平台支持、交互式图谱、导览功能 |
| 文档知识库 | LLM Wiki | 完整的 Karpathy 模式实现、桌面 GUI |

### 4.3 预期收益

- 新成员 onboarding 时间减少 40-60%（通过导览 + 图谱导航）
- AI 编码助手的代码理解 Token 消耗降低 35-57%（通过图查询替代 grep/read 循环）
- 架构腐化早期发现（通过依赖图谱 diff + lint）
- 跨团队代码复用率提升（通过可探索的代码地图）

## 五、开放问题

1. **工具标准化程度**：是否统一推荐某一款工具，还是给出多选让各团队自行评估？
2. **图存储方案**：小项目用 SQLite，大项目用图数据库——两者的分界线是什么（代码行数？模块数？团队人数？）
3. **与 AI 工作流的集成**：MCP 工具应当作为开发环境标配，还是可选增强？
4. **可视化频率**：每次提交都更新图谱？还是每日构建？还是按需生成？
5. **隐私与安全**：代码图谱是否会暴露敏感的内部架构信息？

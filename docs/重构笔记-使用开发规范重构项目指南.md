## 一、核心原则：先探后改，不猜测

> **最重要的规则**：绝对不要凭"应该是""大概有"做修改。每一项变更必须有审计证据支撑。

```
探索 → 审计报告 → 优先级排序 → 逐项修复 → 验证
  ↑                                                  │
  └──────────────── 修复引入新问题则回头 ─────────────┘
```

### 1.1 两个档次

| 档位 | 触发条件 | 做什么 |
|------|---------|--------|
| **快速对齐** | 项目已有完善代码、CI 能跑、只是缺规范文件 | 只补配置 + 文档，不动代码 |
| **全面重构** | 项目有红线违规（print / 硬编码密钥 / 内部Mock） | 红线修复 + 配置 + 文档 |

本次 research-tool 是"快速对齐 + 少量红线修复"的混合。

## 阶段 1：全量审计

### 1.1 用什么工具做审计

**推荐：用 `explore` 子代理一次性审计全维度。** 写一个自包含的审计 prompt，把 6 个规范的审计要点都放进去。子代理会读代码、做分析、返回结构化报告——只返回结论，不污染你的上下文。

```
审计 prompt 关键要素：
1. 明确路径（目标项目、规范文件夹的绝对路径）
2. 按规范维度分节（01-架构 / 02-代码 / 03-Git / 04-API / 05-测试 / 06-文档）
3. 每节标注："✅ 合规项" 和 "❌ 违规项"
4. 违规项必须包含：文件:行号 + 违反的具体规范条目 + 风险等级
```

### 1.2 审计清单（按规范逐维）

#### 01-架构
- [ ] 检查模块间是否有循环依赖（A→B→A）
- [ ] 领域层是否有框架/ORM import（FastAPI / SQLAlchemy / Django）
- [ ] 表现层是否直接调基础设施层（跳过应用层/领域层）
- [ ] 是否已有 importlinter.ini 或等效的架构校验
- [ ] 敏感配置是否走环境变量（`${VAR}` 语法或 `os.environ`）
- [ ] 是否有 config.example.yaml（不包含真实密钥）
- [ ] 是否有 ADR 记录（worklogs/decisions/）

#### 02-代码
- [ ] **无条件扫描**：`search_content "print("` 排除测试文件——这是审计第一步
- [ ] SQL 是否有字符串拼接（`"SELECT * FROM " + table`）
- [ ] 密钥/Token/密码是否硬编码（正则：`(api_key|secret|password|token)\s*=\s*["']`）
- [ ] 是否有 `except: pass` 或裸 `except Exception` 无日志
- [ ] 是否有注释掉的代码块（排除 TODO/FIXME/HACK）
- [ ] 是否有 `pdb.set_trace()` / `breakpoint()` / `debugger`
- [ ] 日志是否脱敏（密码/身份证号不应出现在日志中）
- [ ] 是否有 ruff.toml 或 pyproject.toml 中的 `[tool.ruff]`
- [ ] 是否有 .pre-commit-config.yaml

#### 03-Git
- [ ] .gitignore 是否包含 `.env` / `config.yaml`（密钥不入库）
- [ ] 是否有 commitlint.config.js
- [ ] 是否有 .gitmessage（提交模板）
- [ ] 是否有 .github/pull_request_template.md
- [ ] 是否有 .pre-commit-config.yaml（含 gitleaks）

#### 04-API（仅 REST API 项目）
- [ ] URL 是否用名词复数 + kebab-case，无动词
- [ ] 响应是否统一 `{code, message, data}` 结构
- [ ] 错误码是否唯一
- [ ] 是否所有接口默认需认证
- [ ] 输入是否做了类型校验
- [ ] 是否有 .spectral.yaml（OpenAPI lint）

> **注意**：CLI 工具 / Gradio UI 项目，04 规范大部分不适用——审计结论中明确标注 "N/A" 即可，不要强行套。

#### 05-测试
- [ ] 测试是否独立（pytest 乱序跑仍全绿）
- [ ] 是否只 Mock 外部边界（DB/HTTP/文件/时间）——**审计重点**
- [ ] 是否覆盖正常 + 边界 + 异常路径
- [ ] 是否有 flaky test
- [ ] pytest 配置中是否有 `--cov` 和 `--cov-fail-under`
- [ ] dev 依赖中是否有 pytest-cov

#### 06-文档
- [ ] 根目录是否有 README，含"快速开始"（≤5 条命令能跑起来）
- [ ] 是否有 CHANGELOG.md
- [ ] 公共 API 是否有 docstring（参数/返回/异常）
- [ ] 是否有 CLAUDE.md（AI 上下文文件）
- [ ] 是否有 STATUS.md（进度追踪）
- [ ] 是否有 worklogs/ 目录
- [ ] 是否有 meta/FILE_GRAPH.md

审计完成后，你应该有一份结构化的违规清单：

```
| 优先级 | 维度 | 条款 | 文件:行 | 风险 | 修复建议 |
|--------|------|------|---------|------|----------|
```

## 阶段 2：红线修复

### 2.1 🔴 优先：print() 改 logging

**这是最常见的违规。** 处理步骤：

```
1. search_content "print(" 排除 tests/ → 定位所有 print
2. 确认 print 的用途：
   - CLI 输出 → 保留（Typer 的 echo 不算 print）
   - 调试输出 → 改 logging.debug()
   - 状态提示 → 改 logging.info()
   - 错误信息 → 改 logging.error()
3. 在文件头部添加 `import logging` + `logger = logging.getLogger(__name__)`
4. 替换所有 print(f"...") → logger.info("...", *args)  # 注意 %s 格式化，不是 f-string
```

**我犯的错**：第一次替换时用 `print(f"端口 {port} 不可用")` → `logger.warning(f"端口 {port} 不可用")`。这是错的！logging 应该用 `%s` 占位符而非 f-string，因为 logging 的惰性求值依赖 `%` 格式化。

正确写法：
```python
# ❌ 错误
logger.warning(f"端口 {server_port} 不可用")
# ✅ 正确
logger.warning("端口 %s 不可用", server_port)
```

### 2.2 🔴 优先：内部 Mock 改构造器注入

**这是最容易被忽略的违规。** 05规范明确"只 Mock 外部边界，禁止 Mock 项目内部模块"。

检查方法：
```
search_content "monkeypatch.setattr.*self\.\w+"   # Python
search_content "monkeypatch.setattr.*_\w+"          # 匹配 _get_llm 等私有方法
```

修复模式：
```python
# ❌ 违规：Mock 类内部方法
monkeypatch.setattr(pipe, "_get_llm", lambda: MockLLMClient())
pipe = ResearchPipeline(cfg)

# ✅ 合规：构造器注入（需同步改类定义）
pipe = ResearchPipeline(cfg, llm=MockLLMClient())

# 类定义变更：
#   def __init__(self, config):            →  def __init__(self, config, *, llm=None):
#   self._llm: LLMClient | None = None     →  self._llm: LLMClient | None = llm
```

**我犯的错**：只修了审计发现的两处（test_pipeline_mock.py），却漏了 test_deepen.py 里的另外两处。因为 audit 结果只说 "test_pipeline_mock.py:53"，没说 deepen.py 也有。所以修复后要 `search_content` 全局验证模式是否清零。

### 2.3 处理 except Exception 的取舍

22 个文件用了 `except Exception as e:  # noqa: BLE001`，全在 search 后端和 LLM 调用层——属于边界层防御性惯例。三方库的异常类型不稳定，无法穷举。

**不要一刀切禁掉**。做法：
1. 在 ruff.toml 中用 `per-file-ignores` 标记已知的豁免文件
2. 保留现有的 `# noqa: BLE001` 注释作为意图标记
3. 记录为技术债（Future: 当三方库稳定后收紧）

## 阶段 3：基础设施落盘

### 3.1 文件创建顺序

```
1. ruff.toml               ← 先建，因为要用来验证阶段2的修复
2. pyproject.toml 补充     ← [tool.coverage] + pytest-cov 依赖
3. .pre-commit-config.yaml ← 提交前拦截
4. commitlint.config.js    ← 提交格式校验
5. .gitmessage             ← 提交模板
6. .github/pull_request_template.md
7. importlinter.ini        ← 架构分层契约
```

### 3.2 ruff.toml 模板

```toml
[tool.ruff]
line-length = 100

[tool.ruff.lint]
select = ["E", "F", "N", "T20", "T100", "S", "PLR0915"]

[tool.ruff.lint.pylint]
max-statements = 80

[tool.ruff.lint.per-file-ignores]
# 根据你的项目实际情况调整豁免范围
"src/search/*.py" = ["S110"]
"tests/*.py" = ["S110", "T20"]
```

**关键决策**：`S110`（except-pass）按文件级忽略，不要全局关——全局关会让新代码悄悄引入裸 except。

### 3.3 importlinter.ini 怎么写

**这是最需要理解项目模块结构的环节。** 不要抄通用模板——要先画模块依赖图。

```
操作步骤：
1. 用 directory_tree 看 research_tool/ 下的子模块
2. 读关键文件的 import 头部（pipeline.py, cli.py, webui.py, models.py）
3. 判断四层归属：
   - 表现层：cli.py, webui.py
   - 应用层：pipeline.py, config.py
   - 领域层：models.py, errors.py, slug.py
   - 基础设施层：stages/, search/, llm/, ingest/
4. 写 layers 契约
5. 写 domain-pure 契约（领域层禁依赖框架）
6. 写 no-cross-layer 契约（表现层禁直调基础设施层）
```

**我犯的错**：一开始照抄了规范里的 layers 示例（`myapp.presentation` / `myapp.domain` 等），但 research-tool 是单包结构（`research_tool.cli` / `research_tool.models`），需要改成项目实际的模块名。**必须按实际代码结构写，不能照抄示例。**

### 3.4 pyproject.toml 补充

```toml
# dev 依赖加 pytest-cov
dev = ["pytest>=8.0", "pytest-asyncio>=0.23", "pytest-cov>=5.0"]

# pytest 加覆盖率参数
[tool.pytest.ini_options]
addopts = "--cov=research_tool --cov-report=term-missing"

# 覆盖率阈值（新建项目从 60 起步，老项目按实际设）
[tool.coverage.run]
source = ["research_tool"]

[tool.coverage.report]
fail_under = 60
```

**我犯的错**：加了 `--cov` addopts 到 pyproject.toml 后，直接跑 `pytest` 就报错 "unrecognized arguments: --cov=research_tool"——因为 .venv 里没装 pytest-cov。**加配置之前先确认依赖是否已安装。** 解决方案：要么先 `pip install pytest-cov`，要么用 `-o "addopts="` 临时跳过。

## 阶段 4：文档体系搭建

### 4.1 创建顺序

```
1. CLAUDE.md         ← 最先，因为会后每步引用它
2. STATUS.md         ← 基于审计结果写当前状态
3. CHANGELOG.md      ← 记录本次重构变更
4. docs/plan/背景.md  ← 项目缘起（从 README 提取）
5. docs/plan/开发清单.md ← 已有功能+待规划
6. meta/FILE_GRAPH.md ← 最后，因为要汇总所有新增文件
7. worklogs/YYYY-MM-DD_重构.md ← 记录本次重构全过程
```

### 4.2 CLAUDE.md 写作要点

**CLAUDE.md 是给未来 AI 会话读的索引，不是人类文档。** 参考开发规范自身的 CLAUDE.md 格式：

必须包含：
- **项目概述**（一句话 + 技术栈）
- **目录索引**（只到顶层，关键文件加一行注释）
- **模块依赖图谱**（ASCII 单向箭头，显示调用方向）
- **核心规则**（2-5 条，可执行）
- **当前状态**（版本 + 测试数 + 合规状态）
- **常见任务速查表**（"用户说 X → AI 做 Y"）

不要写：
- 长篇的设计理念——放 docs/plan/背景.md
- 逐文件的详细说明——这是 README 的职责
- 不精确的触发条件——如"累计 N 个功能点"

### 4.3 FILE_GRAPH.md 写作要点

**FILE_GRAPH 是"新文件放哪"的决策树。** 我写的版本包含：
1. 全貌目录树（带注释说明每项是什么）
2. 决策树（新文件类型 → 落点目录）

关键：
- **最后写**——因为要统计所有新增文件都已纳入
- 决策树用条件分支格式（`问题？├─ 答案A → 落点A │ └─ 答案B → 落点B`）

### 4.4 worklog 写作要点

worklog 模板（从开发规范复制）：
```markdown
# 标题 — 工作日志
> 日期 / 类型 / 关联规范

## 做了什么
（列表，每项一行）

## 验证结果
（测试 / lint / 截图 等可观测证据）

## 关键决策
（本次做的取舍 + 理由。这是未来 ADR 的素材）

## 交接
（给下一个接手的人/下一个自己的注意事项）
```

## 我犯过的错误

> 这些是我在本次 research-tool 重构中实际犯下的错误，按影响程度排序。

### ❌ 错误 1：试图用 `cd` 切换工作目录

```
# ❌ 不工作
"cd C:/Users/... && .venv/Scripts/python.exe -m pytest tests"

# ✅ 正确
"C:/Users/yhn/.../.venv/Scripts/python.exe" -m pytest "C:/Users/.../tests"
```

**原因**：run_command 的 cwd 固定在项目根目录，`cd` 在链式命令中不改变后续命令的工作目录。**永远用绝对路径。**

### ❌ 错误 2：加 `--cov` 配置前没检查 pytest-cov 是否安装

在 pyproject.toml 里加了 `addopts = "--cov=research_tool"`，然后跑 pytest 就全炸——因为 .venv 里没有 pytest-cov。

**正确顺序**：
1. 先检查 .venv 里有没有 `pytest-cov`：`pip list | grep pytest-cov`
2. 没装就 `pip install pytest-cov`
3. 最后再加 `--cov` addopts

### ❌ 错误 3：审计报告说"A 文件有违规"就只修 A 文件

审计说 `test_pipeline_mock.py:53` 有内部 Mock 违规，我只修了那一处。但 `test_deepen.py:110` 和 `test_deepen.py:135` 有**同模式违规**——审计报告没提到，我也没主动搜。

**教训**：修完一个模式后，用 `search_content` 全局扫描确认是否清零。不要相信审计报告的覆盖率是 100%。

### ❌ 错误 4：修复后忘了清除未使用的参数

把 `monkeypatch.setattr(pipe, "_get_llm", ...)` 删掉后，函数签名里还留着 `monkeypatch` 参数——pytest 会报 "unexpected argument" 错。

**教训**：修改函数调用方式后，立即检查函数签名是否需要同步清理。

### ❌ 错误 5：importlinter.ini 照抄规范示例

规范里的示例写的是 `myapp.presentation` / `myapp.domain`，我直接复制到 research-tool 的 importlinter.ini 里——但 research-tool 用的是 `research_tool.cli` / `research_tool.models`。

**教训**：importlinter.ini 的模块名必须来自实际代码的 `directory_tree` 输出，不能照抄示例。

### ❌ 错误 6：logging 用了 f-string 而非 %s 占位

```python
# ❌ 我最初写的（f-string 在 logging 中是反模式）
logger.warning(f"端口 {server_port} 不可用，改用 {port}")

# ✅ 正确（logging 的惰性求值依赖 % 格式化）
logger.warning("端口 %s 不可用，改用 %s", server_port, port)
```

**原因**：logging 模块在消息不需要输出时（如 DEBUG 级别在生产环境被关闭），会跳过字符串格式化以节省性能。f-string 会立即求值，破坏了这个优化。

### ❌ 错误 7：前端/可视化改动只做 syntax check + HTTP 200 就声称"验证通过"

**这是最严重的不诚实。** 在 AgentHub 给 dashboard 做整站可视化（架构图/依赖图/概览）后，我只跑了：JS `node --check` 语法检查 + 另写脚本复刻数据变换 + `curl` 拿到 HTTP 200，就报告"验证通过"。**但我根本没在浏览器里渲染过它**——没确认 Tab 能切、SVG 能画出来、点节点会不会高亮。用户一句"你测试了没有？"就戳穿了。

**教训**：对前端/可视化/任何"渲染才知道对不对"的改动，"可观测验证"的标准是**真实浏览器渲染**，不是语法+接口可达。Mock 数据变换 ≠ 跑真身代码。说"验证通过"前先问自己：我亲眼看到它工作了吗？

**没有 jsdom/puppeteer 也能测**——系统装了 Chrome 就行，用一次性截图（render-and-exit，不开守护进程、不占调试端口）：

```bash
# 1) 起本地静态服务（相对路径 fetch 需要 HTTP）
python scripts/start_server.py   # 或 python -m http.server 8080

# 2) 为需要交互的视图生成临时副本：加载后自动切到该 Tab / 选中节点
#    （把 load(); 替换成 load(); setTimeout(()=>switchView('graph'),500); 等）

# 3) headless Chrome 渲染并退出，截图
chrome --headless=new --disable-gpu --hide-scrollbars \
  --user-data-dir=<临时profile> --window-size=1440,1100 \
  --virtual-time-budget=6000 --screenshot=out.png \
  "http://localhost:8080/__t_view.html"

# 4) 用能看图的工具打开 out.png 亲眼确认；测完删掉临时 html
```

**Windows 上的坑**：想用 CDP 远程调试端口（`--remote-debugging-port`）驱动时，如果用户已开着 Chrome/Edge，新进程会把 URL 交给已有实例、**端口根本不监听**。一次性 `--screenshot` 模式不受此影响（它用独立临时 profile、渲染完就退），更可靠。

### ❌ 错误 8：远程 main 分叉时差点 force-push，会抹掉队友已合并的 PR

本地领先、远程也领先（diverged）时，我第一反应想 `git push --force` 把本地推上去。**但远程那 3/14 个提交是队友刚合并的 PR（Pi Agent、长驻 CLI 等），force-push 会直接从 main 抹掉别人的工作。**

**教训**：
- 共享分支（main）**永不 force-push**。要推就先 `git fetch` → `git rev-list --left-right --count main...origin/main` 看分叉。
- push 被拒（non-fast-forward）说明远程有你没有的提交——**先看会丢什么**：`git log --oneline main..origin/main`，再 `git merge` 保留双方，绝不 `--force` 覆盖。
- 实在要改写自己刚推的提交，用 `--force-with-lease`（有人在你之后推过会自动中止），且只在确认没动到别人提交时。

### ❌ 错误 9：未经明确要求就 push 到远程

用户让我"整理"/"改文件"，我做完顺手 `git push` 了——push 是**对外、难撤销**的操作，不该自作主张。

**教训**：默认只到**本地 commit**，把推送时机交给人。只有用户明确说"推/push/提交到远程"才推。即使任务看着是上一个推送任务的延续，也重新确认。对外/不可逆动作（push / 删除 / 发布）宁可多问一句。

### ❌ 错误 10：移动"生成文件"时没同步生成它的脚本

把 `CODE_MAP.md` 从根目录移到 `docs/`（`git mv`）后，我差点漏改 `gen_codegraph.py` 里的写出路径——**下次一跑生成器，它又会把文件写回老位置**，移动等于白做。

**教训**：移动/重命名**生成产物**时，真正的源头是生成它的脚本。清单：① 生成器的写出路径 ② 生成器的 docstring/日志 ③ 所有引用（CLAUDE.md/README/dashboard 等）④ `.gitignore` 注释 ⑤ `meta/FILE_GRAPH.md`。移完跑一次生成器验证落点正确。

## 二、各项目详细分析

### 1. CodeGraph — colbymchenry/codegraph

**链接**: https://github.com/colbymchenry/codegraph

**核心思路**: 将代码库预索引为**可查询的知识图谱**，让 AI 编码智能体用一次图查询代替昂贵的 grep/glob/Read 循环。

**4 阶段流水线**:
1. tree-sitter 解析 AST
2. 提取节点（函数、类、方法）与边（调用、导入、继承、实现）
3. 存入本地 SQLite（FTS5 全文搜索）
4. 跨文件引用解析（含框架感知路由：Django/Flask/FastAPI/Express/NestJS）

**关键能力**:
- 影响分析（追溯调用者/被调用者/传递影响）
- 调用路径追踪（逐跳映射 + 动态分派启发式）
- 全文本搜索 & OS 文件监视器增量更新

**基准测试** (7 个真实代码库): 工具调用减少 71%、Token 减少 57%、成本降低 35%。

### 2. CodeGraphContext — CodeGraphContext/CodeGraphContext

**链接**: https://github.com/CodeGraphContext/CodeGraphContext

**核心思路**: MCP 服务器将本地代码库索引到真正的**图数据库**，为 AI 助手提供结构化、可查询的上下文。

**关键能力**:
- 多后端支持（KuzuDB / FalkorDB / Neo4j）
- 调用链分析（直接 + 传递）、类层次结构分析
- 死代码检测（未使用函数）、圈复杂度分析
- 影响分析（"修改 X 会破坏什么？"）
- **跨语言分析**（同一图中 Python→Rust FFI）
- 交互式力导向/层次化可视化

**核心哲学**: 没有嵌入相似度，只有确定性关系结构。

### 3. Understand-Anything — Lum1104/Understand-Anything

**链接**: https://github.com/Lum1104/Understand-Anything

**核心思路**: 多平台 AI 编码助手插件，将代码库转换为交互式、可探索的知识图谱。运行在 15+ 平台。

**6 智能体协调流水线**:
- `project-scanner` → 扫描项目结构
- `file-analyzer` → 分析单个文件
- `architecture-analyzer` → 推断架构层次
- `tour-builder` → 生成导览
- `graph-reviewer` → 审查图质量
- `domain-analyzer` → 领域映射

**关键能力**:
- tree-sitter（确定性结构事实）+ LLM（自然语言摘要）混合管道
- 交互式力导向图谱（React Flow），按架构层颜色编码
- 模糊 + 语义搜索、引导式导览
- 差异影响分析、角色自适应 UI
- 增量更新，支持 26+ 文件类型

### 4. Karpathy's LLM Wiki Gist

**链接**: https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f

**核心思路**: 设计模式而非代码——LLM 智能体从原始源文档增量构建并维护一个由相互链接的 markdown 文件组成的持久 wiki。核心批评 RAG："LLM 在每个问题上都从零开始重新发现知识。没有积累。"

**三层架构**:
1. **Raw 源** — 不可变的原始文档
2. **Wiki** — LLM 生成的 markdown 页面（实体页、概念页、对比页、综合页、查询页）
3. **Schema** — `CLAUDE.md` 或 `AGENTS.md`，定义结构与工作流

**操作原语**: `Ingest` → `Query` → `Lint`（矛盾/过时/孤立检查）→ `Index + Log`

**本质**: Markdown 格式的知识图谱——实体页=节点，wiki 链接=边，索引=图索引。

---

### 5. LLM Wiki — nashsu/llm_wiki

**链接**: https://github.com/nashsu/llm_wiki

**核心思路**: Karpathy 模式的最完整实现——桌面应用（Tauri v2 + React 19 + Rust），自动将文档转换为组织有序、互相关联的 wiki。

**关键能力**:
- 两步 CoT 摄取流水线（分析→生成）
- SHA256 增量缓存（跳过未更改文件）
- 4 信号相关性模型：直接 wikilinks ×3、源重叠 ×4、Adamic-Adar ×1.5、类型亲和性 ×1
- sigma.js + ForceAtlas2 知识图谱渲染
- Louvain 社区检测自动发现知识聚类
- 可选项量搜索（LanceDB）+ Deep Research 功能

## 三、对开发规范的启示

### 3.1 分层架构标准

| 原则 | 说明 | 来源 |
|------|------|------|
| **源层→图层→查询层** | 不可变原始文档→结构化图表示→查询接口，各层职责分离 | Karpathy Gist、CodeGraph |
| **增量更新优先** | 只处理变化部分，避免全量重建 | CodeGraph 文件监视、LLM Wiki SHA256 |

### 3.2 图构建规范

- **混合精度提取**: tree-sitter 做确定性结构提取（调用、继承、导入）+ LLM 做语义增强（摘要、架构层级分配）。tree-sitter 保证正确性，LLM 保证可理解性。
- **边类型标准化**: 统一边类型命名（CALLS / IMPORTS / EXTENDS / IMPLEMENTS / CONTAINS / DEFINES / ROUTES）
- **跨语言统一图**: 多语言代码库应在同一个图中关联（如 Python→Rust FFI）

### 3.3 存储与查询规范

| 层级 | 适用场景 | 方案 |
|------|---------|------|
| 单开发者/小团队 | SQLite + FTS5（CodeGraph 模式） |
| 大团队/跨项目 | 图数据库（KuzuDB / Neo4j） |
| 查询接口 | MCP 工具封装（每个工具只做一件事） |

### 3.4 可视化规范

- 两种视图：力导向图（浏览探索）+ 分层树（架构理解）
- 按架构层颜色编码：API / Service / Data / UI / Utility
- 角色自适应：初级开发者看导览，架构师看依赖，PM 看领域映射

### 3.5 维护与质量规范

- **"Lint" 操作**: 定期检查图中是否存在矛盾、孤立节点、过时声明
- **社区检测**: 自动发现知识聚类（Louvain 算法）
- **版本控制**: 图结构应随代码变更进行版本管理

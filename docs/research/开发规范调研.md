# 开发规范最佳实践调研

> 调研日期: 2026-05-26
> 更新: 2026-07-11

## 信息来源

- Google 工程实践文档 (google.github.io/eng-practices)
- ThoughtWorks 技术雷达
- 《代码整洁之道》(Clean Code) — Robert C. Martin
- 《Google 软件工程》(Software Engineering at Google)
- GitHub 开源项目规范（Kubernetes、Rust、React 等）

## 关键结论

### 1. 规范必须「够用就好」

过度规范导致执行成本过高，团队会选择忽略。每条规则必须回答「违反这条会发生什么」。

**启示**：每条规范配一个「为什么」段落，不做纯教条。

### 2. 分层是最稳定的架构原则

无论前端后端、微服务还是单体，分层（展示层/业务层/数据层）是跨越技术栈的不变量。

**启示**：架构规范以「分层」为骨架，技术选型为肌肉。

### 3. Git 规范是协作的「交通规则」

分支策略（Git Flow / GitHub Flow / Trunk-Based）的选择取决于团队规模：
- 1-3 人：Trunk-Based + tag
- 4-10 人：GitHub Flow + feature branch
- 10+ 人：Git Flow

**启示**：Git 规范应给出多种方案，而非一刀切。

### 4. API 规范的黄金标准是 RESTful + OpenAPI

无论 GraphQL 多流行，RESTful 仍是工业界最大公约数。OpenAPI 3.x 是文档生成的标配。

**启示**：API 规范以 RESTful 为主线，GraphQL/gRPC 作为附录。

### 5. 测试金字塔依然有效

单元测试（70%）+ 集成测试（20%）+ 端到端测试（10%）的配比经过大量项目验证。

**启示**：测试规范给出金字塔配比 + 各层工具推荐。

### 6. 文档的价值 = (被阅读次数) × (信息密度)

没人读的文档是负资产。README 必须 30 秒内让人理解项目能做什么。

**启示**：文档规范聚焦「如何写能让人读」。

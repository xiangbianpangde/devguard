01 · 背景与痛点 —— 只读审查发现的 4 个缺陷

治理工具自身，也曾落在治理链路之外

devguard V2.2.0（governance-as-code 模板仓，163 L4 tests 基线）接受了一轮全面只读审查。 测试与 lint 全绿的表象下，审查在「门禁脚本行为、版本钉版、模板镜像、渲染器健壮性」四个角落发现系统性缺陷——它们的共同点是：都逃过了现有 CI 与钩子的检测。

🔴 ① markdownlint 门禁在 POSIX 空转 lint_markdown.py:47

subprocess.run(cmd_list, shell=True) 在 POSIX 的语义是「只执行 args[0]」——实际跑的是裸 npx，markdownlint 从未被调用，却稳定返回 0。

后果：pre-commit 的 markdownlint 钩子与 CI「markdownlint V3.4」步在 macOS / Linux（含 Ubuntu runner）上从未真正 lint 过任何文件。审查实测：塞入坏 md（MD018+MD009），旧脚本 exit=0 零输出。

pre-commit 钩子失效 CI lint 步失效 仅 Windows 幸免

🔴 ② ruff 版本四方漂移 4 处钉版 3 个版本

同一仓库同时钉着 0.6.0（pyproject dev extras、CI ×4、pre-commit rev）、0.15.20（requirements-dev.txt、本地 .venv）、0.11.7（scaffold 模板 pre-commit）。

后果：本地 lint 结果与 CI 不可互证——0.15.20 的新规则与格式化器行为都与 0.6.0 不同；check_consistency.py 还把 0.6.0 字符串硬编码为「期望值」，版本治理自我锁死。

pyproject 0.6.0 req-dev 0.15.20 scaffold 0.11.7 CI 0.6.0 ×4

🔴 ③ 模板脚本漂移落在双盲区 3 副本失同步

check_template_drift.py 只查 STATUS/CLAUDE/开发清单/dashboard 的章节关键词；check_consistency.py 只比对 ci.yml 文本——scripts/ ↔ docs/templates/devguard/scripts/ 的镜像漂移恰好落在两个机制的交叉盲区。

实测 drift：render_meta.py 模板副本缺 pass_filenames 支持与 LF 规范化（约 20 处差异）；collect_l4_stats.py 模板副本还是 V6.3 旧版，带着已被根版修复的 L4_STATS=0/0 假通过 bug。

render_meta 缺特性 collect_l4_stats 带旧 bug lint_markdown 待同步

🟡 ④ render_meta 截断缺陷 render_meta.py:230

_strip_existing_grade_section 在「## 分级标签」后找不到 \n---\n 分隔符时返回 content[:idx]——把标记之后的全部正文截断，而 --render 是原地写回：一次重渲即毁文（审查用临时副本实测复现）。

伴生缺陷：子串匹配使 ### 分级标签说明 被误判为小节起点，同样会造成文件中间截断。

数据丢失路径 ### 子串误判 原地写回放大后果
!
根因：治理工具自身缺少「行为级」治理——门禁脚本只有存在性检查（文件在不在），没有行为级 L4 测试（跑起来拦不拦）； 模板镜像只有入口文件章节级对比，没有脚本逐字节比对；版本钉版没有单一真源，四处各写各的。 本轮修复的共同手法：把「存在性治理」升级为「行为级治理」。

逐项拆解（点击展开）

① lint_markdown.py —— 从「假通过」到「真门禁」 🔴
拦截什么：Markdown 格式违规（06-documentation L1）。
怎么修：_run_batches(md_files, *, platform, runner, which) 把平台判定、子进程执行、PATH 查找全部参数化——测试可注入替身（只 mock 外部边界，符合 05-testing 红线）；win32 保留 shell=True（.cmd shim 必需），POSIX 列表直跑。
失败闭合：which("npx") 为 None → 打印原因返回 1，不尝试执行；runner 抛 FileNotFoundError → 返回 1。
实测：仓内置坏 md → exit=1 且报 MD018/MD009；干净仓 261 个 md → exit=0。
② toolchain.ruff 真源 —— 从「四处各写」到「一处真源双闸门」 🔴
为什么选 _meta.yaml 而非解析 requirements-dev.txt：项目哲学即「_meta.yaml 是一切渲染/校验的 single source of truth」；pyyaml 已是硬依赖，而 tomllib 需 3.11+（项目声明 ≥3.10）。
双闸门：check_consistency.py（CI compliance 阶段跑）把 CI 中全部 ruff==X 钉版集合与 pre-commit rev 比对真源；test_ci_contract.py 三个版本断言全部改读真源，另加「真源 ↔ requirements-dev ↔ pyproject」一致性测试。
失败闭合：toolchain 节缺失 → formatter 事实直接判不一致（test_consistency.py::test_missing_toolchain_source_fails_closed）。
③ check_script_mirrors() —— 补上模板治理的最后一块拼图 🔴
为什么逐字节而非章节级：脚本无「章节」概念；且既有的 check_ai_workflow.py / check_code_understanding.py 两副本本就逐字节相同——字节级正是项目已有约定。
意外收获：新检查启用当天即检出 collect_l4_stats.py 模板副本是 V6.3 旧版，带着「失败时输出 L4_STATS=0/0 假通过」的已修复 bug——若被 scaffold 分发出去，等于把假通过扩散到每个新项目。
实测：/tmp 副本人为追加一行 → 检出并点名文件。
④ render_meta 防截断 —— 治理工具先学会「不毁文」 🟡
三条边界：缺 --- → GradeSectionError（render 主循环捕获后 FAIL、exit=1，不写该文件）；### 分级标签说明 → 行级正则不命中，原文保留；正常小节 → 剥离后正文完整（三条均有实测与 L4 测试）。
原子性升级：render_convention_grade 先在内存算完全部新内容（任一篇畸形整体中止），再统一落盘——消除「第 N 篇畸形留下前 N-1 篇半成品」的部分写入窗口。

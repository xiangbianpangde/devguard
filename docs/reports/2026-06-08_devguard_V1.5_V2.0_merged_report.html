01 · V1 第一版的崩塌 —— 纯文字规范的 5 个具体痛点

V1 旧范式：规范写在 .md 里，机器看不见

V1 第一版（仅含 8 份散落规范：01-architecture / 02-coding / 03-git / 04-api / 05-testing / 06-documentation / 07-ai-workflow / 08-code-understanding）， 全文 ~3 万字纯 Markdown。表面齐全，实际执行时浮现 5 个系统性痛点：

① 红线靠人记

规范正文写"密钥走环境变量"、"SQL 必须参数化"、"禁循环依赖"、"禁 print 调试"——但 git commit 不拦截、CI 不检测、AI 写代码时也无强制流程。全靠开发者背下来。

02 红线 ×4 01 红线 ×2 04 红线 ×3 零自动检测

② 规范 ↔ 代码脱节

规范讲"怎么做"，代码库没有可运行的落地配置。新人学完规范，照样写不出"合规代码"——因为没有 ruff.toml / commitlint.config.js / L1 检测脚本等可复制即用的工具链。

规范 8 份 落地配置 0

③ AI 不守规范

AI 收到"按规范开发"指令后，会"看到"规范文件，但读 ≠ 守。V1 没有 task routing 机制让 AI 知道"实现任务 → 必读 02-coding / 提交任务 → 必读 03-git"，于是出现 AI 写 commit message 不带 type 前缀、PR description 不引 worklog 等"反复返工"现象。

零 task routing 零 L4 测试

④ 收束靠自觉

原 06-收束节点定义执行四阶段（整理→测试→审计→验证），但触发完全靠人记得，没有"硬闸门"。连续开发 5+ 功能点不收束，僵尸代码、过期文档、错位引用逐步积累。

触发：人记得 拦截：无

⑤ 无可观测性

规范"是否被执行"完全无数据。L1 检测通过率、commit message 合格率、L4 套件覆盖率、收束触发频率——这些度量全部缺失，无法判断规范健康度。

无 L1 仪表盘 无 L4 数据

根本原因

V1 把规范当成"人读的文件"，但工程现实是：人会忘、不会查、不同人会理解错。规范必须能被机器读、能被机器挡、能被机器测，才能从"建议"变成"约束"。

人读 → 机器读 建议 → 约束
!
V1 旧范式的失败本质：把约束逻辑放在"开发者头脑"里。开发者可以忘、AI 可以没读到、审查者可以看走眼——所有执行环节都依赖个体自觉。 V1.x 的核心改造是：把约束从"人脑"搬到"机器"——pre-commit 钩子挡在 commit 前、CI 挡在合并前、收束闸门挡在下一批功能点前。
02 · V1.x 强制约束范式

三层 + 闭环：约束从"人脑"搬到"机器"

V1.x 不是简单"写更多规范"，而是把同样的 17 份规范织成三层互相校验的强制约束网。

V1 旧范式：纯文字 + 自觉

  • ✗ 规范 8 份散落 .md，无 L4 套件
  • ✗ 零 pre-commit 钩子
  • ✗ 零 CI 流水线
  • ✗ 收束靠人记得，无闸门

  • ✗ 规范健康度无数据

  • ✓ 5 阶段 CI 挡 merge
  • ✓ 13 收束报告 + AI 闸门拦截

  • ✓ render_meta.py --check 漂移即 fail

第 8 号规范 v3.0：双图谱范式

CodeGraph（AI 用）

为 AI 提供代码的程序化结构——节点（函数/类/模块）+ 边（调用/依赖/继承）。AI 读 CodeGraph 后能精准知道"我应该改哪里、改完后影响谁"。

JSON 格式 机器可消费 覆盖全量符号

Understand-Anything（人用）

为人类开发者提供代码的语义化解释——自然语言层级的"这个模块做什么、为什么这样组织、和谁交互"。新人 onboarding 不再靠"口口相传"。

自然语言 人在回路 新人友好

① trailing-whitespace pre-commit-hooks
拦截：行尾空格。
怎么拦：自动 trim 后报错。
对应规范：02-coding §格式化（一行 119 字符、行尾零空格）。
② end-of-file-fixer pre-commit-hooks
拦截：文件末尾无换行 / 多余空行。
对应规范：02-coding §格式化（POSIX 文件结尾）。
③ check-yaml / check-json pre-commit-hooks

对应规范：_meta.yaml 是 YAML 真值源——任何破坏性 commit 立即被挡。
④ check-added-large-files pre-commit-hooks

⑤ commitlint (Conventional Commits) commitlint-pre-commit-hook
拦截：不符合 Conventional Commits 的 commit message（feat: / fix: / chore: / docs: 等）。
对应规范：03-git §2.1 Conventional Commits。

⑥ ruff (lint) ruff-pre-commit

对应规范：02-coding 全章。

⑧ gitleaks (密钥扫描) gitleaks-pre-commit

⑨ markdownlint local: scripts/lint_markdown.py
拦截：Markdown 格式违规（标题层级、列表缩进、代码块语言、空行）。

⑩ commit-msg-worklog-ref local: scripts/check_worklog_ref.py

✓

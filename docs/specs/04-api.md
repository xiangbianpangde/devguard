# BDD 规格: API 设计规范

> 对应 `04-api_API设计规范.md`（v2.0：第二步 §2.1 实现（接口）的细化）
> 更新: 2026-05-30

---

## 功能：API 设计规范文档

### 场景 0：声明细化环节 + 红线配自动检测 + 落地配置
- 前置条件：无
- 操作步骤：
  1. 打开 `04-api_API设计规范.md`，读引言与 §一/§二
- 预期结果：
  - 声明细化「第二步 §2.1 实现」，§2.2 可观测验证中「API 接口」证据对照本规范
  - §一红线（名词复数+kebab-case 无动词、统一响应、状态码与 code 一致、错误码唯一、默认认证、输入校验、向后兼容）标注 spectral / 中间件检测
  - §二提供可复制物料：统一响应拦截器骨架 + `.spectral.yaml` OpenAPI lint 规则

### 场景 1：RESTful URL 设计有据可查
- 前置条件：需要设计新 API 端点
- 操作步骤：
  1. 阅读"URL 设计"章节
  2. 按规范设计 URL 路径
- 预期结果：
  - 明确资源命名规则（名词复数、kebab-case）
  - 有正确 URL 和错误 URL 的对照示例
  - 明确 HTTP 方法的使用场景（GET/POST/PUT/DELETE/PATCH）

### 场景 2：统一错误码体系
- 前置条件：需要定义 API 错误响应格式
- 操作步骤：
  1. 阅读"错误码规范"章节
- 预期结果：
  - 有统一的错误响应 JSON 结构
  - 错误码分段定义（4xxx 客户端 / 5xxx 服务端）
  - 有常见错误码对照表

### 场景 3：版本管理策略清晰
- 前置条件：API 需要升级
- 操作步骤：
  1. 阅读"版本管理"章节
- 预期结果：
  - 明确版本号规则（URL 路径 / Header / Query 三种方式选一）
  - 明确兼容性策略（向后兼容的最低要求）
  - 有弃用流程（deprecation timeline）

### 场景 4：分页/排序/过滤标准化
- 前置条件：列表接口需要分页
- 操作步骤：
  1. 阅读"查询参数规范"章节
- 预期结果：
  - 有标准分页参数（page/size 或 offset/limit）
  - 有排序参数格式（sort=field:asc）
  - 有过滤参数格式（filter[field]=value）

# BDD 规格: 架构设计规范

> 对应 `01-architecture_架构设计规范.md`（v2.0：第一步·编写计划的细化）

---

## 功能：架构设计规范文档

### 场景 1：声明所细化的流程环节
- 前置条件：无
- 操作步骤：
  1. 打开 `01-architecture_架构设计规范.md`
  2. 阅读开头引言
- 预期结果：
  - 明确声明本规范是 ai-workflow「第一步·编写计划」技术选型与架构设计环节的细化
  - 声明同时作为「第二步 §2.3 DeepSeek 审查」架构层面的判据
  - 含档位裁剪说明（轻量只守红线）

### 场景 2：红线配自动检测手段
- 前置条件：CI / 审查前
- 操作步骤：
  1. 阅读 §一「红线」
- 预期结果：
  - 红线含：禁止循环依赖、领域层不依赖框架/ORM、禁止跨层调用
  - 每条红线标注「怎么自动抓」（import-linter / dependency-cruiser）
  - 密钥硬编码不在此重复，指向 02

### 场景 3：落地配置可复制
- 前置条件：新项目搭建依赖约束
- 操作步骤：
  1. 阅读 §二「落地」
  2. 复制配置到项目
- 预期结果：
  - 提供可复制的 `importlinter.ini`（layers + forbidden 契约）
  - 说明 TypeScript 用 dependency-cruiser 对应
  - 明确「跑 lint-imports 即可拦截红线 1/2/3」

### 场景 4：分层与「代码放哪层」可判定
- 前置条件：读者不确定一段代码归属
- 操作步骤：
  1. 阅读 §三分层决策表
- 预期结果：
  - 四层（表现/应用/领域/基础设施）职责 + 不应包含项
  - 依赖方向单向向下
  - 有「这段代码是…→放哪层」决策表

### 场景 5：模块划分 / 配置 / 选型有决策表
- 前置条件：新项目搭建
- 操作步骤：
  1. 阅读 §四决策表
- 预期结果：
  - 模块划分按规模二选一（业务域 / 技术层）
  - 配置管理含环境变量、`.example`、分环境
  - 技术选型结论写入 ADR（指向收束节点）

### 场景 6：检查清单可操作
- 前置条件：审查或提交前
- 操作步骤：
  1. 逐项核对 §六检查清单
- 预期结果：
  - checkbox 格式，可「是/否」回答
  - 与红线一一对应
  - 不少于 5 项

### 场景 7：调用图 / 依赖黑洞内容已移交 08
- 前置条件：读者寻找调用关系可视化、依赖黑洞、孤儿模块内容
- 操作步骤：
  1. 在 01 中查找上述主题
- 预期结果：
  - 01 不再单独定义这些（去重）
  - §五 / §七 以一句话指向 08 作为唯一权威

# BDD 规格: AI 协作开发流程

> 更新: 2026-07-11
> 对应 `conventions/ai-workflow_AI协作开发流程/` 的 README 与 01-09 流程文档。

---

## 功能：AI 协作开发流程可执行、可拦截、可审计

### 场景 1：双轨制与端到端流程无歧义

- 前置条件：项目准备开始调研、设计或开发
- 操作步骤：
  1. 阅读流程 README、01-流程全景与 06-端到端流程
  2. 按任务规模选择长程开发或思考设计轨道
- 预期结果：
  - 需求、设计、实现、验证、收束和部署的入口明确
  - 复杂任务先形成 Owner 决策，未拍板项不得伪装为已交付
  - 工程模块、基础模块和功能模块的边界与装配顺序一致

### 场景 2：开发循环不跳过关键步骤

- 前置条件：Owner 已拍板，功能点进入实现
- 操作步骤：
  1. 执行仓库扫描、实现、验证、diff 审查、记录和提交
- 预期结果：
  - 每个步骤都有可执行入口和可复核证据
  - 可视化功能必须真实渲染验证；接口必须真实调用核对
  - 功能点完成后同步 worklog、STATUS 与开发清单

### 场景 3：进度投影来自同一事实源

- 前置条件：`STATUS.md`、`CLAUDE.md` 与 `docs/plan/开发清单.md` 存在
- 操作步骤：
  1. 运行 `python scripts/check_consistency.py`
- 预期结果：
  - 三处 `devguard-progress` 机器标记的 completed/total 完全相同
  - STATUS 的详细功能点编号连续且与 total 一致
  - 任一投影漂移都返回非零退出码，不以手写说明覆盖失败

### 场景 4：一致性阈值是实测结果

- 前置条件：规范、BDD、渲染产物和强制配置已提交
- 操作步骤：
  1. 运行 `python scripts/check_consistency.py`
- 预期结果：
  - 输出逐维通过数、总数和总分
  - 维度来自真实文件关系与命令结果，不使用写死百分比
  - 总分必须 `>= 80%`；低于阈值时进程失败

### 场景 5：强制性拦截率由故障注入证明

- 前置条件：所有治理 gate 可执行
- 操作步骤：
  1. 运行 `python scripts/check_enforcement.py`
  2. 在隔离临时 Git 仓库中注入提交信息、worklog、STATUS、文件放置、豁免、更新时间、文档同步和收束等违规
- 预期结果：
  - 每个注入项分别记录“被拦截/漏拦截”与实际退出码
  - 拦截率按 `被拦截数 / 总注入数` 计算，必须 `>= 90%`
  - 注入矩阵少于 10 项、命令未运行或异常退出都不得计为通过
  - 临时仓库必须在 baseline commit 前用本地空 `core.hooksPath` 隔离用户级/ECC 全局 Hook，防止外部 Hook 污染测量或造成递归超时

### 场景 6：收束节点四阶段缺一不可

- 前置条件：Owner 已明确触发收束
- 操作步骤：
  1. 依次执行整理、测试、审计和真实验证
  2. 生成收束报告与必要 ADR，并更新 STATUS 和开发清单
- 预期结果：
  - 一致性和强制性阈值属于审计阶段硬门禁
  - 未运行项明确标记为未执行，不写成通过
  - AI 只报告证据；Owner 签核状态保持独立

### 场景 7：工具不可用时降级但不伪造证据

- 前置条件：图查询、浏览器、外部扫描器或网络不可用
- 操作步骤：
  1. 使用仓库允许的本地替代工具继续可执行部分
  2. 在收束报告中记录未执行项、原因和影响
- 预期结果：
  - 可安全降级的探索不阻塞实现
  - 需要外部真实状态的门禁保持未通过/待验证，不因 fallback 自动变绿

# BDD 规格: 代码编写规范

> 对应 `02-coding_代码编写规范.md`（v2.0：第二步 §2.1 实现 + §2.3 审查的细化）
> 更新: 2026-05-30

---

## 功能：代码编写规范文档

### 场景 1：声明所细化的流程环节
- 前置条件：无
- 操作步骤：
  1. 打开 `02-coding_代码编写规范.md`
  2. 阅读开头引言
- 预期结果：
  - 明确声明细化「第二步 §2.1 Opus 实现」与「§2.3 DeepSeek 审查」
  - 声明 §六检查清单 = §2.3 审查清单的展开
  - 含档位裁剪说明

### 场景 2：红线配自动检测手段
- 前置条件：CI / 审查前
- 操作步骤：
  1. 阅读 §一「红线」
- 预期结果：
  - 红线含：禁 print/console.log、SQL 参数化、密钥禁硬编码、禁静默吞异常、禁注释代码块/调试断点、日志脱敏、输入校验
  - 每条标注检测手段（ruff 规则 ID / gitleaks / 审查）
  - 不少于 6 条

### 场景 3：落地配置可复制
- 前置条件：新项目落地代码规范
- 操作步骤：
  1. 阅读 §二「落地」
  2. 复制配置
- 预期结果：
  - 提供可复制的 `pyproject.toml`（ruff select 规则）
  - 提供 `.pre-commit-config.yaml`（ruff + gitleaks）
  - 说明 TypeScript 用 eslint + prettier 对应

### 场景 4：命名 / 结构以决策表 + 可调阈值给出
- 前置条件：开发者写代码
- 操作步骤：
  1. 阅读 §三决策表
- 预期结果：
  - 命名表覆盖变量/类/常量/布尔/文件，含示例
  - 结构阈值（函数行数/参数/成员/文件行数）明确标注为「可调警告阈值」而非铁律
  - 注释表区分「必须写 / 不要写」

### 场景 5：错误处理与日志有统一范例
- 前置条件：项目需统一错误处理
- 操作步骤：
  1. 阅读 §四
- 预期结果：
  - 有「尽早失败、明确传播、留上下文」的完整代码范例
  - 含具体异常类型、上下文管理器、日志级别
  - 错误响应格式指向 04

### 场景 6：检查清单 = 审查清单
- 前置条件：审 git diff 时
- 操作步骤：
  1. 逐项核对 §六检查清单
- 预期结果：
  - 前若干项对应红线
  - 含「无越界变更」「未重复造轮子（对照 CLAUDE.md 图谱）」等审查项
  - checkbox 格式

### 场景 7：代码可理解性内容已移交 08
- 前置条件：读者寻找 @entrypoint / @callers / CALL_GRAPH 标注
- 操作步骤：
  1. 在 02 中查找上述主题
- 预期结果：
  - 02 不再定义这些（去重）
  - §七以一句话指向 08 作为唯一权威

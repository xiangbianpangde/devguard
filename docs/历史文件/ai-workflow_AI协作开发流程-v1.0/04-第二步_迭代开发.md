# AI 协作开发流程 — 第二步：迭代开发

## 开发循环（每个功能点）

```
1. 扫描仓库 → 更新 CLAUDE.md
          ↓
2. Opus 实现
          ↓
3. 可观测验证（必须）
          ↓
4. DeepSeek 审查 diff
          ↓
5. 人确认差异点（只看审查标记的）
          ↓
6. 更新 plan/ + worklog + STATUS.md
          ↓
7. Git commit + tag
          ↓
8. 3 分钟回顾
          ↓
   → 回到 1，开始下一个功能点
```

---

### 2.1 Opus 实现

**前置步骤（每个功能点开始前）：**

0. **备份本地文件** — 动用户本地文件前必备份（git 不是备份，详见 [user-file-safety](../../user-file-safety.md)）。策略选其一：整目录克隆 / 提交 baseline / worktree 隔离。
1. **扫描仓库**：读取当前目录树和关键模块头，确认已有代码结构
2. **更新代码库图谱**：将模块依赖关系写入 `CLAUDE.md`（新会话自动加载，避免遗忘已有模块）
3. **标记可复用点**：在 `CLAUDE.md` 中标注当前功能可复用的已有模块

**开发规则：**
- 一次只做**一个功能点**，不做范围蔓延
- **采用 TDD 自检**：先写测试（红）→ 实现（绿）→ 重构。TDD 是 Opus 的自检手段，保证代码按 BDD 预期运行；它**不替代** DeepSeek 的独立审查关卡（§2.3）
- 代码产出包含测试
- 完成后立即跑验证命令
- 开发完成后更新 `CLAUDE.md` 的目录结构和模块图谱

---

### 2.2 可观测验证（硬性要求）

**每一个功能点必须留下可观测的客观证据**——核心是"不开黑盒"，证据形式由功能类型决定，截图只是其中一种，不强求一律截图。

| 功能类型 | 可观测证据（示例） |
|---------|-----------|
| 前端页面 | 浏览器截图 / 录屏 |
| API 接口 | Swagger 截图 / curl 输出 / 测试断言 |
| 数据库变更 | 表结构图 / 查询结果 / 迁移日志 |
| 算法逻辑 | console 输出 / 日志 / 测试用例输入输出 |
| CLI 工具 | 终端运行输出 |
| 重构 / 配置 | 测试全绿对比 / diff + 行为不变的证据 |

> 原则不是"必须截图"，而是"必须有他人能复核的客观证据"。对重构、配置类改动，TDD 测试全绿本身就是合格证据，无需为截图而截图。

**⚠️ 伪证据（不算可观测验证）**：对**前端 / 可视化 / 渲染才知对错**的功能点，下列都**不能**充当合格证据——

- 语法检查（`node --check` / `tsc --noEmit`）：只证明能解析，不证明能渲染。
- HTTP `200` / 接口可达：只证明文件能取到，不证明页面画对了。
- 自己另写脚本"复刻"数据变换：跑的是复刻逻辑，不是真身代码 + 真实 DOM。

> 前端改动**未经真实浏览器渲染就说"验证通过"=不实陈述**。说之前先自问：我亲眼看到它工作了吗？

**无 GUI / Agent 环境怎么取证**：没有 jsdom/puppeteer，只要系统有 Chrome，就用一次性截图（render-and-exit，不开守护进程、不占调试端口）——

```bash
# 1) 起静态服务（相对路径 fetch 需 HTTP）：python -m http.server 8080
# 2) 为交互视图做临时副本：加载后自动 switchView(...)/selectNode(...)
# 3) headless 渲染并退出，截图：
chrome --headless=new --disable-gpu --virtual-time-budget=6000 \
  --user-data-dir=<临时profile> --window-size=1440,1100 \
  --screenshot=out.png "http://localhost:8080/<临时页>.html"
# 4) 用能看图的工具打开 out.png 亲眼确认；测完删临时文件
```

> **Windows 坑**：想用 CDP `--remote-debugging-port` 驱动时，若用户已开着 Chrome/Edge，新进程会把 URL 交给已有实例、**端口不监听**。一次性 `--screenshot` 模式用独立临时 profile、渲染完即退，不受影响、更可靠。

### 2.3 DeepSeek 审查

将 `git diff` 发给 DeepSeek，附带 `plan` 中的当前功能点描述。

审查清单：
- [ ] 是否实现了 BDD 中所有场景
- [ ] 是否有越界变更（改了不该改的）
- [ ] 是否与已有模块功能重复（对照 `CLAUDE.md` 的代码库图谱）
- [ ] 是否有安全隐患
- [ ] 是否有遗留调试代码

### 2.4 人确认

只看 DeepSeek 标记的**差异点**和**高风险项**。正常情况下无需介入。

### 2.5 更新记录

- `docs/plan/开发清单.md`：当前功能点标记完成，修正后续估算
- `worklogs/YYYY-MM-DD_描述.md`：记录做了什么、遇到的问题、给下一位的交接
- `STATUS.md`：更新进度表

### 2.6 Git 提交

```bash
git add .
git commit -m "feat(xxx): 完成{功能点名称}"
git tag v0.1.0  # 里程碑时打 tag
```

### 2.7 回顾（3 分钟）

- 计划估算 vs 实际耗时（修正后续）
- 是否有意外发现（更新 `docs/research/`）
- 是否留下技术债（记录到 `STATUS.md` 的技术债区）

### 2.8 汇报（功能点级默认内联）

功能点完成后，**汇报默认在对话里内联输出**（必做的是 §2.5 基础文件更新；汇报是其对外提炼）。**不单独建 `docs/reports/` 文档**——仅当人明确要求留档时才落盘为 `docs/reports/功能点{编号}-{简短名}.md`。详见 `07-汇报.md`。

### 2.9 工具链增强：MCP 代码理解工具

在开发循环的 **第 0 步（扫描仓库）**和 **第 1 步（Opus 实现）**中，传统方式依赖 `grep` + `read_file` 循环探索代码库，效率低且容易遗漏上下文。推荐使用 MCP 代码理解工具替代：

> 仅中大型项目（> 10 模块 / > 5 万行）才需引入图谱工具，完整规范见 [08-代码理解与图谱规范](../08-code-understanding_代码理解与图谱规范.md)。

| 场景 | 推荐工具 | 效果 |
|------|---------|------|
| CI 流水线自动生成调用图 | **CodeGraph** | 零配置，增量文件监视，节省 token |
| 深度跨语言分析 | **CodeGraphContext** | KuzuDB/Neo4j 后端，死代码检测，圈复杂度 |
| 团队日常代码探索 | **Understand-Anything** | 交互式图可视化，自动导览 |
| Python 脚本分析 | **NetworkX + pydeps** | 轻量，适合自定义 CI 脚本 |

**为什么用图查询替代 grep**：图查询（如"谁调用了这个函数"）一次遍历即可得到完整上下游，而 grep 只能做文本匹配，无法识别间接依赖和调用链。具体性能/召回率对比数据见 `docs/research/` 下的调研报告（数据随工具版本变化，以调研为准，不在此固化具体数字）。

**集成建议**：
- 在每个功能点的**仓库扫描阶段**（§1），先跑 CodeGraph 更新调用图（增量模式，秒级完成）
- 在 **Opus 实现阶段**（§2.1），将调用图信息写入 `CLAUDE.md` 的模块图谱区，替代人工 grep
- 在 **DeepSeek 审查阶段**（§2.3），用代码图谱验证"变更影响范围"是否与 BDD 预期一致

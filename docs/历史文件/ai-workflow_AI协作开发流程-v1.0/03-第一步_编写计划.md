# AI 协作开发流程 — 第一步：编写计划

> 更新: 2026-06-10

### 1.0 档位选择（流程入口，第一个决策点）

写计划前先确定**流程档位**——避免小项目被拖进重型流程，也避免大项目走过场。

| 档位 | 适用 | 启用环节 |
|------|------|---------|
| **轻量** | 纯文档 / 小工具（功能点总数 < 5） | 计划 + 开发循环 + 整理/审计两阶段收束（跳过性能基准、PR 抽查等团队项） |
| **标准** | 一般软件项目 | 全流程：调研→计划→开发循环→四阶段收束 |
| **团队** | 多人协作的正式软件 | 标准 + 团队专属审计项（PR/CR 抽查、Review 实质性、分支策略） |

> 档位在 `docs/plan/开发清单.md` 头部声明：`> 流程档位: 轻量 | 标准 | 团队`。后续所有环节按档位裁剪——可视化、收束阶段数、审计项均随档位伸缩。

### 1.1 流程笔记
人先写流程式笔记，理解整体流程。输出到 `docs/plan/背景.md`。

### 1.2 功能拆分
以功能为粒度拆分检查点，每个功能点满足：
- 独立可测试 / 独立可演示 / 不超过一个对话轮次的开发量

### 1.3 BDD 规格
用统一模板为每个功能点编写 BDD。存放在 `docs/specs/`。

**BDD 模板：**
```markdown
## 功能：{功能名称}

### 场景 1：{场景描述}
- 前置条件：
- 操作步骤：
  1. ...
  2. ...
- 预期结果：

### 场景 2：{异常场景}
- 前置条件：
- 操作步骤：
- 预期结果：
```

### 1.4 技术选型
人使用 DeepSeek 为每个 BDD 场景标注实现路径（一句话即可）。

### 1.5 人与 DeepSeek 讨论完善

- 将粗浅想法与 DeepSeek 讨论，完善设计
- 遇到能力边界外的概念，先让 DeepSeek 解释实现路径，再决定是否纳入本期
- 如果存在概念不清等理解问题，哪怕一直对话，也要先理解，后写计划；当一个话题对话5轮以上，仍未理解，创建这个问题的专门笔记文件，总结之前的对话内容作为背景，将对话内容记录在笔记中，辅助理解

### 1.6 生成计划提纲
人让 DeepSeek 根据以上产出编写 `docs/plan/开发清单.md`，包含：
- 功能点列表（按优先级排序）/ BDD 引用 / 预估轮次 / 依赖关系
- **预设收束节点**：在清单头部声明收束间隔（默认每 3 个功能点）并标出每个收束节点落在哪个功能点之后，例如 `> 收束节点: 功能点3后 / 功能点6后 / ...`。预设后即成为**硬闸门**——AI 在到达收束节点却未执行收束时，拒绝开始下一个功能点（见 `06-第三步_收束节点.md` §2.1）。

### 1.7 系统模拟运行
对计划进行模拟运行，检查是否存在设计问题

### 1.8 交叉审查（Opus）
将计划交给 Opus 审查，**同时附上当前仓库状态**（目录树 + 关键文件头）。

检查点：
- 执行层面：计划是否可落地
- 架构层面：是否有设计缺陷
- 根据审查结果生成开发计划文件（`docs/plan/开发清单.md` + `STATUS.md`）
- 对优化后的计划进行第二次系统模拟运行

### 1.9 创建基础设施

```bash
git init
mkdir -p conventions docs/{research,specs,templates,plan/design,reports} worklogs/decisions src
touch docs/plan/背景.md docs/plan/开发清单.md STATUS.md
cat > CLAUDE.md << 'EOF'
# {项目名} — Claude 项目上下文
> 本文件在新会话中自动加载。每次功能点完成后更新。
## 目录结构   ## 模块依赖图谱   ## 当前开发状态   ## 关键决策
EOF
```

`STATUS.md` 模板：
```markdown
# 项目状态
> 更新: YYYY-MM-DD

## 当前进度
| 功能点 | BDD | 状态 | 完成日期 |
|--------|-----|------|---------|
| 用户登录 | specs/login.md | 进行中 | - |

## 阻塞项 / 技术债
（无）
```

`dashboard.html` 模板：见项目根目录 `dashboard.html`。

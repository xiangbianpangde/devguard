# AI 协作开发流程 — 第三步：收束节点

> 定义功能批次完成后的阶段性质量闸门。整理代码、集中测试、审计合规、验证效果——在进入下一批功能点之前，确保当前成果经得起审视。
>
> **红线级别**：收束节点是强制流程，跳过 = 后续功能点不通过。
> 更新: 2026-07-11

---

## 适用范围

- **项目类型**：所有使用本 AI 协作开发流程的软件项目
- **团队规模**：1 人以上均适用（单人项目同样需要停下来审视）
- **设定**：在第一步（计划）时预设收束节点（默认每 3 个功能点），写入 `docs/plan/开发清单.md` 头部，成为**硬闸门**
- **执行触发**：收束**由人触发执行**——AI 不自动启动收束四阶段
- **闸门拦截**：到达预设收束节点却未执行收束时，**AI 拒绝开始下一个功能点**并提醒人触发收束。即"AI 不替人做收束，但 AI 会拦住越过闸门的开发"——解决了"强制却仅靠自觉"的矛盾

---

## 一、核心原则

### 原则 1：先收束，再前进

没有通过收束节点的代码，不得进入下一批功能点的开发。收束不是"建议做完"——是"不做完不能走"。

**为什么**：连续开发 10 个功能点不做收束，代码库会积累僵尸代码、过时文档、隐藏 bug。等到发布前再回头修，成本是及时收束的 5-10 倍。

### 原则 2：AI + 人双线审计，互不替代

AI 负责广覆盖（扫描全量代码、检测模式违规），人负责深判断（业务逻辑合理性、设计意图偏离）。AI 发现不了"这个需求已经不需要了"，人发现不了"这个文件里有 3 个循环依赖"——两者互补。

**为什么**：AI 的强项是模式匹配和全量扫描，弱项是理解业务上下文。人的强项是判断"这样做对吗"，弱项是逐行检查几百个文件。双线审计让两者各做擅长的。

### 原则 3：效果验证 ≠ 功能测试

功能测试验证"代码按设计运行了"，效果验证追问"这个功能真的解决了问题吗"。BDD 通过了不代表用户体验好了，覆盖率 90% 不代表没有遗漏场景。

**为什么**：开发者容易陷入"把功能做完了 = 做对了"的错觉。效果验证强迫回到用户视角，防止精美的代码解决了错误的问题。

### 原则 4：收束产物必须落盘

收束过程中产生的所有报告、签核、数据必须写入仓库文件系统，不留在对话里。后续的你和未来的同事需要能翻出"v0.2 收束报告"来看。

**为什么**：收束是项目历史的里程碑。没有落盘的收束等于没做——下一个人不知道上次收束发现了什么问题、修了什么、还剩什么。

## 二、触发条件

### 2.1 触发规则

| 条件 | 说明 |
|------|------|
| **人在计划时预设（硬闸门）** | 第一步（计划）时设定收束节点（默认每 3 个功能点），写入 `docs/plan/开发清单.md` 头部 |
| **AI 闸门拦截** | 到达预设收束节点而收束未执行时，AI **拒绝开始下一个功能点**，并提醒人触发收束。AI 自己不启动收束四阶段，但会拦住越过闸门的开发 |
| **人触发执行** | 人确认"收束"后，由人主导执行四阶段 |
| **版本号变更** | 版本号变更时人可随时触发收束，不受间隔限制 |

### 2.2 可调整参数

| 参数 | 默认值 | 调整方式 |
|------|--------|---------|
| 收束间隔 N | 3 | 在 `docs/plan/开发清单.md` 头部声明 `> 收束间隔: 5` |
| 轻量收束 | 关闭 | 小项目（< 5 功能点总数）可声明 `> 收束模式: 轻量`，缩减为整理+审计两阶段 |

### 2.3 触发信号

当人确认"收束"后，在当轮功能点的"3 分钟回顾"之后，不立即开始下一个功能点——而是进入收束节点：

```
功能点 N 完成 → 3 分钟回顾 → 【收束节点触发】→ 四阶段 → 收束报告 → 功能点 N+1 开始
```

### 3.1 第一阶段：文件与代码整理

**目标**：消除本批次开发积累的熵增——僵尸代码、重复模块、过时文档。

#### 整理检查清单

##### 代码整理

| 检查项 | 工具/方法 | 依据规范 |
|--------|----------|---------|
| 清理未使用的 import | Linter（ESLint/Pylint）自动检查 | 02-代码 §2.7 禁止遗留调试代码 |
| 清理未使用的函数/变量 | CodeGraph：`codegraph_impact` 或 `codegraph_callers` 检测无调用者的函数 | 08-图谱 §一 |
| 清理注释掉的代码块 | `grep -r "//\|#.*\(TODO\|FIXME\|HACK\)"` 除外，其余直接删除 | 02-代码 §2.7 |
| 删除遗留调试代码 | `print()` / `console.log()` / 调试断点 | 02-代码 §2.7 |
| 识别并合并孤儿模块 | 图谱查询同职责分散模块 → 人工确认合并 | 01-架构 反模式5 |
| 检查依赖黑洞 | 图谱查询无调用者且无被调用者的模块 → 确认后删除或文档标注 | 01-架构 反模式4 |

##### 文件整理

| 检查项 | 工具/方法 | 依据规范 |
|--------|----------|---------|
| `worklogs/` 是否齐全 | 核对本批次每个功能点是否有对应 worklog | AI流程 §2.5 |
| 过时文档归档 | 逐份检查 `docs/` 下文档与代码一致性，过时移入 `archive/` | 06-文档 §2.7 |
| CLAUDE.md 模块图谱更新 | 核查目录结构、模块依赖是否反映当前代码状态 | AI流程 §2.1 |
| FILE_GRAPH.md 同步 | 本批次新增/移动/删除的文件是否已反映到文件图谱 | meta/FILE_GRAPH.md |
| 清理僵尸分支 | `git branch --merged` 确认已合入的功能分支已删除 | 03-Git §2.1 |
| **决策整理** | 回顾本批次 worklog 的"关键决策"段，判断哪些值得提升为 ADR → 写入 `worklogs/decisions/` | ADR 0006 |
| **开发清单更新** | 根据status文件更新开发清单 |  |

### 3.2 第二阶段：集中测试

**目标**：做 CI 增量测试不会做的事——全量回归、跨模块集成、手工探索。

#### 测试矩阵

| 测试类型 | 覆盖范围 | 执行方式 | 通过标准 |
|---------|---------|---------|---------|
| 全量单元测试 | 全部模块 | `npm test` / `pytest` 全量 | 100% 通过 |
| 全量集成测试 | 跨模块协作 | 全量集成测试套件 | 100% 通过 |
| 跨模块回归 | 本批次涉及模块 + 间接依赖方 | 按调用图影响分析执行（05-测试 §2.8） | 100% 通过 |
| 手工探索性测试 | 核心用户流程 3-5 条 | 人工按"用户会怎么用"随机路径操作 | 无阻塞问题 |
| 性能基准对比 | 关键 API / 页面加载 | 对比上一个收束节点的性能数据 | 无显著退化（> 20%） |

#### 测试产出

```markdown

### 性能基准
| 指标 | 上节点(v0.1) | 本节点(v0.2) | 变化 |
|------|-------------|-------------|------|
| 订单创建 P95 | 320ms | 345ms | +7.8% |
| 商品列表 P95 | 180ms | 175ms | -2.8% |
```

**目标**：全面检查本批次开发是否违反任何规范。AI 和人各司其职，互不替代。

```

        ↓                               ↓

        │                               │

        │                               │
        └───────────────┬───────────────┘
```

#### 3.3.1 AI 审计线

AI 对全量代码进行模式化扫描，输出结构化报告。**AI 不判断对错，只标记疑点**——每个标记项附带：位置（文件:行）、违反的规范条目、建议修复方案。

##### AI 审计维度

**架构红线扫描**（依据 01-架构规范）：

| 检测项 | 检测方式 | 风险 |
|--------|---------|------|
| 循环依赖 | CodeGraph：`codegraph_trace` 追踪调用链，检查是否有路径回到自身 | 🔴 高 |
| 跨层调用 | CodeGraph：`codegraph_callers` 追踪调用来源，检查是否有跨层跳级 | 🔴 高 |
| 领域层依赖框架 | `grep` 领域层文件中的框架 import | 🔴 高 |
| 敏感信息硬编码 | `grep` 密钥/密码/Token 模式 | 🔴 高 |

**代码红线扫描**（依据 02-代码规范）：

| 检测项 | 检测方式 | 风险 |
|--------|---------|------|
| `print()` 替代日志 | `grep "print("` 排除测试文件 | 🟡 中 |
| 裸 SQL 拼接 | `grep` 字符串拼接 + SQL 关键字 | 🔴 高 |
| 密钥硬编码 | `grep -E "(api_key|secret|password|token)\s*="` | 🔴 高 |
| 注释代码块 | `grep` 被注释的代码行（排除 TODO/FIXME/HACK） | 🟡 中 |
| 遗留调试代码 | `grep "console.log\|debugger\|print("` | 🟡 中 |

**Git 规范扫描**（依据 03-Git规范）：

| 检测项 | 检测方式 | 风险 |
|--------|---------|------|
| main 直接 push | 检查 Git reflog / 平台审计日志 | 🔴 高 |
| commit 格式 | `git log --oneline` 检查 Conventional Commits 格式 | 🟡 中 |
| 僵尸分支 | `git branch --no-merged` 检查未合入的旧分支 | 🟢 低 |

**API 规范扫描**（依据 04-API规范）：

| 检测项 | 检测方式 | 风险 |
|--------|---------|------|
| 错误码一致性 | 检查所有 API 返回的错误码格式 | 🟡 中 |
| 废弃接口未标注 | `grep "deprecated\|@deprecated"` 检查是否有未标注的废弃接口 | 🟡 中 |
| 版本管理 | 检查是否有破坏性变更未升主版本号 | 🔴 高 |

**测试规范扫描**（依据 05-测试规范）：

| 检测项 | 检测方式 | 风险 |
|--------|---------|------|
| 覆盖率下降 | 对比上节点覆盖率数据 | 🟡 中 |
| Flaky test | 检查 CI 中最近 5 次的测试通过率波动 | 🟡 中 |
| 未测试的 public API | CodeGraph：`codegraph_explore` 列出 public 函数，交叉比对测试覆盖 | 🟡 中 |

**图谱缺陷检测**（依据 `08-code-understanding_代码理解与图谱规范.md` §一，通过 CodeGraph MCP 工具执行）：检测项包括循环依赖（`codegraph_trace`）、未使用函数（`codegraph_impact`）、上帝类（`codegraph_explore` 统计函数数）、跨层违规（`codegraph_callers` 检查调用来源）等。详见 08 规范。

#### 3.3.2 人审计线

人在 AI 审计之后进行，聚焦 AI 无法判断的领域。**人审计的核心价值是"理解上下文"**。

> **档位适配**：标 `[团队]` 的检查项仅在团队档执行；轻量档 / 单人项目可跳过这些项，不算违规（单人无 PR/CR 协作场景）。其余项所有档位通用。

##### 人审计清单

**AI 标记项复核**：
- [ ] 逐条确认 AI 审计报告中的 🔴 红线违规——是否真的是问题？
- [ ] 抽查 🟡 中风险项——有没有 AI 误报或遗漏？

**业务逻辑合理性**：
- [ ] 本批次的业务逻辑是否有互相矛盾的地方？（如：订单模块允许负数价格，但支付模块拒绝）
- [ ] 是否有"为了通过测试而写代码"的情况？（如：硬编码返回值来通过单元测试）
- [ ] 是否有"技术上正确，业务上错误"的实现？

**设计意图偏离检查**：
- [ ] 本批次实现是否偏离了 `docs/plan/背景.md` 和对应 BDD 的原始设计意图？
- [ ] 如果有偏离，是合理演进还是无意漂移？合理偏离需补写 ADR（`worklogs/decisions/`，仅收束节点）

**PR/CR 流程合规抽查** `[团队]`：
- [ ] 随机抽取 2-3 个功能点的 PR，检查：描述是否完整、Review 是否有实质反馈、合入前是否通过 CI
- [ ] 是否有 "自己 Approve 自己" 的 PR？

**主观质量判断**（AI 无法替代）：
- [ ] 新代码的命名是否能让团队其他人一眼看懂？
- [ ] 新代码的结构是否合理？（如果让另一个开发者接手，他能不能2小时内理解？）
- [ ] 是否有"这里感觉不对劲"的直觉性发现？——记录为"待讨论"，不阻塞收束

##### 人审计签核模板

> 模板文件：`docs/templates/人审计签核模板.md`。签核完成后结论合入收束报告。

#### 3.3.3 双线合入

| AI 审计结论 | 人审计结论 | 合入结果 |
|------------|-----------|---------|
| 无 🔴 红线 | 通过 | ✅ 审计阶段通过 |

### 3.4 第四阶段：效果验证

**目标**：回到用户视角，验证"我们做的功能真的解决问题了吗"。

#### 效果验证清单

**BDD 场景回演**：
- [ ] 逐条对照 `docs/specs/` 中本批次功能点的 BDD 场景
- [ ] 每个场景在当前版本上重新执行一遍（非开发时的截图，是现在重新跑）
- [ ] 异常场景（`### 场景 N: {异常场景}`）是否仍然按预期处理？

**可视化产出复核**：
- [ ] 本批次每个功能点的可视化产出（截图/录屏/curl输出）在当前版本重新生成
- [ ] 与开发阶段的截图对比——是否有退化？

**用户故事回溯**：
- [ ] 回到 `docs/plan/背景.md`，重新读"为什么做这个项目"
- [ ] 本批次的 3 个功能点，是否让你离目标更近了？还是偏到了另一个方向？
- [ ] 如果现在给一个不认识这个项目的人演示，他能理解这些功能的价值吗？

**技术债盘点**：
- [ ] 整理本批次的"3 分钟回顾"中记录的技术债
- [ ] 本收束节点新发现的问题
- [ ] 统一记录到 `STATUS.md` 的技术债区，标注发现时间、优先级、预计修复节点

## 五、与微观回顾的区别

很多人会问：每个功能点结束已经有"3 分钟回顾"了，为什么还要收束节点？

| 维度 | 3 分钟回顾（微观） | 收束节点（中观） |
|------|-------------------|-----------------|
| **粒度** | 单个功能点 | 一批功能点（3-5个） |
| **关注点** | 本功能点是否按计划完成 | 一批功能点之间是否协调 |
| **测试** | 本功能点的增量测试 | 全量回归 + 跨模块集成 |
| **审计** | 不审计（靠 DeepSeek 审查 diff） | AI + 人双线全量审计 |
| **验证** | 可视化截图（本功能点） | BDD 回演 + 用户故事回溯 |
| **产出** | worklog + STATUS 更新 | 收束报告 + AI审计报告 + Git Tag |
| **决策** | 修正后续估算 | 决定是否调整方向、重排优先级 |
| **主导** | AI 为主，人确认 | AI + 人并行，人拍板 |

**一句话区分**：3 分钟回顾回答"这个功能点做完了吗"，收束节点回答"这一批功能点做对了吗"。

## 六、反模式

### ❌ 反模式 1：收束走过场

```
收束节点到了 → 随便填几个 ✅ → 收束报告写"一切正常" → 继续开发
```

**问题**：把收束当成行政流程而非质量闸门。积压的问题不会因为"标记通过"而消失，只会在后续加倍偿还。

### ✅ 正确做法：收束是真正的质量对话

- AI 审计报告中出现 🔴 红线必须修复，不得以"后续再说"跳过
- 人审计中发现"感觉不对劲"必须写下来（标记为"待讨论"），即使不阻塞收束
- 每一次收束至少发现 1 个需要改进的点——如果"完美无问题"，可能是不够深入

---

### ❌ 反模式 2：只做 AI 审计，不做人审计

```
收束节点 → 跑完 AI 审计脚本 → 报告显示 0 红线 → "通过了" → 不通知人复核
```

**问题**：AI 只能检测模式化问题，无法判断"业务逻辑是否正确"、"设计意图是否偏离"。一个功能可能通过所有 AI 检查但仍然是个错误的功能。

### ✅ 正确做法：双线缺一不可

- AI 审计和人审计都必须执行，且人审计在 AI 之后（先看 AI 报告，再独立判断）
- 人审计的核心价值是"AI 发现不了的东西"——不要把人的时间花在复核 AI 的每一行输出上，聚焦在 AI 不擅长的领域
- 如果只有一个人开发，人审计也必须做——换一个时间、换一个视角（比如第二天早上），用"如果是别人写的代码，我能看懂吗"的心态审视

---

### ❌ 反模式 3：收束阶段还在写新功能

```
收束节点开始了... "趁收束的空档，我把下一个功能点的小优化做了吧"
```

**问题**：收束节点期间写入的新代码未经审计就混入了本批次。下一次收束时，这些代码可能被遗忘。更糟的是，它们可能引入新的违规，让审计结论失效。

### ✅ 正确做法：收束期间冻结功能开发

- 收束节点的四阶段只做整理、测试、审计、验证——不写新功能
- 如果审计要求修改代码（如修复 🔴 红线），修改后重新跑对应阶段的检查
- 收束期间发现的"可以做的小优化"记入技术债或下一个功能点的计划，不在收束中实现

---

### ❌ 反模式 4：用收束替代日常规范遵守

```
"没事，先这么写吧，收束的时候再改"
```

**问题**：把收束节点当成"垃圾回收站"，日常开发中随意违反规范，指望收束时一把修。结果收束时发现问题堆积如山，修复成本远超日常遵守的微成本。

### ✅ 正确做法：日常遵守 + 收束兜底

- 日常开发中严格遵守规范（每个功能点的 DeepSeek 审查会拦截明显违规）
- 收束节点是"查漏补缺"，不是"集中还债"——如果收束时发现大量违规，说明日常流程执行不到位，需要反思
- 如果连续两次收束都发现同一类违规，在 CLAUDE.md 中强化对应规范的提醒

## 七、检查清单

### 触发阶段
- [ ] 是否满足触发条件（≥ 3 个功能点完成 OR 版本号变更）？
- [ ] 是否已冻结新功能开发（收束期间不写新代码）？

### 阶段1：整理
- [ ] 未使用的 import/函数/变量是否已清理？
- [ ] 注释掉的代码块是否已删除（TODO/FIXME/HACK 除外）？
- [ ] 遗留调试代码（print/console.log）是否已清理？
- [ ] 孤儿模块是否已识别并处理（合并/废弃标注）？
- [ ] 依赖黑洞是否已识别并处理（删除/文档标注）？
- [ ] 过时文档是否已归档（archive/）？
- [ ] CLAUDE.md 模块图谱是否已更新？
- [ ] FILE_GRAPH.md 是否已同步？
- [ ] 僵尸分支是否已清理？
- [ ] 决策整理：本批次 worklog 中的"关键决策"是否已回顾并判断是否需要提升为 ADR？

### 阶段2：测试
- [ ] 全量单元测试是否 100% 通过？
- [ ] 全量集成测试是否 100% 通过？
- [ ] 跨模块回归测试是否覆盖了本批次所有影响范围？
- [ ] 手工探索性测试是否执行（≥ 3 条核心用户流程）？
- [ ] 性能基准与上一个收束节点对比是否有显著退化？

### 阶段3：审计
- [ ] AI 审计是否已执行（架构/代码/Git/API/测试/图谱 全维度）？
- [ ] AI 审计报告是否已落盘（`docs/reports/AI审计-vX.Y.md`）？
- [ ] 🔴 红线违规是否已全部修复（或确认为误报）？
- [ ] 人审计是否已执行（AI 标记复核 + 业务逻辑 + 设计意图 + PR 抽查）？
- [ ] 人审计签核是否已完成？
- [ ] 双线合入结论是否为"通过"？

### 阶段4：验证
- [ ] 本批次 BDD 场景是否逐条回演通过？
- [ ] 可视化产出是否重新生成并与开发阶段对比？
- [ ] 用户故事回溯是否确认方向未偏离？
- [ ] 技术债是否已更新到 STATUS.md？

### 收束关闭
- [ ] 收束报告是否已落盘（`docs/reports/收束报告-vX.Y.md`）？
- [ ] STATUS.md 是否已更新（收束节点行 + 技术债）？
- [ ] 开发清单是否更新
- [ ] CLAUDE.md 是否已更新？
- [ ] Git Tag 是否已打（`vX.Y`）？

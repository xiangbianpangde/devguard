01 · 模块定位

scripts/ 是什么 / 不是什么

scripts/ 是 devguard 的自动化工具集——把 4 类上游输入翻译成 4 类下游产物。理解它的正反面，比理解它"能做什么"更重要。

它是什么（4 类）

  • 渲染枢纽   render_meta.py 从 _meta.yaml 投射到 4 个产物
  • L1 红线   5 个 check_*.py + lint_markdown.py
  • 入口工具   仪表盘启动 + L4 统计 + 历史一次性
  • 跨平台   start_server.py 走 127.0.0.1 回环 / .bat 走 Windows 启动器
⇄

它不是什么（4 类）

  • ✗ 不是配置中心 —— 配置在根目录 .pre-commit-config.yaml / ci.yml
  • ✗ 不是产品代码 —— 不参与生产运行时，只服务仓库自身
  • ✗ 不是规范正文 —— 规范在 conventions/，scripts 是"执行器"
  • ✗ 不是测试 —— L4 测试在 tests/conventions/，scripts 只采集

--render / --check / 单 target

同一脚本两用，CI 校验与开发者渲染无歧义。支持 4 种用法：

python scripts/render_meta.py --render # 全量渲染 4 产物 python scripts/render_meta.py --check # 比较当前产物与预期 python scripts/render_meta.py --render pre-commit # 单 target 渲染 python scripts/render_meta.py --check pre-commit # 单 target 校验

退出码 · 区分环境故障与漂移违规

0 = OK · 1 = 渲染失败（环境故障 / yaml 缺节） · 2 = 校验失败（漂移违规）

CI 阶段可通过退出码分辨"机器坏了"还是"人没改规范"——前者告警，后者阻断。

仅 pyyaml · 不引入 jinja2 / click

依赖克制是设计纪律：不引入 jinja2 / click 等，跨环境一致（pip install pyyaml 是绝大多数 Python 环境的最低公约数）。模板逻辑用 Python f-string + 简单 split 即可。

pyyaml 1 jinja2 0 click 0 typer 0

UTF-8 显式强制

open(..., encoding="utf-8") 显式声明——避免 Windows ANSI 损坏中文（如规范名、worklog 文件名、commit message）。

这是 Windows 用户最常踩的坑：默认 encoding 在中文 Windows 是 GBK，读 UTF-8 文件直接抛 UnicodeDecodeError。

钩子排序 · 最便宜的最先

ADR 约定：pre-commit/pre-commit-hooks 排最前（trailing-whitespace / end-of-file-fixer 等无外部依赖），然后是 ruff / gitleaks / markdownlint。

原则：先做最便宜的检查——提前 fail 不浪费后续昂贵钩子的时间。

模板版本同步

_meta.yaml 改了 → 同步 docs/templates/devguard/conventions/_meta.yaml（B3 策略的纪律落地）。

不模板同步会导致：新项目 fork 时渲染产物漂移——B3 策略要求模板与生产仓库严格一致。

06 · 与上游（真源）的同步纪律

VERDICT · 范式总结

scripts/ 是 devguard 的"规范执行器"

它不写规范、不配 CI、不写产品代码、不写测试——它把规范（_meta.yaml）翻译成机器可读的钩子配置，把 L4 测试结果翻译成 dashboard 数据，把 commit-msg 翻译成"是否引 worklog"的二元判断。370 行的 render_meta.py 是这个翻译器的中枢——也是 devguard 强制约束范式的单点权威。所有"规范变了" → "产物必须跟着变"的纪律，由这一个脚本承担。

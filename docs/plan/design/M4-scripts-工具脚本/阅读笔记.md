## 设计.md 章节导航

### §一 模块定位
- **看什么**：它是自动化工具集；不是配置中心 / 不是产品代码 / 不是规范正文 / 不是测试
- **想什么**：为什么 scripts 不和 conventions 合并？—— 因为 scripts 是"执行器"而 conventions 是"规则源"，职责分离

### §二 目录结构
- **看什么**：10 个脚本（9 Python + 1 .bat），1039 行总计
- **易踩坑**：fix_render_date.py 看起来是"空文件"——其实是 V0.3 临时脚本的存档约定（intentional empty）

### §三 核心设计：渲染枢纽 render_meta.py
- **看什么**：B3 渲染策略 + 4 个产物 + 6 个关键设计判断
- **核心理解**：**改 `_meta.yaml` → 4 个产物自动同步**——整个 devguard 的真源到下游的桥梁
- **易踩坑**：钩子排序（pre-commit/pre-commit-hooks 最便宜排最前）是 ADR 约定，不能随便改

### §四 10 脚本逐个介绍
- **看什么**：每个脚本的输入/输出/触发场景/核心功能/对应规范
- **想什么**：为什么 L1 检测脚本要"一篇规范 = 一个 check_*.py 文件"？—— 与 `_meta.yaml.l1_check` 字段一一对应，便于 CI 编排
- **易踩坑**：
  - `check_ai_workflow.py` 是"章节级 L1"（比"关键内容"更严），必须看 7 篇流程文档的 §一 关键词
  - `collect_l4_stats.py` 输出 `L4_STATS=65/65` 这种 last line 格式是有意为之（CI `sed` 提取）
  - `start_server.py` 强制 `Cache-Control: no-store` 是为了避免浏览器缓存 STATUS.md

### §五 L 级别体系
- **看什么**：L0=准入 / L1=红线 / L4=测试（本模块分布）
- **想什么**：为什么 scripts 没有 L2 警告脚本？—— 警告类由规范正文承担，scripts 只执行硬约束

### §六 同步纪律
- **看什么**：改 `_meta.yaml` 必须 4 步走（render + 同步模板 + commit 标记 + 兜底钩子）
- **核心理解**：同步纪律是 B3 策略的纪律落地

### §七 边界与降级
- **看什么**：7 种降级场景 + 4 项明确不做
- **易踩坑**：跨平台兼容是核心约束，Windows .cmd shim / 命令行 8KB 限制 / 中文路径乱码都要处理

### §八 验收标准
- **看什么**：8 个维度 + 通过标准
- **易踩坑**：dashboard 漂移检测靠 `git diff --exit-code dashboard.html`，手改 dashboard.html 会被 CI fail

## 思考题

1. **为什么 scripts 不和 conventions 合并？** —— 因为 scripts 是"执行器"而 conventions 是"规则源"，职责分离。scripts 改一处需要回退整个 yaml，不应该耦合。
2. **L 级别体系是否需要扩展到 L2 警告？** —— 当前警告由规范正文承担，scripts 只执行硬约束。如果未来加 lint 警告类脚本，会形成"L0 准入 + L1 红线 + L2 警告 + L4 测试"4 级体系。
3. **渲染漂移漏跑如何防住？** —— pre-commit 钩子 `meta-render` 兜底（V0.1 设计预留），CI `test` 阶段 `--check` 兜底。开发者本地如果钩子没装就 commit，会被 CI 拦下。
4. **start_server.py 能否用现成工具替代？** —— 试过 `python -m http.server`，但它不支持强制 `Cache-Control: no-store`，导致浏览器缓存 STATUS.md，dashboard 不更新。当前自写 QuietHandler 是最小成本解。
5. **临时脚本（fix_render_date.py）应该删掉还是保留？** —— 保留作为"临时脚本存档"约定示例（intentional empty + 文件头说明），让后续读代码的人知道"为什么是空文件"。删除会让历史 context 丢失。
6. **dashboard 漂移检测为何用 `git diff --exit-code`？** —— 因为 dashboard.html 应该是渲染产物而非手改产物，任何手改都会引入漂移，CI 必须 fail。

## 易踩坑速查

| 症状 | 真因 | 修复 |
|------|------|------|
| 改了 `_meta.yaml` 但 dashboard 不更新 | 没跑 `render_meta.py` | pre-commit 钩子兜底 + 习惯性 `--render` |
| `.pre-commit-config.yaml` 被 CI 覆盖 | 那是渲染产物 | 改 `_meta.yaml.pre_commit` 节 + 跑 `--render` |
| commit 失败提示"必须引用 worklog" | `check_worklog_ref.py` 拦下 | commit message 加 `worklogs/YYYY-MM-DD_*.md` 引用 |
| markdownlint 误报"附加文件" | markdownlint-cli 附加 glob 行为 | 用 `lint_markdown.py` 包装 + 分批调用 |
| 仪表盘 8080 端口冲突 | 其他程序占用 | start_server.py 自动顺延；或显式指定端口 `start_server.py 9000` |
| L4 统计 0/0 | collect_l4_stats.py 解析失败 | V6.3 retry + V7.2 `${VAR:-0}` 防御；CI 重跑 |
| Windows 双击 bat 窗口闪退 | `pause` 缺失 | 当前脚本末尾有 `pause`（保留启动信息） |
| AI 流程新增第 8 篇但 CI 红 | `check_ai_workflow.py` 关键词表未更新 | 在 EXPECTED_FILES_AND_KEYWORDS 加新篇 |
| 08 AST 函数改名但 CI 不报 | 期望函数名硬编码在脚本中 | 改 `EXPECTED_FILES_AND_FUNCS` 或同步 `src/code-understanding/call_graph_example.py` |
| 临时脚本空文件困惑后续读者 | 没有"intentional empty"说明 | 保留为示例 + 文件头注释解释（fix_render_date.py 模式） |
| devguard fork 到新项目脚本漂移 | 没同步 `docs/templates/devguard/conventions/_meta.yaml` | 模板版本同步纪律（commit 标 `[meta]`） |
| 跨平台 subprocess 失败 | Windows npx .cmd shim | `shell=True` 强制；V6.3 用 Python 跑 pytest 替代 bash grep |

## 一、核心发现（来自调研工具的报告）

调研工具采集了55份资料，自动生成了知识树+报告。其中关键结论：

### 1.1 代码图谱的价值已被量化验证

| 指标 | 数据 | 来源 |
|------|------|------|
| 实体抽取召回率（Java） | 92% | 报告§3.1.1 |
| 实体唯一性准确率（消歧后） | 78%→94% | 报告§3.1.1 |
| 关系抽取F1值（C++ 10万行） | 0.87 | 报告§3.1.2 |
| 增量更新 vs 全量重建 | 节省85%计算资源 | 报告§3.1.3 |
| 跨语言实体对齐准确率 | 89%（Java/Python/JS） | 报告§3.1.3 |
| 图遍历查询：Neo4j vs MySQL | 快100-1000倍 | 报告§3.4.1 |
| 缺陷检测准确率（VulGraph） | 91.2% | 报告§5.2 |

### 1.2 四大核心争议 = 规范必须回答的设计决策

报告§4指出了以下规范编写时必须面对的权衡：

| 争议 | 维度 A | 维度 B | 规范建议 |
|------|--------|--------|---------|
| 静态分析 vs 深度学习 | 精确、可解释、规则成本高 | 泛化强、黑盒、依赖标注 | **混合精度**：静态分析做基础图谱，LLM做语义增强 |
| 图DB vs 关系型DB | 遍历快、专用、生态待成熟 | 成熟、事务完善、深度查询慢 | **分层存储**：小项目SQLite+FTS5，大项目KuzuDB/Neo4j |
| 细粒度 vs 粗粒度 | AST节点级，信息丰富成本高 | 函数/类级，简单高效丢细节 | **按需粒度**：默认函数/类级，关键模块可下沉到AST级 |
| 通用性 vs 专用性 | 多语言支持好但深度不足 | 单语言深耕，局限性大 | **渐进支持**：先覆盖主力语言，再扩展 |

### 1.3 五大应用场景 → 规范的对应输出

| 报告§5应用场景 | 对规范的要求 |
|---------------|-------------|
| 代码搜索 | 要求项目必须产出调用图/依赖图 |
| 缺陷检测与安全审计 | 要求图查询模板库 + CI集成 |
| 代码重构 | 要求影响分析作为重构前置步骤 |
| 代码推荐与补全 | 要求图谱支持实时增量更新 |
| 文档生成 | 要求文档与代码实体双向链接 |

## 二、实施方案

不再纠结「方案A还是方案B」。而是**两步走**：

### 第一步：在现有规范体系上做最小侵入式增强（本周可做）

在现有6+1规范中嵌入「图谱可理解性」要求，改动最小、落地最快。

| 现有规范 | 新增内容 | 改动量 |
|---------|---------|--------|
| **01-架构设计规范** | §新增小节「依赖关系可视化」：要求每个模块给出模块间调用图，作为架构文档标配。反模式增加「依赖黑洞」「循环依赖」「孤儿模块」。 | ~200字 |
| **02-代码编写规范** | §新增小节「代码可理解性」：要求函数/类的调用关系标注、入口点标注、public API必须有调用者/被调用者分析。 | ~300字 |
| **05-测试规范** | §影响分析测试：要求变更时通过调用图识别受影响测试用例，而非全量跑测试。 | ~150字 |
| **06-文档规范** | §知识互联要求：文档之间必须建立双向链接（wiki模式），文档必须关联对应代码实体。同时新增「代码地图」章节作为大型项目文档标配。 | ~300字 |
| **07-AI协作开发流程** | §工具链增强：将 MCP 代码理解工具（如 CodeGraph / CodeGraphContext / Understand-Anything）列为推荐工具，用于替代人工 grep/read 循环。 | ~200字 |

**改动总量**：约5处，共~1200字，不新增文件。

### 第二步：新建第08号规范「代码理解与图谱」（按报告结构，后续迭代）

当第一步的增强落地并积累经验后，正式新建独立规范。其章节结构直接映射调研报告的知识树：

## 四、推荐工具具体建议

基于调研报告的§3.4和§6工具对比，推荐分工：

| 场景 | 推荐工具 | 理由 |
|------|---------|------|
| CI流水线自动生成调用图 | **CodeGraph** | 零配置单二进制，增量文件监视，Token节省57% |
| 深度跨语言分析 | **CodeGraphContext** | 支持KuzuDB/Neo4j，死代码检测，圈复杂度 |
| 团队日常代码探索 | **Understand-Anything** | 15+平台，交互式图可视化，自动导览 |
| 文档知识库 | **LLM Wiki** | Karpathy模式完整实现，4信号图谱，Louvain聚类 |

## 五、预期收益（按报告数据推算）

- **新成员 Onboarding**：通过调用图 + 自动导览，理解时间减少40-60%
- **AI编码助手效率**：图查询替代grep/read循环，Token消耗降低57%（CodeGraph基准）
- **缺陷提前发现**：图模式匹配 + CI集成，缺陷召回率提升23%（报告§3.2.2）
- **重构安全**：影响分析全覆盖，减少80%的"改A坏B"事故
- **跨团队复用**：可探索的代码地图，复用率预期提升

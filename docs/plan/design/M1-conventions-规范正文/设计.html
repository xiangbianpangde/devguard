01 · 模块定位

它是什么 · 不是什么

✓ 它是规则源

所有"红线、警告、推荐"在此定义，通过 _meta.yaml 真源向外辐射到 pre-commit 钩子（10 个）、CI 5 阶段、L4 套件（65 tests）、dashboard 与各篇规范顶部的"分级标签"。

规则源 真源 辐射

✗ 不是文档库

文档库是 docs/（M2 模块）。conventions/ 只放规范本身和导航，不放 BDD、模板、调研报告。文档须随码同 PR 但不归 conventions 管辖。

非 docs/ 非 BDD

✗ 不是配置中心

实际钩子 / CI 配置在根目录（.pre-commit-config.yaml、.github/workflows/），conventions/ 只放"真源 yaml"——配置由 render_meta.py 渲染生成，禁止手改。

非根配置 禁止手改

✗ 不是示例代码

可运行示例在 src/<规范维度>/（M3 模块）——例如 src/coding/ruff.toml 是 02-coding §二落地配置的可运行版。conventions/ 只讲规范，src/ 讲"按规范跑起来"。

非 src/ 非可执行

边界：与 M2/M3 的关系

M1（conventions）讲"规则 + 红线"，M2（docs）讲"BDD 验收 + 模板 + 调研"，M3（src）讲"按规范跑起来"。三方互检：改规范必同步 BDD + 落地配置。

M1 规则 M2 验收 M3 落地

核心价值：单点变更全网同步

改 _meta.yaml 一个字段 → pre-commit 钩子、CI 配置、dashboard 数据、CLAUDE-规范导航 路由表 全部自动同步——避免"文档与代码漂移"的经典问题。

单点变更 全网同步 零漂移

03 · 真源 + 渲染策略（B3 变体）

为什么是 B3，而不是 A 或 C

三个备选方案对比，揭示 B3（真源 + 渲染）如何在"单源无漂移"与"正文可读性"之间取得平衡。

A 方案 · 全集中（单 yaml + 钩子全部生成）

把所有规范的正文、红线、配置全部结构化为 yaml，单一真源。钩子 / CI / 仪表盘全部由 yaml 字段生成。

  • + 真正的单源，无任何漂移
  • − 规范正文被压缩成结构化 yaml，可读性差
  • − 团队成员抗拒"用 yaml 写规范"
→

B3 方案 · 真源 + 渲染（当前选择）

_meta.yaml 只放元数据（红线数、检测工具、路由等），正文仍是自由 markdown。渲染脚本生成钩子 / CI / 仪表盘。

  • + 元数据集中，正文自由（人爱读）
  • + 钩子配置可再生（改 yaml 必同步产物）
  • − 需维护 render_meta.py 渲染脚本

C 方案 · 全分散（每篇 frontmatter）

每篇规范 .md 顶部用 frontmatter 自带元数据。仪表盘需要遍历所有文件聚合。

  • + 文件自包含，移走单篇不破坏元数据
  • − 难以全局聚合，仪表盘要遍历 17 文件
  • − 改一处需手改多处 frontmatter
→

B3 优势 · 元数据集中 + 正文自由

改 _meta.yaml 一行 l3_route 字段，CLAUDE-规范导航 + dashboard 全部自动同步，避免"文档与代码漂移"。

  • + 单文件可读（人读 markdown）
  • + 全局可聚合（机读 yaml）
  • + 改 yaml → 全网同步

双轨制 · 研究 vs 工程

研究类任务（调研 / 原型 / POC）

  • 验收方式：输出报告即可
  • 流程：跳过 TDD / E2E
  • 钩子：L1 检测保留
  • worklog：汇报要点即可
  • 典型：技术选型 / 性能基准 / 概念验证
→

工程类任务（功能交付 / 重构）

  • 验收方式：真机 E2E + 完整链路
  • 流程：严格走全流程
  • 钩子：10 钩子全开
  • worklog：每功能点必写
  • 典型：新功能 / 重构 / 修复 / 集成
!

09 · 边界与验收

降级行为 + 明确不做 + 5 维验收

边界清单是规范"何时退让"的硬规则。降级行为全部落盘到 worklog，技术债透明。

降级行为 · 4 场景

① _meta.yaml 改了但没跑 render_meta.py
降级：pre-commit 钩子 meta-render 会自动跑（如果配置了）；否则 CI 阶段 2 render_meta.py --check 检测漂移，漂移即 fail。
② 某规范 L1 检测工具不可用
降级：dashboard 标红 + worklog 记录技术债。CI 阶段 3 L4 套件可继续跑，但缺工具的规范健康度为 0%。
③ AI 流程新增第 11 篇
降级：必须先入 _meta.yaml（id=07-ai-workflow 下的 sub_files）+ 更新 CLAUDE-规范导航 的 l3_route。否则 AI 找不到该流程文档。
④ 治理规范文件缺失
降级：CI 阶段 4 compliance 报错阻断 PR。如 LICENSE 或 SECURITY.md 缺失，PR 不能合并。

明确不做 · 3 项

✗ 不做 · 规范正文 yaml 结构化

牺牲可读性。规范是"人读的文件"，结构化 yaml 让人读不下去。

✗ 不做 · 渲染产物入 git

会被 CI 覆盖产生噪音。所有渲染产物（钩子配置、dashboard.html）由 CI 阶段 5 build 阶段再生。

✗ 不做 · 手动维护分级标签

必须由 render_meta.py 生成。手改会被 CI 阶段 2 检测为漂移。

M1 规范正文 · 一句话定位

17 份规范 + 10 篇流程 + _meta.yaml 真源 + 双入口导航 = 整个 devguard 的"规则大脑"

M1 不是孤立文档，而是 M2 docs / M3 src / 外部钩子 + CI + L4 + dashboard 的"真源中枢"。改 _meta.yaml 一行 → 全网 5 个产物同步。AI 走 CLAUDE-规范导航 路由，人走 README-规范导航 速查。

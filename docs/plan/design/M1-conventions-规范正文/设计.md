## 一、模块定位

### 它是什么

`conventions/` 是 devguard 的**规则源**。整个项目的所有"红线、警告、推荐"全部在这里定义，并通过 `_meta.yaml` 真源向外辐射到：

- **pre-commit 钩子**（10 个）
- **CI 5 阶段**（lint / test / l4-conventions / compliance / build）
- **L4 测试套件**（65 tests passed）
- **进度仪表盘**（`dashboard.html`）
- **每篇规范顶部的"分级标签"小节**（渲染产物）

### 它不是什么

- **不是文档库** —— 文档库是 `docs/`（M2）；conventions 只放规范本身和导航
- **不是配置中心** —— 实际钩子/CI 配置在根目录，conventions 只放"真源 yaml"
- **不是示例代码** —— 可运行示例在 `src/<规范维度>/`（M3）

## 三、核心设计：真源 + 渲染策略（B3 变体）

### 真源 = `_meta.yaml`

```yaml
conventions:
  - id: 01-architecture
    title: 架构设计规范
    file: conventions/01-architecture_架构设计规范.md
    grade:
      red_line: 4        # 红线数
      warning: 0
      recommend: 0
      l1_check_path:
        - importlinter.ini
    l1_check: importlinter
    l1_check_doc: https://importlinter.readthedocs.io/
    l3_route: 任务类型=架构设计 → 必读本篇
```

每篇规范的元数据全部聚合在此。**改这一个文件 → 钩子/CI/仪表盘全部自动同步**。

### 渲染产物（自动生成，禁止手改）

| 产物 | 路径 | 由谁生成 |
|------|------|---------|
| pre-commit 钩子配置 | `.pre-commit-config.yaml` | `scripts/render_meta.py` |
| 仪表盘数据 | `dashboard.html` 数据段 | `scripts/render_meta.py` |
| 每篇规范顶部"分级标签"小节 | 各 `NN-*.md` 头部 | CI 自动渲染 |

> **变更纪律**：改 `_meta.yaml` → 必须同步 `docs/templates/devguard/conventions/_meta.yaml`（模板版本）→ commit message 加 `[meta]` 标记。

### 为什么选 B3（而不是 A 全集中 / C 全分散）

| 方案 | 优点 | 缺点 |
|------|------|------|
| A 全集中（单 yaml + 钩子全部生成） | 单源，无漂移 | 规范正文文体被压缩成结构化 yaml，可读性差 |
| **B3 真源 + 渲染（当前选择）** | 元数据集中 + 正文自由 | 需写渲染脚本 |
| C 全分散（每篇 frontmatter） | 文件自包含 | 难以全局聚合，仪表盘要遍历所有文件 |

## 六、双入口设计

### CLAUDE-规范导航.md（AI 入口，61 行）

- 任务类型 → 规范定位映射表
- 红线总表（17 篇汇总）
- 去重映射（避免同一主题在多篇出现）

### README-规范导航.md（人入口，50 行）

- 快速索引（17 篇一句话简介）
- 红线速查（按严重度排序）

### 为什么双入口

| 对象 | 阅读模式 | 需要什么 |
|------|---------|---------|
| AI | 按任务找规范 | 路由表 + 完整红线 + 去重信号 |
| 人 | 查具体红线 | 一句话索引 + 速查表 |

强制分离避免"一份文档两用"的妥协。

### 为什么 v2.0 引入双轨制

| 任务类型 | 验收方式 | 流程差异 |
|---------|---------|---------|
| 研究类（调研/原型/POC） | 输出报告即可 | 跳过 TDD / E2E |
| 工程类（功能交付/重构） | 真机 E2E + 完整链路 | 严格走全流程 |

避免对研究类任务上"工程级"流程负担。

## 八、与红线检测的关系

```
_meta.yaml grade.l1_check_path  → CI 检测这些文件存在
_meta.yaml l1_check             → CI 跑这些工具
_meta.yaml grade.red_line       → dashboard 显示这个数字
_meta.yaml l3_route             → CLAUDE-规范导航.md 路由表的数据源
```

**完整闭环**：写规范 → 加红线 → 写 L1 检测脚本 → 在 `_meta.yaml` 登记 → CI 自动跑 → dashboard 显示数字。

## 九、边界与降级

| 场景 | 降级行为 |
|------|---------|
| `_meta.yaml` 改了但没跑 `render_meta.py` | pre-commit 钩子 `meta-render` 会自动跑（如果配置了）；否则 CI 检测漂移 |
| 某规范的 L1 检测工具不可用 | dashboard 标红 + worklog 记录技术债 |
| AI 流程新增第 11 篇 | 必须先入 `_meta.yaml`（id=07-ai-workflow 下的 sub_files）+ 更新 CLAUDE-规范导航的 l3_route |
| 治理规范文件缺失 | CI 阶段 `compliance` 报错阻断 PR |

### 明确不做

- **不做**：规范正文用 yaml 结构化（牺牲可读性）
- **不做**：渲染产物入 git（会被 CI 覆盖产生噪音）
- **不做**：手动维护"分级标签"小节（必须由 `render_meta.py` 生成）

## 设计.md 章节导航

### §一 模块定位
- **看什么**：强制约束防线；不是规范定义地
- **想什么**：为什么 M8 分散在根目录？—— 钩子/CI/治理文件必须"在使用现场"才生效，单独建 `infrastructure/` 反而难找

### §二 目录结构
- **看什么**：CI + 钩子 + L4 + 治理文件 + 模板版本同步源
- **易踩坑**：`docs/templates/devguard/` 是模板版本同步源——改根目录配置必须同步模板版本

### §三 CI 5 阶段详解
- **看什么**：每阶段的"做什么 / 阻断什么 / 依赖什么"
- **核心理解**：**build 依赖前 4 阶段**——前 4 阶段任意失败都不会触发 build，节省 CI 时间
- **易踩坑**：V0.1 跳过 ruff-format check 的原因是"跨平台 newline 差异"，不是"忘了配"

### §四 10 钩子详解
- **看什么**：pre-commit 9 + commit-msg 1 (+ commitlint) 的双层设计
- **核心理解**：**pre-commit 处理文件内容，commit-msg 处理 commit message**——不能放错
- **易踩坑**：gitleaks 必须在 pre-commit（要扫文件内容），不能放 commit-msg

### §五 L4 测试 65 条
- **看什么**：10 个测试文件 + AST 静态分析策略
- **易踩坑**：L4 用"配置存在性 + 内容匹配"验证，不验证"真拦下所有违规"——避免脆弱依赖

### §六 治理文件作用矩阵
- **看什么**：6 治理文件 × 3 防线（CI / pre-commit / L4）的覆盖关系
- **核心理解**：**CODEOWNERS 只被 GitHub Web UI 使用**（PR 自动 assign），不在 CI / 钩子 / L4 中跑

### §七 钩子和 CI 覆盖关系
- **看什么**：17 规范红线如何在三条防线（pre-commit / CI / L4）分配
- **核心理解**：**冗余设计是有意的**——gitleaks / markdownlint / importlinter 在 pre-commit 和 CI 都跑

### §九 dogfood 关系
- **看什么**：V2.0.1 在 devguard 自身拦截 8 次的明细
- **核心理解**：dogfood 是"规范不只是文档"的最强证据

## BDD 验收清单

实施时如要新增/修改本模块，必须满足：

- [ ] **BDD 1（5 阶段 CI）**：`.github/workflows/ci.yml` job 数 = 5（lint / test / l4-conventions / compliance / build）
- [ ] **BDD 2（10 钩子）**：`.pre-commit-config.yaml` hooks 项数 = 10（含 commitlint）
- [ ] **BDD 3（65 L4 tests）**：`pytest tests/conventions/` 65 passed
- [ ] **BDD 4（6 治理文件）**：CODEOWNERS / commitlint / markdownlint / gitleaks / spectral / importlinter 全部存在
- [ ] **BDD 5（真源同步）**：`render_meta.py --check` 退出码 0
- [ ] **BDD 6（dashboard 无漂移）**：`git diff --exit-code dashboard.html` 无输出
- [ ] **BDD 7（依赖图）**：build needs: [lint, test, l4-conventions, compliance]
- [ ] **BDD 8（双层钩子）**：pre-commit 阶段有 9 钩子，commit-msg 阶段有 1 钩子（+ commitlint）
- [ ] **BDD 9（dogfood 实证）**：worklog 检索"拦下" ≥ 1 次（V2.0.1 已 8 次）
- [ ] **BDD 10（治理文件矩阵）**：每个治理文件至少被 1 防线使用（CODEOWNERS 除外）
- [ ] **BDD 11（allowlist）**：`.gitleaks.toml` 包含教学反例文件路径
- [ ] **BDD 12（CI 阻断红线）**：8 种阻断方式全部就位（ruff / gitleaks / markdownlint / commitlint / worklog-ref / spectral / importlinter / dashboard 漂移）

## 思考题

1. **为什么不用 husky 替代 pre-commit？** —— 因为 devguard 是"零 Node 依赖"项目（用 pymarkdownlnt 替代 markdownlint-cli 也是这个原因），引入 husky 会强制引入 Node 工具链。
2. **CI 5 阶段可以合并吗？** —— 不能：lint 失败不需要跑 test 浪费时间；compliance 是治理类独立失败信号；build 依赖前 4 阶段是兜底设计。
3. **L4 测试为什么用 AST 静态分析而非实际 import？** —— 因为 src/architecture/ 等目录是教学示例，实际 import 会触发 sys.path 配置 + 真实环境依赖，AST 静态分析更稳定。
4. **dogfood 8 次拦截的分布说明了什么？** —— ruff 3 次（代码风格） + markdownlint 6 次（文档格式） + worklog-ref 1 次（文件名） + 自研脚本 1 次（性能），说明 devguard 自身最容易踩的是"代码风格"和"文档格式"两类规范。
5. **如果第 11 阶段（compliance）扩展到 10+ 文件怎么办？** —— 仍维持单阶段，因为治理文件都是"存在性 + 格式"检查，复杂度 O(N) 线性。
6. **为什么不把 gitleaks 升级为 secretlint？** —— secretlint 有更多规则（覆盖 npm / GitHub token 等），但 gitleaks 8.18.0 仍稳定支持所有 AWS/GCP/Azure 密钥，且本地资源占用更低。

## 易踩坑速查

| 症状 | 真因 | 修复 |
|------|------|------|
| 改了 `.pre-commit-config.yaml` 被 CI 报错 | 那是渲染产物 | 改 `conventions/_meta.yaml` + 跑 `render_meta.py` |
| 改了 `_meta.yaml` 但钩子没变 | 没跑 `render_meta.py` | 跑 `render_meta.py --render` |
| L4 测试 fail | 规范文件结构变了 | 看具体 fail 用例 → 修复规范文件 |
| gitleaks 误报 `security_demo.py` | allowlist 没加 | `.gitleaks.toml` 加白名单 |
| commitlint 报 config warning 噪音 | 已知技术债 | 暂时忽略（技术债 #2）|
| worklog-ref 钩子拒绝合法 commit | commit message 格式不符 `YYYY-MM-DD_*.md` | 调整 commit message 格式 |
| dashboard.html 漂移 | 改了真源但没渲染 | 跑 `render_meta.py --render convention-grade` |
| CI build 阶段 fail 但前 4 阶段全绿 | dashboard 渲染后未 commit | `git add dashboard.html` |
| lint_markdown.py 报"参数太多" | Windows shell=True 长命令行 | 已修复（V2.0.1.2）：分批 50/批 |
| 钩子本地工作但 CI 失败 | pre-commit 未在 CI 镜像中安装 | CI 镜像需 `pip install pre-commit && pre-commit install` |
| V0.1 ruff-format check 在 CI 误报 | 跨平台 newline 差异 | 暂跳过（V2 修 newline 处理后启用）|
| CODEOWNERS assign 失败 | 文件格式错（pattern 与 owner 间必须 tab 分隔）| 用 `cat -A` 检查 whitespace |

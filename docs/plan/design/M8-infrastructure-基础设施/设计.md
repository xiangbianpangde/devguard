## 一、模块定位

### 它是什么

`M8 infrastructure` 是 devguard 的**强制约束防线**。把 M1 `conventions/` 定义的 34 条红线以**可执行形态**落到 3 个地方：

- **本地拦截**（pre-commit 10 钩子）：commit 时立即阻断
- **服务端拦截**（CI 5 阶段流水线）：push/PR 时阻断
- **规范自检**（L4 65 tests）：用 Python 代码验证规范本身写对了

### 它不是什么

- **不是规范定义地** —— 规范在 `conventions/`（M1）；本模块只是规范的可执行化
- **不是单一文件** —— 它是 5 + 10 + 65 + 6 = 86 个组件的协同集合
- **不是可选开关** —— 所有钩子默认开启，禁用要走 ADR

## 三、CI 5 阶段详解

### 阶段 1：lint（L1 自动检查）

```yaml
lint:
  name: L1 lint (ruff + gitleaks)
  steps:
    - ruff check . --config src/coding/ruff.toml
    - gitleaks detect --source . --config .gitleaks.toml
    - python scripts/lint_markdown.py
```

**做什么**：
- **ruff check**：跑 02 编码规范的 L1 检测（禁 print / F401 未用 import / 命名风格）
- **gitleaks detect**：跑 02 编码规范 + 14 SECURITY 的密钥扫描
- **lint_markdown.py**：跑 06 文档规范的 markdownlint 检查

**阻断什么**：代码风格违规、密钥泄漏、文档格式问题

**依赖**：独立（无 `needs`），最先跑

**为什么 V0.1 跳过 ruff-format check**：跨平台 newline 差异会误报，V2 修 newline 处理后启用

---

### 阶段 2：test（pytest 收集 + 真源校验）

```yaml
test:
  name: pytest + cov gate
  steps:
    - render_meta.py --render convention-grade
    - render_meta.py --check
    - pytest -q --co
```

**做什么**：
- **render_meta.py --render convention-grade**：渲染工作区最新（防止 commit 没渲染）
- **render_meta.py --check**：验证真源与产物无漂移
- **pytest -q --co**：只收集测试，不实际跑（V0.1 兼容期；后期会接 `--cov-fail-under`）

**阻断什么**：`_meta.yaml` 改了但没渲染、pytest 不可收集

**依赖**：独立

---

### 阶段 3：l4-conventions（L4 规范测试）

```yaml
l4-conventions:
  name: L4 convention tests
  needs: test
  steps:
    - pytest tests/conventions/ -v
    - check_ai_workflow.py（V6.1 07 章节级 L1）
    - check_code_understanding.py（V6.1 08 章节级 L1）
```

**做什么**：
- **pytest tests/conventions/ -v**：跑 65 条 L4 测试（验证 17 规范的正确性）
- **check_ai_workflow.py**：V6.1 引入，验证 07 章节级 L1
- **check_code_understanding.py**：V6.1 引入，验证 08 章节级 L1

**阻断什么**：规范文件结构错误、章节缺失、L1 检测工具名漂移

**依赖**：`needs: test`（依赖真源校验先过）

---

### 阶段 4：compliance（合规扫描）

```yaml
compliance:
  name: Compliance scan
  needs: test
  steps:
    - python scripts/check_compliance.py
```

**做什么**：跑 `check_compliance.py`，验证治理文件（CODEOWNERS / LICENSE / CHANGELOG / SECURITY / SUPPORT / CONTRIBUTING）存在性 + 格式

**阻断什么**：治理类规范（11-17）文件缺失或格式错误

**依赖**：`needs: test`

---

### 阶段 5：build（构建验证）

```yaml
build:
  name: Build verification
  needs: [lint, test, l4-conventions, compliance]
  steps:
    - python -m compileall src/ -q
    - python docs/templates/devguard/html-report-template/render.py --meta conventions/_meta.yaml --status STATUS.md --out dashboard.html
    - git diff --exit-code dashboard.html
```

**做什么**：
- **compileall src/**：验证 Python 代码可编译
- **render.py**：从 `_meta.yaml` + `STATUS.md` 渲染 `dashboard.html`，自动注入 L4 数字
- **git diff --exit-code dashboard.html**：验证 dashboard 无漂移（必须等于 render 产物）

**阻断什么**：语法错、dashboard 与真源漂移

**依赖**：`needs: [lint, test, l4-conventions, compliance]`（前 4 阶段全绿才能 build）

## 四、10 钩子详解

### 4.1 pre-commit 阶段（9 钩子）

| # | 钩子 ID | 仓库 | rev | 作用 | 阻断示例 |
|---|---------|------|-----|------|---------|
| 1 | `trailing-whitespace` | pre-commit-hooks | v4.6.0 | 删除行尾空格 | 自动修复 |
| 2 | `end-of-file-fixer` | pre-commit-hooks | v4.6.0 | 文件末尾补换行 | 自动修复 |
| 3 | `check-yaml` | pre-commit-hooks | v4.6.0 | YAML 语法校验 | `yaml.safe_load` 失败即阻断 |
| 4 | `check-json` | pre-commit-hooks | v4.6.0 | JSON 语法校验 | `json.load` 失败即阻断 |
| 5 | `check-added-large-files` | pre-commit-hooks | v4.6.0 | 检查 >10MB 文件 | `args: ['--maxkb=10000']` 阻断 |
| 6 | `ruff` | ruff-pre-commit | v0.6.0 | 静态检查 | F401 / T201 / N806 等 |
| 7 | `ruff-format` | ruff-pre-commit | v0.6.0 | 自动格式化 | 长行自动拆行 |
| 8 | `gitleaks` | gitleaks | v8.18.0 | 密钥扫描 | 检测到 AWS key / API token 即阻断 |
| 9 | `markdownlint`（local） | local | -- | 文档格式检查 | MD024 重复标题 / MD013 行长等 |

### 4.2 commit-msg 阶段（1 钩子 + commitlint）

| # | 钩子 ID | 仓库 | rev | 作用 | 阻断示例 |
|---|---------|------|-----|------|---------|
| 10 | `commit-msg-worklog-ref`（local） | local | -- | worklog 引用校验 | commit message 必须引用 `worklogs/YYYY-MM-DD_*.md` |
| -- | `commitlint` | commitlint-pre-commit-hook | v9.16.0 | Conventional Commits 校验 | type 不在 enum / subject < 5 字符 / upper-case 等 |

> **为什么 commitlint 也算"10 钩子"内**：任务定义 10 钩子包含 commitlint，但 `stages: [commit-msg]` 表明它属于 commit-msg 阶段。

### 4.3 双层设计原因

| 阶段 | 适用 | 原因 |
|------|------|------|
| **pre-commit** | 文件内容相关 | 此时文件已写入工作区，可读可改可 lint |
| **commit-msg** | commit message 相关 | 此时 commit 还未生成，message 可校验可阻断 |

> **不能放错**：`gitleaks` 必须 pre-commit（要扫文件内容）；`commitlint` 必须 commit-msg（要扫 message）。

- **正则匹配红线**（如 `FORBIDDEN_FRAMEWORKS = frozenset({"fastapi", "sqlalchemy", ...})`）

## 七、钩子和 CI 的覆盖关系

```
                  17 规范红线（共 34 条）
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   pre-commit 拦截    CI 阶段拦截        L4 测试拦截
   （fast-fail）      （服务端兜底）      （自证正确）
   ─────────────      ──────────────     ──────────
   ruff (6 红线)      importlinter (4)   test_01-08
   gitleaks (1)       gitleaks (1)       test_meta_l1
   markdownlint (5)   spectral (4)
   commitlint (3)     markdownlint (5)
   worklog-ref (1)    check_compliance (9 治理类)
   trim/EOF/yaml/...  dashboard 漂移 (1)
   ─────────────      ──────────────     ──────────
   共 17+ 条          共 24+ 条          共 65 条
```

> **冗余设计**：gitleaks / markdownlint / importlinter 在 pre-commit 和 CI 都跑——pre-commit 拦截失败的成本最低，CI 是服务端兜底。

## 八、边界与降级

| 场景 | 降级行为 |
|------|---------|
| Python 脚本在 Windows shell=True 报错 | V2.0.1.2 修复：分批调用 + N806 命名风格（参考 `worklogs/2026-06-08_v20-devguard-dogfood.md`）|
| `render_meta.py --check` 失败 | pre-commit 钩子阻断（如果配置）；CI 阶段 `test` 阻断 |
| markdownlint 误报 MD013（行长）| `.markdownlint.json` 关闭 `MD013: false` |
| gitleaks 误报教学反例文件 | `.gitleaks.toml` allowlist `src/coding/security_demo.py` |
| commitlint 报 `config warning` 噪音 | 不阻断但污染输出（技术债 #2）|
| L4 测试 fail | CI `l4-conventions` 阶段阻断 PR |
| 钩子不工作（pre-commit 未安装）| 不阻断；CI 兜底 |

### 明确不做

- **不做**：用 husky 替代 pre-commit（避免引入 Node 工具链）
- **不做**：把钩子放服务端 hook（pre-commit 必须本地）
- **不做**：跳过 L4 测试的开关（必须跑）

## 九、与 devguard dogfood 的关系（V2.0.1 实证）

V1.5-V2.0.1 期间，devguard 用自己的规范约束自己，被钩子拦截 **8 次**（`worklogs/2026-06-08_v20-devguard-dogfood.md`）：

| 次数 | 钩子 | 拦截内容 | 修复 |
|:---:|------|---------|------|
| 1 | ruff | `import re` F401 未用 | 删除 import |
| 2-3 | ruff | `print()` T201 × 2 | 改为 logger |
| 4 | markdownlint | STATUS.md MD024 重复标题 | 合并章节 |
| 5 | commit-msg-worklog-ref | `v1.5` 文件名不匹配 `YYYY-MM-DD_` 正则 | 改名 |
| 6-7 | markdownlint | MD024 × 5 标题 | 删除重复节 |
| 8 | lint_markdown.py | Windows shell=True 长命令行 bug | 分批 50/批 |

> **核心意义**：dogfood 是 V1.x 强制范式"100% 闭环"的**最强证据**——devguard 自己的代码也得遵守自己定义的规范。

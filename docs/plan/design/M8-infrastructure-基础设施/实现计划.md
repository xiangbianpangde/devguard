## 二、阶段 V0.1 收束：9 钩子 + 5 阶段 CI（2026-05-27）

### 目标

把"5 阶段 CI + pre-commit 钩子"作为 devguard 强制范式的最小可行集。

### 实现路径

```
conventions/01-08 规范确定
    │
    ▼
设计 5 阶段 CI（lint / test / l4 / compliance / build）
    │
    ▼
设计 9 钩子（pre-commit-hooks 5 + ruff 2 + gitleaks 1 + markdownlint 1 local + worklog-ref 1 local）
    │
    ▼
写 .github/workflows/ci.yml + .pre-commit-config.yaml
    │
    ▼
tests/conventions/ 13 条 L4 测试
    │
    ▼
收束：v0.1 人审计签核（首个签核的收束节点）
```

### 改动

| 模块 | 改什么 | 状态 |
|------|--------|:---:|
| `.github/workflows/ci.yml` | 新建（173 行），5 阶段流水线 | ✅ |
| `.pre-commit-config.yaml` | 新建（54 行），9 钩子 | ✅ |
| `tests/conventions/` | 新建，13 条 L4 测试 | ✅ |
| `scripts/check_compliance.py` | 新建，CI compliance 阶段用 | ✅ |
| `scripts/check_worklog_ref.py` | 新建，commit-msg 钩子用 | ✅ |
| `scripts/lint_markdown.py` | 新建，markdownlint 钩子用 | ✅ |
| `worklogs/decisions/` | 收束报告 v0.1 + 人审计签核 | ✅ 已签核 |

### 关键决策

- **5 阶段而非 3 阶段**：lint / test / l4-conventions / compliance / build 各管一摊，失败信息更精准
- **9 钩子而非单脚本**：拆成独立钩子便于"禁用某条不改其他"
- **L4 用 Python 而非 BDD 框架**：避免引入 behave / pytest-bdd 等新依赖

## 七、阶段 V0.7：L4 retry 成功（2026-06-07）

### 目标

解决 L4 数字注入 dashboard 时的脆弱性（V6.3 失败 → V7.2 修复）。

### 实现路径

```
dashboard.html 渲染时 L4 数字
    │
    ▼
V6.3：硬编码（FAIL — 数字与实际不符）
    │
    ▼
V7.2：用 collect_l4_stats.py 自动收集 + bash ${VAR:-0} 防御
    │
    ▼
CI build 阶段 run collect_l4_stats.py → 注入 render.py 参数
```

### 改动

| 文件 | 改什么 |
|------|--------|
| `scripts/collect_l4_stats.py` | 新建（V7.2 修复） |
| `.github/workflows/ci.yml` | build 阶段改用 collect_l4_stats.py |
| `dashboard.html` | 自动注入 L4_PASSED / L4_TOTAL |

### 关键技术

```bash
L4_STATS=$(python scripts/collect_l4_stats.py 2>/dev/null || echo "L4_STATS=0/0")
L4_PASSED="${L4_PASSED:-0}"  # bash 防御空字符串
L4_TOTAL="${L4_TOTAL:-0}"
```

## 十、阶段 V2.0.1：devguard 自身 dogfood（2026-06-08）

### 目标

用 V1.x 强制范式约束 devguard 自己的代码库，证明闭环生效。

### 实现路径

```
V1.5 收尾后，devguard 自己的代码从未被 V1.x 钩子跑过
    │
    ▼
V1.5.1-V1.5.4 提交期间，故意触发各种违规
    │
    ▼
观察钩子行为：拦下 8 次 → 全部修复
    │
    ▼
写 worklog 2026-06-08_v20-devguard-dogfood.md 记录
```

### 8 次拦截明细

| 次数 | 钩子 | 拦截内容 | 修复方式 |
|:---:|------|---------|---------|
| 1 | ruff | `import re` F401 | 删除 import |
| 2-3 | ruff | `print()` T201 × 2 | 改为 logger |
| 4 | markdownlint | STATUS.md MD024 重复标题 | 合并章节 |
| 5 | commit-msg-worklog-ref | `v1.5` 文件名不匹配 `YYYY-MM-DD_` 正则 | 改名 |
| 6-7 | markdownlint | MD024 × 5 标题 | 删除重复节 |
| 8 | lint_markdown.py | Windows shell=True 长命令行 bug | 分批 50/批 + N806 命名 |

### 改动

| 文件 | 改什么 |
|------|--------|
| `scripts/lint_markdown.py` | V2.0.1.2 修复：分批调用 |
| `worklogs/2026-06-08_v20-devguard-dogfood.md` | 8 次拦截记录 |
| 多个文件 | 移除 F401 / T201 / MD024 违规 |

### 核心意义

> "dogfood 是 V1.x 闭环最强证据"（参考 `worklogs/2026-06-08_v20-devguard-dogfood.md` D-V2.0.1）
> - 比写"测试套件验证规范"更直接——devguard 自己就是 17 规范的应用对象
> - 比"外部项目 fork"更省事——不需要新建项目
> - 比"单元测试"更可信——commit 真能拦住违规代码

### worklog 引用

`worklogs/2026-06-08_v20-devguard-dogfood.md`（完整记录 8 次拦截 + 关键决策）

## 设计.md 章节导航

### §一 模块定位
- **看什么**：它是规范落地态；不是产品代码结构
- **核心理解**：反直觉强调 —— `src/` 整体"按维度分目录"是反模式，仅作教学用；而 `src/architecture/` 内部演示的是真实可用的分层架构
- **易踩坑**：把 src/ 当成"产品模板"直接 fork 投产

### §二 目录结构
- **看什么**：7 个子目录每个对应 1 篇规范 §二 的可运行版本
- **易踩坑**：07 AI 协作流程没有对应子目录（流程是文档不是代码）
- **想什么**：为什么 `architecture/` 内部是分层（产品结构）而 `src/` 整体按维度分（教学结构）？—— **范围不同**：单个子目录演示"分层是产品代码结构"，整个 `src/` 演示"按规范维度分目录是教学组织方式"

### §三 按规范维度分目录
- **看什么**：A/B/C 三方案对比，A（按规范维度）是当前选择
- **核心理解**：选 A 的根本原因是"与 conventions/ 1:1 对应，查阅零成本" —— 这是"教学示例"与"产品代码"的根本差异

### §四 7 个子目录详细介绍
- **看什么**：每个子目录的文件清单 + 关键演示 + 核心契约
- **重点关注**：
  - `architecture/`：4 层 + DI + importlinter 2 契约
  - `coding/`：命名正反例 + 教学反模式豁免（`naming_bad.py` 故意违规）
  - `api/`：统一响应 `{code, message, data}` + 错误码体系
  - `code-understanding/`：AST 教学演示双图谱底层

### §五 落地配置策略
- **看什么**：src/ 内每份配置文件都能在规范 §二 找到原文
- **核心理解**：**1:1 对齐是纪律** —— 改规范要同步改 src/，改 src/ 要在 worklog 注明依据

### §六 与 `_meta.yaml` 引用关系
- **看什么**：3 处 `l1_check_path` 指向 src/（`src/coding/ruff.toml` / `src/architecture/importlinter.ini` / `src/api/.spectral.yaml`）
- **易踩坑**：根目录和 src/ 内有"双份配置"（演示用 vs 项目级真用），同步维护

### §七 互检关系
- **看什么**：`conventions/01-08 ↔ src/<维度>/ ↔ docs/specs/01-08 + 08-examples.md`
- **核心理解**：三向互检，src/ 是中间桥梁

### §八~§九 边界 + 验收
- **看什么**：5 类降级场景 + 8 项验收标准

## 思考题

1. **为什么 src/ 整体按维度分目录（反模式），但 architecture/ 内部按分层（产品结构）？** —— 范围不同：src/ 整体是"教学组织方式"（与规范 1:1 对应），单个子目录内部是"演示该规范的最佳实践"。两者并不矛盾。
2. **`naming_bad.py` 故意违规，会不会让 ruff 在 CI 误报？** —— 不会，`ruff.toml` 加 per-file-ignores `"src/coding/naming_bad.py" = ["N", "E", "F"]` 显式豁免。L4 测试反过来要断言"违规必须存在"（教学反模式 ≠ 真实违规）。
3. **code-understanding 教学用纯 Python AST，为什么不用真实 CodeGraph？** —— CodeGraph 是 C++ 写的二进制工具，集成到示例项目会让 src/ 依赖外部安装。教学用 Python AST 实现底层原理，让人理解"图谱是怎么来的"；真实项目 README 指向真工具。
4. **改规范 §二 必同步 src/ 配置文件，但 src/ 内有教学反模式怎么办？** —— 教学反模式（如 `naming_bad.py`）由 per-file-ignores 豁免，不在同步范围。同步时只看"真用配置"（如 `ruff.toml` 的规则集），不看"豁免项"。
5. **`call_graph_example.py` 顶部的 `# ruff: noqa` 列表由谁维护？** —— L4 测试断言它包含的豁免项（N802/F401/F541/N803/N806/E731/E741），改这个列表要同步 L4 测试。
6. **新增第 7+ 个子目录（如 `security/`）的最小工作集是什么？** —— 6 件：conventions/ 加 02 §一/§二子篇 + src/security/ 新建 + docs/specs/ 加 BDD + _meta.yaml 加 l1_check_path + worklog + docs/templates/devguard/src/security/ 同步。

## 易踩坑速查

| 症状 | 真因 | 修复 |
|------|------|------|
| 改规范 §二 后 src/ 配置未同步 | 忘了"1:1 对齐"纪律 | 写 worklog 时显式列出"src/<维度>/ 同步清单" |
| ruff 报 `naming_bad.py` 大量违规 | 忘了 per-file-ignores | 确认 `ruff.toml` 含 `src/coding/naming_bad.py = ["N", "E", "F"]` |
| `python -m src.architecture.main` 报循环依赖 | importlinter 契约失败 | 检查 4 层文件是否越层引用 |
| `pytest src/testing/` 报覆盖率不足 | 没装 pytest-cov | `pip install pytest-cov` 后重跑 |
| `uvicorn main:app` 启动失败 | src/api/ 内没装 fastapi | `pip install fastapi uvicorn` |
| `spectral lint` 报路径含动词 | URL 设计违规 | 路径只用名词 + HTTP 方法表达动作 |
| `call_graph_example.py` 顶部的 noqa 列表被改 | L4 测试断言会断 | 改前先看 L4 测试期望哪些豁免项 |
| `__pycache__` / `.ruff_cache` 提交了 | .gitignore 未生效 | 确认 `__pycache__/` `.ruff_cache/` `.pytest_cache/` 都在 .gitignore |
| `src/README-示例代码总览.md` 找不到 | 工作目录不对 | 在仓库根目录下访问 |
| 改 `src/<维度>/` 后未加 worklog | 违反"改完即汇报"原则 | 改完立刻写 worklog 注明依据 |

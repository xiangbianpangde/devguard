阶段 1 关键决策 · "独立创作" 的代价

阶段 1 起步时，各子目录是"独立创作"的——目录结构、文件命名、配置语法都按"能跑就行"原则。优点是开发快，缺点是：

  • 与规范的 §二「落地」段没有强制 1:1 对应
  • 改规范 §二 时不知道 src/ 内哪些文件要同步
  • 新人学完规范，照着 src/ 抄容易抄出与规范不严格对齐的版本

这 3 个问题在阶段 3 重构时统一解决：强制要求 src/ 内每个配置文件都要在规范 §二 找到原文。

关键反直觉保留 · naming_bad.py

问题

教学反模式（naming_bad.py 故意命名违规）会被 ruff 自动修复，导致 L4 规范测试无法断言"违规存在"。

# naming_bad.py（故意保留违规） def process(d): # N803 命名违规 x = 42 # 魔法数字 return d + x

解决方案

加 per-file-ignores 豁免——教学反模式 ≠ 真实违规。L1 检测会跳过该文件；L4 套件断言"豁免范围被合理使用"。

# ruff.toml [lint.per-file-ignores] "src/coding/naming_bad.py" = ["N", "E", "F"]

关键反直觉 · 教学反模式 ≠ 真实违规

本阶段最关键的"反直觉保留"是 naming_bad.py 的命名违规：

  • L1 检测 跳过该文件（per-file-ignores 豁免）
  • L4 套件 断言"该文件确实存在违规"（L4 测试断言豁免范围被合理使用）
  • L4 套件 同时断言"该豁免仅用于 src/coding/，不蔓延到其他文件"

这是"教学反模式"与"真实违规"的本质区别：教学反模式有 L4 套件"圈养"，真实违规直接被 L1 检测挡掉。

测试策略的关键决策 · 为什么引入 L4 套件

阶段 1 起步时只有"入口可跑"验证——这种验证很弱，因为：

  • 不能防止"配置文件漏改"（规范 §二 加了规则，但 ruff.toml 没同步）
  • 不能防止"教学豁免被滥用"（naming_bad.py 的豁免蔓延到其他文件）
  • 不能防止"BDD 互检漂移"（specs 加了场景，但 src/ 没实现）

阶段 3 引入 L4 套件（tests/conventions/）后，这些问题被机器自动拦截——这是 V1.x 强制约束范式的核心改造。

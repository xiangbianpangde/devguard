01 · 模块定位

它是什么 · 它不是什么

src/ 是 devguard 的规范落地态——7 个子目录每个对应 1 篇规范（01-06 + 08）的"§二 落地配置"可运行版本。本节先划清边界。

它是什么

7 子目录每个对应 1 篇规范的"§二 落地"段的可运行版本。改 conventions/NN-*.md §二 时，对应 src/<维度>/ 要同步更新——1:1 对齐是硬约束。

规范落地态 可运行 1:1 对齐

它不是产品代码结构

真实项目应按规范 §一 建议组织（业务域或技术层次），本目录是按规范维度分目录（与 conventions/ 一一对应）。

反模式 仅教学用

它不是规范测试套件

规范测试在 tests/conventions/（L4 测试），src/ 是给人/AI 阅读和复制的"参考实现"。

L4 在 tests/ 参考实现

它不是产品服务的源码

本目录的所有代码都是教学/演示用，禁作为模板项目直接 fork 后投产——会缺失业务领域建模、错误处理、监控埋点等生产化设计。

禁投产 禁直接 fork

关键反直觉（重申）

src/architecture/ 内部演示的是真实可用的分层架构（表现→应用→领域→基础设施），而 src/ 整体的"按维度分目录"是反模式——仅作教学用。

内部真实 顶层反模式

07 缺号说明

07 AI 协作流程不需要示例代码——流程本身是文档，第零步调研和第三步收束等都是 markdown 流程说明，不是可运行代码。

07 = 文档流程 无源码
!
三句边界总结：① src/ 是规范的可读 + 可复制版本；② src/ 不是产品代码结构，真实项目按业务域或技术层次分；③ src/ 不是规范测试，L4 在 tests/conventions/。读示例时先确认这三条。

03 · 核心设计原则

为什么按"规范维度"分目录（不按产品分层）

  • 但 7 个子目录要互相引用
  • 破坏"规范示例"独立性

  • 与 conventions/ 1:1 对应
  • 查阅零成本
  • README 顶部明确"非推荐产品结构"

反直觉强调 · 真实项目应该怎么分

业务复杂 → 按业务域分

src/user/ · src/order/ · src/payment/ ··· 每域内再做分层（controller / service / repo）。

DDD 风格 业务边界清晰

项目简单 → 按技术层次分

src/controllers/ · src/services/ · src/repositories/ ··· 简单项目用经典三层即可。

MVC 风格 快速上手
!

01 architecture · 四层架构 + DI

关键文件清单

  • domain/order.py 实体
    dataclass + OrderStatus enum；纯标准库
  • domain/order_repository.py 接口
    ABC + @abstractmethod；只定义接口不实现
  • domain/discount_calculator.py 领域服务
    纯业务逻辑（VIP/满减），零外部依赖
  • application/order_service.py 业务编排
    注入 repo + calculator；只编排不写规则
  • infrastructure/postgres_order_repository.py 数据库实现
    实现领域接口；dataclass 注入 db_url
  • presentation/order_controller.py HTTP 控制器
    极薄：参数校验 → 调 service → 统一响应
  • main.py 入口
    依赖注入（在入口处连接所有具体实现）
  • importlinter.ini L1 检测
    2 个契约：分层单向 + 领域层禁框架

核心契约（机器可拦截）

  • ✓ 表现层 → 应用层 → 领域层：单向依赖
  • ✓ 领域层 forbidden_modules = fastapi, sqlalchemy, django
  • ✓ 跨层调用（表现→基础设施）禁止

02 coding · 命名/错误/安全

关键文件清单

  • naming_good.py 正例
    见名知意 + 魔法数字常量化 + 布尔 is_/has_/can_ 前缀 + PascalCase/snake_case 统一
  • naming_bad.py 反模式
    模糊命名（process(d)） + 缩写不可读 + 魔法数字 + 风格不统一
  • error_handling_demo.py 错误处理
    自定义异常类（UserNotFoundError / DatabaseError） + 日志含上下文 + raise from e
  • security_demo.py 安全规范
    环境变量读密钥（os.environ.get） + SQL 参数化（? 占位） + 输入白名单 + 日志脱敏
  • ruff.toml L1 检测
    select = ["E", "F", "N", "T20", "T100", "S", "PLR0915"]；含 per-file-ignores 配置

关键反直觉 · naming_bad.py

故意保留命名违规（用于 L4 规范测试断言），ruff.toml 加 per-file-ignores 豁免——教学反模式 ≠ 真实违规。

# ruff.toml 豁免 [lint.per-file-ignores] "src/coding/naming_bad.py" = ["N", "E", "F"]
禁 print SQL 参数化 密钥走环境变量 禁静默吞异常 日志脱敏

真用配置随项目演化（CI 触发）；教学配置随规范版本升级（V 大版本时整体重写）。

4 条"明确不做"

不做 ① · 按业务域分目录

破坏与 conventions/ 的 1:1 对应。规范演示的核心价值是"查阅零成本"，按业务域分会切碎这种价值。

不做 ② · 把 src/ 当产品代码投产

这是教学/参考，会缺失业务领域建模、错误处理、监控埋点等生产化设计。直接投产 = 反模式。

不做 ③ · 跨子目录的共享代码

每个子目录独立自治。共享代码会形成"src/common" 这种跨维度耦合，破坏"规范示例"独立性。

不做 ④ · CI 跑 src/ 内的演示代码

避免误把演示当生产。演示代码的失败不应该阻塞项目 CI（演示代码不在生产路径上）。

!
新增第 7+ 个子目录的门槛极高：要先在 conventions/ 升 02 §一/§二（新增安全子篇）+ 改 _meta.yaml + 同步 BDD 互检。不能"src/ 加个目录就完事"——这是规范落地的硬约束。

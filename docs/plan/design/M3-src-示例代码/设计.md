## 一、模块定位

### 它是什么

`src/` 是 devguard 的**规范落地态** —— 7 个子目录每个对应 1 篇规范（01-06 + 08）的"§二 落地配置"的可运行版本。改 `conventions/NN-*.md` §二 时，配套的 `src/<维度>/` 文件要同步更新。

### 它不是什么

- **不是产品代码结构** —— 真实项目应按规范 §一 建议组织（业务域或技术层次），本目录是**按规范维度**分目录（与 conventions/ 一一对应）
- **不是规范测试套件** —— 规范测试在 `tests/conventions/`（L4 测试），src/ 是给人/AI 阅读和复制的"参考实现"
- **不是产品服务的源码** —— 本目录的所有代码都是教学/演示用，禁作为模板项目直接 fork 后投产

> **关键反直觉**：`src/architecture/` 内部演示的是**真实可用的分层架构**（表现→应用→领域→基础设施），而 `src/` 整体的"按维度分目录"是反模式 —— 仅作教学用。

## 三、核心设计：按规范维度分目录（不按产品分层）

### 原则

| 选项 | 优点 | 缺点 | 是否采用 |
|------|------|------|---------|
| **A. 按规范维度分（当前）** | 与 conventions/ 1:1 对应，查阅零成本 | 不是推荐的产品结构 | ✅ |
| B. 按业务域分 | 真实项目结构 | 7 个子目录要互相引用，破坏"规范示例"独立性 | ❌ |
| C. 按技术层次分（控制器/服务/仓储） | 通用分层 | 与规范维度不对应，查阅要跨目录 | ❌ |

### 反直觉强调

README 顶部明确告知"**这不是推荐的产品代码结构**" —— 真实项目应：
- 业务复杂 → 按业务域分（`src/user/` `src/order/` …）
- 项目简单 → 按技术层次分（`src/controllers/` `src/services/` …）

而 `src/architecture/` 内部演示的才是真实可用的分层结构。

## 四、7 个子目录详细介绍

### 4.1 `architecture/` — 01-架构设计 §二

| 文件 | 类型 | 关键演示 |
|------|------|---------|
| `domain/order.py` | 实体 | `dataclass` + `OrderStatus` enum；纯标准库 |
| `domain/order_repository.py` | 抽象接口 | `ABC` + `@abstractmethod`；只定义接口不实现 |
| `domain/discount_calculator.py` | 领域服务 | 纯业务逻辑（VIP/满减），零外部依赖 |
| `application/order_service.py` | 业务编排 | 注入 `repo` + `calculator`；只编排不写规则 |
| `infrastructure/postgres_order_repository.py` | 数据库实现 | 实现领域接口；`dataclass` 注入 `db_url` |
| `presentation/order_controller.py` | HTTP 控制器 | 极薄：参数校验 → 调 service → 统一响应 |
| `main.py` | 入口 | **依赖注入**（在入口处连接所有具体实现）|
| `importlinter.ini` | L1 检测 | 2 个契约：分层单向 + 领域层禁框架 |

**核心契约**：
- 表现层 → 应用层 → 领域层：单向依赖
- 领域层 `forbidden_modules = fastapi, sqlalchemy, django`
- 跨层调用（表现→基础设施）禁止

### 4.2 `coding/` — 02-代码编写 §二

| 文件 | 类型 | 关键演示 |
|------|------|---------|
| `naming_good.py` | 正例 | 见名知意 + 魔法数字常量化 + 布尔 `is_/has_/can_` 前缀 + PascalCase/snake_case 统一 |
| `naming_bad.py` | 反模式 | 模糊命名（`process(d)`） + 缩写不可读 + 魔法数字 + 风格不统一 |
| `error_handling_demo.py` | 错误处理 | 自定义异常类（`UserNotFoundError` / `DatabaseError`） + 日志含上下文 + `raise from e` |
| `security_demo.py` | 安全规范 | 环境变量读密钥（`os.environ.get`） + SQL 参数化（`?` 占位） + 输入白名单 + 日志脱敏 |
| `ruff.toml` | L1 检测 | select = `["E", "F", "N", "T20", "T100", "S", "PLR0915"]`；含 per-file-ignores 配置 |

### 4.3 `git/` — 03-Git 协作 §二

| 文件 | 类型 | 关键演示 |
|------|------|---------|
| `.gitignore` | 忽略规则 | 依赖目录 + 环境变量（含密钥） + IDE 配置 + 构建产物 + 测试覆盖率 |
| `.pre-commit-config.yaml` | L1 检测 | 4 个钩子：pre-commit-hooks（基础） + ruff + gitleaks + commitlint |
| `commitlint.config.js` | L1 检测 | type-enum 限定 8 种类型 + subject 至少 5 字符 + body 不超 100 字符 |

### 4.4 `api/` — 04-API 设计 §二

| 文件 | 类型 | 关键演示 |
|------|------|---------|
| `main.py` | FastAPI 微服务 | 统一响应 `{code, message, data}` + 错误码体系（`40401`/`40001`/`40901`）+ 分页+排序+搜索 + Pydantic 校验 + 全局异常处理 |
| `.spectral.yaml` | L1 检测 | 3 个规则：路径 kebab-case + 路径禁动词 + 操作必须声明安全方案 |

**亮点**：
- `list_users` 演示分页+排序+搜索三合一查询
- `create_user` 演示 Pydantic 字段约束（`min_length` / `pattern`）
- 全局异常处理器把 `HTTPException` 转换为统一错误响应

### 4.5 `testing/` — 05-测试规范 §二

| 文件 | 类型 | 关键演示 |
|------|------|---------|
| `discount.py` | 被测代码 | 纯业务逻辑（`calculate_discount` + `split_bill`），零外部依赖 |
| `test_discount.py` | 测试套件 | AAA 模式 + 命名 `test_方法_场景_期望` + 4 类场景（Happy/Boundary/Exception） |
| `pytest.ini` | L1 检测 | `--cov-branch --cov-fail-under=80 -v` |

**测试覆盖**：
- Happy Path：普通用户 / VIP / 满 5 件 / VIP+满 5 件（取最优）
- Boundary：极小金额（0.01）
- Exception：金额=0 / 金额<0 / 数量=0 / 人数=0 / 金额<0（分账）

### 4.6 `documentation/` — 06-文档规范 §二

| 文件 | 类型 | 关键演示 |
|------|------|---------|
| `CHANGELOG.md` | 变更日志 | Keep a Changelog 1.0.0 格式；`Added/Changed/Fixed/Security` 四类段 |
| `docstring_example.py` | docstring 注释 | Google 风格：`Args/Returns/Raises/Example` 四段；`@dataclass` 的 `Attributes` 段 |

### 4.7 `code-understanding/` — 08-代码理解与图谱 §二 v3.0

| 文件 | 类型 | 关键演示 |
|------|------|---------|
| `call_graph_example.py` | AST 教学 | 三部分：图谱数据结构（`Entity`/`Relation`/`CodeGraph`）+ `CallGraphVisitor` 遍历器 + `build_graph`/`print_report` 主入口 |

**v3.0 双图谱范式**：
- **AI 用**：CodeGraph（tree-sitter 解析 20+ 语言 + SQLite+FTS5 + MCP 集成）
- **人用**：Understand-Anything（Claude Code 插件）
- **教学用**：`call_graph_example.py`（纯 Python AST 实现，演示底层原理）

## 五、落地配置策略（与规范 §二 1:1 对齐）

| 维度 | 规范 §二 内容 | src/ 落地 | 验证命令 |
|------|--------------|----------|----------|
| 01 架构 | 分层契约 + 领域层纯净 | `src/architecture/importlinter.ini` | `lint-imports --config src/architecture/importlinter.ini` |
| 02 代码 | ruff 规则集 | `src/coding/ruff.toml` | `ruff check src/coding/` |
| 03 Git | pre-commit 钩子 + commitlint | `src/git/.pre-commit-config.yaml` + `commitlint.config.js` | `pre-commit run --all-files` |
| 04 API | OpenAPI 契约 | `src/api/.spectral.yaml` | `spectral lint openapi.json --ruleset src/api/.spectral.yaml` |
| 05 测试 | 覆盖率门禁 | `src/testing/pytest.ini` | `pytest src/testing/` |
| 06 文档 | 模板规约 | `src/documentation/CHANGELOG.md` | `markdownlint src/documentation/CHANGELOG.md` |
| 08 图谱 | 双图谱范式 | `src/code-understanding/call_graph_example.py` | `python call_graph_example.py .` |

> **关键纪律**：配置文件的每一行都要能在规范 §二 找到对应原文 —— 改规范要同步改 src/，改 src/ 要在 worklog 注明依据。

## 六、与 `_meta.yaml` 的引用关系

```yaml
# conventions/_meta.yaml（节选）
- id: 01-architecture
  l1_check_path:
    - importlinter.ini                # 仓库根的 importlinter.ini 与 src/architecture/importlinter.ini 同步
  l1_check: importlinter

- id: 02-coding
  l1_check_path:
    - src/coding/ruff.toml            # ← 指向 src/ 内的具体配置文件
  l1_check: ruff

- id: 04-api
  l1_check_path:
    - .spectral.yaml                  # 仓库根 + src/api/.spectral.yaml 双份（演示用）
  l1_check: spectral
```

> **双份配置**：根目录的 `importlinter.ini` / `ruff.toml` / `.spectral.yaml` 是项目级"真用"配置；`src/<维度>/` 下的是"教学用"配置。两者同步维护，互检。

## 七、与规范互检关系

```
conventions/01-08 (规范)  ←──互检──→  docs/specs/01-08 (BDD 验收)
        │
        │ §二「落地」段
        ▼
src/<维度>/ (可运行示例)    ←──互检──→  docs/specs/08-examples.md
```

**对应 BDD 规范**：`docs/specs/08-examples.md`（场景 1-3）：
- 场景 1：每个子目录有 README-<维度>示例.md
- 场景 2：每个子目录有可运行文件（python/pytest/uvicorn/pre-commit）
- 场景 3：跨语言差异由规范正文（07 §三）说明，src/ 不重复实现

## 八、边界与降级

| 场景 | 降级行为 |
|------|---------|
| 规范 §二 新增配置项 | 同步加到对应 `src/<维度>/` 配置文件中 |
| 规范升 v 大版本（如 02 v3.0） | 重写对应 `src/<维度>/` 的所有演示文件 |
| 教学反模式被 ruff 自动修复 | `naming_bad.py` 必须保留违规（L4 测试复用），加 `per-file-ignores` 豁免 |
| `_meta.yaml.l1_check_path` 加新路径 | 必须在 `src/<维度>/` 真实存在；CI 漂移检测会报错 |
| 新增第 7+ 个子目录（如 `security/`） | 需先在 `conventions/` 升 02 §一/§二（新增安全子篇）+ 改 `_meta.yaml` |

### 明确不做

- **不做**：按业务域分目录（破坏与 conventions/ 的 1:1 对应）
- **不做**：把 src/ 当产品代码投产（这是教学/参考）
- **不做**：跨子目录的共享代码（每个子目录独立自治）
- **不做**：CI 跑 src/ 内的演示代码（避免误把演示当生产）

## 九、验收标准

| 维度 | 验收方式 | 通过标准 |
|------|---------|---------|
| 子目录数 | `ls -d src/*/` | = 7 |

| `_meta.yaml` 引用 | grep `l1_check_path` | ≥ 3 处指向 `src/` |
| BDD 互检 | `docs/specs/08-examples.md` | 3 个场景全过 |

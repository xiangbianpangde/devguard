# 设计提案：流程强制化方案

> 版本: v1.0 | 2026-06-10
> 依据: 2026-06-10 格式一致性审查 + 流程强制化分析
> 关联: `docs/plan/开发清单.md` #36-#46, `conventions/ai-workflow_AI协作开发流程/`

---

## 一、背景

### 1.1 当前问题

项目定义了完整的开发流程（`conventions/ai-workflow_AI协作开发流程/` 8 篇 + `conventions/` 8 篇规范），但**流程步骤的强制执行几乎为零**。具体表现为：

**格式层面**（2026-06-10 审查发现）：
- STATUS.md 格式约束仅靠 `render.py` 硬编码解析，改表头/加表静默失败
- CLAUDE.md 零自动检测，AI 上下文可能过时/错误
- 开发清单.md 零校验，与 STATUS 可能不一致
- `.markdownlint.json` 25 条规则全关，中文 Markdown 项目唯一格式工具形同虚设
- `render.py` 使用 `safe_substitute`，缺变量不报错
- 模板漂移检测仅覆盖 pre-commit-config 和 convention-grade

**流程层面**（本轮分析）：
- "每功能点完成 → 更新 worklog" 只检查 commit message 引用 worklog 文件名，不验证文件存在
- "每功能点完成 → 更新 STATUS" 零检测
- worklog 内容结构零校验（模板定义了"完成了什么/关键决策/遇到的问题/下一步"四段，但不拦）
- 收束节点四阶段（整理/测试/审计/验证）无自动化闸门
- 新文件放置 vs FILE_GRAPH 决策树无校验

### 1.2 核心矛盾

> 规范定义了"该做什么"，但没有工具验证"做了没有"。
> "两头硬中间软"：规范文档（conventions/）有 L1+L4 双重保护，但规范自身的落地文件缺乏硬约束。

### 1.3 强制化的三层模型

```
步骤性质          检测时机          工具
─────────────────────────────────────────
文件产出（落盘）   提交时 / CI         pre-commit hook / CI job
交叉一致性        提交时 / CI         交叉校验脚本
行为规范（AI 自检） 对话中 / 审查时    CLAUDE.md 规则 + AI 自律
```

关键认识：**不是所有流程步骤都能自动化检查**。但能验证**落盘产物的存在性、格式、一致性**。

---

## 二、目标

### 可量化目标

| 目标 | 现状 | 目标值 |
|------|------|--------|
| 流程步骤有工具验证的比例 | ~15%（2/13 步骤） | ≥ 70% |
| 入口文件有章节级 L1 钩子 | 2 个（ai-workflow, code-understanding） | 5 个（+STATUS, CLAUDE, 开发清单） |
| markdownlint 有效规则数 | ~5 条（25 条禁用） | ≥ 15 条 |
| worklog ↔ STATUS 同步检测 | 零 | 提交时自动拦 |
| 模板漂移检测覆盖 | 2 类 | 5 类 |

### 非量化目标

- 让"红线"真正具有可执行性——违反红线的 commit 被工具拦截，而非仅靠人记忆
- 让流程步骤从"AI 自觉"升级为"工具验证"
- 不改变现有流程定义，只在检测层补盲区

## 三、方案

### 3.1 总体架构

```
现有流程定义（不动）
  │
  ├── conventions/ai-workflow_AI协作开发流程/   ← 流程定义（只读参考）
  └── conventions/01-08                         ← 规范定义（红线）

新增检测层（本次建设）
  │
  ├── pre-commit hooks（提交时拦）
  │   ├── check_worklog_ref.py          ← 已有，增强
  │   ├── check_status_updated.py       ← 🆕 worklog ↔ STATUS 同步
  │   └── check_worklog_structure.py    ← 🆕 worklog 内容结构
  │
  ├── CI l4-conventions stage（CI 时拦）
  │   ├── check_status.py               ← 🆕 STATUS.md 格式 L1
  │   ├── check_claude.py               ← 🆕 CLAUDE.md 结构 L1
  │   └── check_plan.py                 ← 🆕 开发清单.md 格式 L1 + 交叉校验
  │
  ├── CI build stage（渲染鲁棒性）
  │   └── render.py strict 模式         ← 🔧 safe_substitute → substitute
  │
  └── CI compliance stage（模板漂移）
      └── check_template_drift.py       ← 🆕 扩展到 5 类入口文件
```

### 3.2 分阶段实施

#### Phase 1：汇报流程强制化（#36-#39）

**目标**：让"功能点完成 → 更新 worklog + STATUS"从 AI 自觉变成提交时自动拦截。

| # | 任务 | 检测时机 | 内容 |
|---|------|---------|------|
| 36 | 增强 `check_worklog_ref.py` | commit-msg | 验证 commit message 引用的 worklog 文件在 `git diff --cached` 中存在；不通过则拦截 |
| 37 | 🆕 `check_status_updated.py` | pre-commit | 若 staged files 含 `worklogs/*.md`，则 `STATUS.md` 也必须在 staged files 中；不通过则拦截 |
| 38 | 🆕 `check_worklog_structure.py` | pre-commit | 验证 staged 的 worklog 文件含必需章节（完成了什么/关键决策/遇到的问题/下一步）+ 至少 1 个 `- [x]`；不通过则拦截 |
| 39 | markdownlint 规则审计 | pre-commit | 逐条评估 25 条禁用规则，区分"中文兼容必需禁"vs"误禁"，启用有效规则 |

**Phase 1 完成后**：commit 时自动验证 (a) worklog 文件真实存在，(b) STATUS 同步更新，(c) worklog 结构完整，(d) Markdown 格式合规。

#### Phase 2：入口文件格式强制化（#40-#42）

**目标**：为 STATUS.md / CLAUDE.md / 开发清单.md 建立章节级 L1 钩子，对标已有的 `check_ai_workflow.py`。

| # | 任务 | 覆盖文件 | 验证内容 |
|---|------|---------|---------|
| 40 | 🆕 `check_status.py` | STATUS.md | `## 当前进度` 节存在 → 表格列数≥4 → 状态值∈{✅,🔄,⏳,🚫}；`## 收束节点历史` 表列数≥9 |
| 41 | 🆕 `check_claude.py` | CLAUDE.md | 文件存在 → 含必需章节标题（项目概述/目录结构/红线表/工作流程/文件放置/当前状态）→ 红线表含≥6 维度行 |
| 42 | 🆕 `check_plan.py` | 开发清单.md | `## 功能点列表` 表格存在 → 列数≥7 → 状态值合法 → 功能点数量与 STATUS.md 一致 |

**Phase 2 完成后**：三类入口文件的结构完整性由 CI l4-conventions 阶段自动验证。

#### Phase 3：渲染鲁棒性 + 模板漂移（#43-#44）

| # | 任务 | 内容 |
|---|------|------|
| 43 | dashboard 渲染 strict 模式 | `render.py`: `safe_substitute` → `substitute`（fail-fast）；`parse_status()` 失败时非空返回 + warn；增加 STATUS 格式 pre-check |
| 44 | 模板漂移检测扩展 | 扩展 `render_meta.py --check` 或新增脚本，对 STATUS/CLAUDE/开发清单 vs `docs/templates/` 模板做章节存在性对比 |

#### Phase 4：深水区（#45-#46，后续）

| # | 任务 | 内容 |
|---|------|------|
| 45 | 🆕 `check_file_placement.py` | 验证新增文件的路径符合 `meta/FILE_GRAPH.md` 决策树 |
| 46 | 🆕 收束节点产物 CI 检查 | 验证收束节点后 ADR + 收束报告文件存在 |

### 3.3 豁免机制

所有新增钩子统一的豁免方式：commit message 末尾加对应标记。

| 钩子 | 豁免标记 |
|------|---------|
| check_worklog_ref.py | `[skip-worklog]`（已有） |
| check_status_updated.py | `[skip-status]` |
| check_worklog_structure.py | `[skip-worklog-structure]` |

## 四、影响范围

### 涉及模块

| 模块 | 影响 |
|------|------|
| `scripts/` | 新增 5 个检查脚本 + 增强 1 个 + 修改 1 个 |
| `tests/conventions/` | 新增 3 个 L4 测试文件（test_status, test_claude, test_plan） |
| `conventions/_meta.yaml` | 扩展 3 个 convention 的 `l1_check_path` |
| `.github/workflows/ci.yml` | l4-conventions 阶段新增 3 个 step |
| `.pre-commit-config.yaml` | 新增 2 个 local hook（由 _meta.yaml 投射） |
| `.markdownlint.json` | 修改规则开关 |
| `docs/templates/devguard/html-report-template/render.py` | 修改模板调用方式 |
| `CLAUDE.md` | 更新当前状态 + 红线表 |

### 不涉及

- 规范正文（conventions/01-08）— 内容不动
- 流程文档（ai-workflow/）— 流程定义不动
- 示例代码（src/）— 不动
- BDD 规格（docs/specs/）— 视需要新增 1-2 个

## 五、风险

| 风险 | 概率 | 缓解 |
|------|------|------|
| 钩子过于严格，频繁误拦 | 中 | 豁免标记 + 渐进启用（先在 CI 中 warn-only，稳定后升级为 error） |
| markdownlint 规则启用后大量既有文件报错 | 高 | 分批启用，每批修复既有问题后再开下一批 |
| `safe_substitute` → `substitute` 后 dashboard 渲染失败 | 低 | 增加 pre-validation，确保数据源正确后再渲染 |
| 跨文件一致性检查（STATUS ↔ 开发清单）因历史不一致立即报错 | 中 | 先短期豁免历史差异，只检查增量变更 |
| Phase 4 深水区（文件放置检查）复杂度高 | 高 | 标记为后续，先不进入本轮实施 |

可点击的演进细节

为何 V0.x 起步时 docs/specs/ 是空目录？
项目"第一天"只完成了基础目录搭建，BDD 内容的填充是后续 V0.2-V1.4 治理期的工作。留下交接事项："docs/specs/ 目录已创建但为空 —— 需要为已完成的功能点 1-7 补写 BDD 规格"。
为何 V0.x 重整要"长文拆目录"？
ADR 0005 决策：当一个 markdown 文档超过 X 行时，应拆分为"总览 + 多个子文件"的目录结构。06-第三步_收束节点.md 的模板提取就是此决策的首次执行。原因是：长文难以维护、难以引用、难以检索。
为何 V0.7 模板规约要入 _meta.yaml？
10 规范"模板与汇报"承担"模板的模板"角色 —— 它规定其他模板怎么写、怎么登记、怎么同步。必须用 _meta.yaml 真源治理类规范同等级处理，否则会出现"模板规则无人维护"的退化路径。
为何 V1.5 final-report 选 118KB 大模板？
V1.x 收尾汇报沉淀了 11 Mermaid + 3 Tab + 5 进度条 + 2 数字滚动 + 1 模拟器，是高密度学术风的成熟形态。沉淀为模板后，新项目 fork 时无需重新设计骨架，直接替换数据即可。
research N1-N6 双命名变体如何处理？
tree/N1-代码知识图谱构建方法.md 与 tree/N1-代码知识图谱的构建方法.md 是调研过程的命名变体（"构建方法" vs "的构建方法"）。V2.0+ 计划去重合并为单一命名。AI 不擅自删文件 —— 等待人审决定。

最终判定

M2 VERDICT · 2026-06-10

5 阶段 · 16 天 · 5 子目录 · 14 收束 · 3 HTML 最终报告

M2 文档资产从 V0.x 起步的 3 子目录到 V2.0.1 的 5 子目录，历经 5 阶段 16 天（2026-05-26 → 2026-06-10），沉淀 14 份收束报告 + 3 份 HTML 最终报告 + 6 份 ADR。V0.x 重整（5 子目录定型）+ V0.7 模板规约（10 规范入 _meta）+ V1.5 final-report-template（学术风 HTML 沉淀）三个关键节点证明了 M2 的演进能力：能在不破坏结构的前提下增加新规范、能在不丢失历史的前提下沉淀新模板。

后续扩展提示：research 子模块 N1-N6 双命名变体 V2.0+ 去重；10 规范当前只校验"登记 + 路径"，不校验"内容质量"；V1.x 收尾汇报 HTML（3 份）目前是手工产出，下个版本 V2.0+ 计划全用 final-report-template/render.py 跑出。

## 设计.md 章节导航

### §一 模块定位
- **看什么**：过程证据库；解决"为什么这样做 / 上周做了什么 / commit 改了什么 / 决策从哪来" 4 个问题
- **想什么**：为什么 worklog 不入 `docs/` 而入根？—— 因为它是"工作流核心"而非"文档集"
- **易踩坑**：worklog ≠ 汇报 ≠ changelog ≠ wiki

### §二 目录结构
- **看什么**：31 份 worklog + 6 份 ADR；按"基础设施 / 模板 / 规范 / 流程 / 决策 / 收束 / 补登" 分类型
- **易踩坑**：ADR 在 `worklogs/decisions/` 子目录，不在 `docs/decisions/`

### §三 命名规约
- **看什么**：`YYYY-MM-DD_<描述>.md` + 收束节点前缀 `v<X>-` + 断档补登前缀 `v<X>-backfill-`
- **核心理解**：命名规约与 `check_worklog_ref.py` 正则双向绑定
- **易踩坑**：同日多份要带下划线后缀（`_二轮审查修复`）

### §四 内容五段式
- **看什么**：做了什么（必填）/ 验证结果（必填）/ 关键决策（可选）/ 遇到的问题（可选）/ 给下一位的交接（可选）
- **核心理解**：必填段不可缺（"汇报不断档"硬约束）
- **易踩坑**：可选段按需增删，但保留必填段即合规

### §五 ADR 设计
- **看什么**：6 份 ADR 列表 + ADR 0006 详解
- **核心理解**：**ADR 0006 是本模块最关键决策**——"ADR 仅收束节点产出"
- **想什么**：为什么不让 ADR 即时创建？—— 避免碎片化 + 经一批功能点验证后更准确

### §六 commit-msg 强制引用
- **看什么**：`scripts/check_worklog_ref.py` 正则 + 4 种引用形式 + 豁免标记
- **核心理解**：红线 6 是"汇报不断档"的兜底机制
- **易踩坑**：不能用 `--no-verify` 绕过；豁免仅 `[skip-worklog]`

### §七 与汇报体系的关系
- **看什么**：5 种产出物（功能点汇报 / worklog / 收束报告 / ADR / STATUS）落点对照表
- **核心理解**：粒度递增 + 频度递减

### §八 生命周期 mermaid
- **看什么**：功能点 → worklog → commit → STATUS → 收束 → ADR 完整链路
- **想什么**：每个节点都有"必填动作"——漏掉任一节点会断档

### §九 耦合关系
- **看什么**：9 个耦合方（conventions / scripts / STATUS / 开发清单 / reports / ai-workflow）
- **核心理解**：worklog 是工作流"中心节点"，不是孤立目录

## 思考题

1. **为什么 ADR 仅收束节点产出？** —— 决策质量更高（经一批功能点验证）+ 减少噪音（微决策留 worklog）+ 可追溯性增强（worklog → decisions 完整链）。但如何避免"该升格的没升格"？
2. **commit-msg 强制引用会不会让 commit 信息变冗长？** —— 当前允许 4 种引用形式（完整路径 / 相对路径 / 裸文件名 / 链接），可以"详见 worklogs/2026-05-27_*.md" 简写。是否需要更激进的模板？
3. **断档补登的代价有多大？** —— V1.5 用 1 份 5KB 巨型 worklog 回填 V0.2-V1.4 全部 30+ 功能点。如果再积压 3 个月，可能需要专门的 "backfill agent" 批量重写？
4. **worklog 与 git commit 是否冗余？** —— 答：worklog 是"为什么这样做"（决策 + 问题 + 交接），commit 是"做了什么"（diff）。两者互补不冗余。但 git log 的 `-S` / `-G` 搜索能否替代 worklog？
5. **同日 9 份 worklog 是不是过细？** —— V0.x 扩展当天确实极端，但每份都对应一个独立功能点（CLAUDE 角色分离 / worklog 决策规则 / 汇报体系 / 命名规范化等）。是否需要"日合并 worklog"机制？
6. **ADR 编号 0001-0006 是否会饱和？** —— 6 份 / 16 天 ≈ 1 份 / 2.7 天。如果持续高强度开发，2-3 年后会到 0500+。是否需要分主题子目录（`decisions/architecture/`）？
7. **`worklogs/decisions/` 能不能也走 commit-msg 强制引用？** —— 答：ADR 是"决策落盘"动作，本身就是"对 worklog 关键决策的引用"，再要求引用反而冗余。但 ADR 路径引用本身可以校验（防止漂移到 `docs/decisions/`）。

## 易踩坑速查

| 症状 | 真因 | 修复 |
|------|------|------|
| commit 被 `check_worklog_ref.py` 拦截 | commit-msg 缺 worklog 引用 | 在 commit message 末尾加 `（详见 worklogs/YYYY-MM-DD_*.md）` |
| worklog 命名带空格 | 违反 `YYYY-MM-DD_<描述>.md` 规约 | 改用下划线 `_` 或 kebab-case |
| 收束节点未到但 worklog 已写 | 不阻止（写入无闸门） | 写完后用 `git commit` 触发 commit-msg 钩子验证 |
| ADR 写到 `docs/decisions/` | 违反 ADR 0006 路径 | 迁移到 `worklogs/decisions/` + 更新内部链接 |
| worklog「关键决策」段写得太长 | 微决策过度形式化 | 简化到一行（决策 / 原因 / 影响），长文写到 ADR |
| 收束节点忘补 worklog | 未即时回填 | 用 `v<X>-backfill-<范围>.md` 一次性回填（参考 V1.5） |
| `[skip-worklog]` 滥用 | 紧急 hotfix 忘了补 worklog | 后续 issue 跟踪 + 人工补 worklog |
| 同日多份 worklog 重名 | 缺下划线后缀 | 加 `_二轮` / `_三轮` 后缀 |
| worklog 文件名超 60 字符 | 描述词堆砌 | 精简到 ≤ 30 字符核心词 |
| worklog 缺"验证结果"段 | 当日太忙偷懒 | 补写"已跑 `pytest tests/conventions/` 65 passed" 等客观证据 |

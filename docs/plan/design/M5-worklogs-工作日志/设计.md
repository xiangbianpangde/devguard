## 一、模块定位

### 它是什么

`worklogs/` 是 devguard 的**过程证据库**。它沉淀每个功能点（含收束节点）的"做了什么、验证了什么、做了什么决策、踩了什么坑、给下一位留什么"，是 AI 协作 + 人类回顾的"时间轴"。

它解决的核心问题：

| 问题 | 解法 |
|------|------|
| "为什么这个决策当时这样做？" | 决策 = worklog 「关键决策」段 → 可上溯 |
| "上周做了什么？" | worklog 日流 → STATUS.md 索引 |
| "commit 都在改什么？" | commit-msg 强制引用 worklog → 一键回溯 |
| "这个架构决策从哪来？" | ADR 升格自 worklog（ADR 0006） |

### 它不是什么

- **不是汇报库** —— 收束报告才落 `docs/reports/`；worklog 是"开发过程"，不是"对外汇报"
- **不是 changelog** —— `CHANGELOG.md` 给用户看；worklog 给后续开发者/AI 看
- **不是会议纪要** —— worklog 关注"代码/配置/规范"变更；会议讨论不在此
- **不是 wiki** —— worklog 是时间序列的；不鼓励跨条目互链成知识网

## 三、worklog 命名规约（强制）

### 格式

```
worklogs/YYYY-MM-DD_<简短描述>.md
```

### 规则

| 段 | 必填 | 规约 |
|----|:---:|------|
| `YYYY-MM-DD` | ✅ | ISO 8601 日期；同日多份用下划线后缀（如 `_二轮审查修复`）|
| `<描述>` | ✅ | 中文为主、kebab-case 英文为辅；动词/对象/范围清晰 |
| `.md` | ✅ | UTF-8 / LF 行尾 |
| 收束节点前缀 | 可选 | `v<X>-<收束主题>`（如 `v15-section3-table-to-charts`） |
| 断档补登前缀 | 可选 | `v<X>-backfill-<范围>`（如 `v15-backfill-v02-v14-worklogs`） |

### 与 commit-msg 引用的对应

- `scripts/check_worklog_ref.py` 正则：`worklogs[\\/]+\d{4}-\d{2}-\d{2}[_\-\w]*\.md`
- 允许的引用形式：
  - 完整路径：`worklogs/2026-05-27_规范正文v2重构.md`
  - 相对路径：`../worklogs/2026-05-27_规范正文v2重构.md`
  - 文件名裸引用：`2026-05-27_规范正文v2重构.md`
  - Markdown 链接：`[worklog](worklogs/2026-05-27_规范正文v2重构.md)`
- 豁免标记：`[skip-worklog]`（仅收束节点 / 紧急 hotfix 用）

## 四、worklog 内容五段式（来自模板）

```markdown
# 工作日志：{简短标题}

> 日期: YYYY-MM-DD
> 功能点: #{编号} {功能点名称}
> 关联 BDD: {docs/specs/NN-xxx.md，无则写 -}

## 做了什么                          # 必填
{本轮改了什么、为什么。可分小节列条目}

## 验证结果                          # 必填
<!-- 客观证据：测试通过数 / 手动验证 / 截图 / dashboard -->
{...}

## 关键决策                          # 可选
| 决策 | 原因 | 影响 |
|------|------|------|
| {决策} | {...} | {...} |

## 遇到的问题                        # 可选
- {问题}：{解决方法}

## 给下一位的交接                    # 可选
> {下一步该做什么 / 哪里容易踩坑 / 临时约定}
```

> 五段式与 `docs/templates/worklog模板.md` 严格对齐。必填段不可缺（"做了什么" + "验证结果"），可选段按需增删。

## 五、ADR 设计（`worklogs/decisions/`）

### 5.1 文件格式

```markdown
# ADR NNNN: {简短标题}

> 日期: YYYY-MM-DD
> 状态: 已采纳 | 已拒绝 | 已替代

## 背景
{为什么需要做这个决策}

## 决策
{具体选了什么}

## 理由
{为什么选这个}

## 后果
### 操作变更
### 文件变更
### 历史 ADR  # （仅迁移类）
```

### 5.2 6 份 ADR 列表

| # | 标题 | 决策 | 影响 |
|---|------|------|------|
| **0001** | 规范体系结构 | 6 维度开发规范 + 治理规范的分层 | conventions 雏形 |
| **0002** | 六维度划分 | 架构/代码/Git/API/测试/文档 为核心 | 01-06 编号定形 |
| **0003** | AI 协作流程独立成文 | AI 协作 ≠ 规范本身，独立目录 | `ai-workflow_*/` 创立 |
| **0004** | 规范移入 conventions 目录 | 规范从根目录移入 `conventions/` | 目录结构稳定 |
| **0005** | 长文件拆文件夹 | 单文件 > 300 行必须拆目录 | ai-workflow 10 篇拆分基础 |
| **0006** | 仅在约束节点通过 worklog 整理并进行决策总结 | ADR 仅收束节点产出 | worklog / ADR 单文档流 |

### 5.3 "ADR 仅收束节点产出"决策详解

> 出处：ADR 0006（2026-05-27 收束节点产物）

**问题**：

1. 决策碎片化：开发过程中随时记录 ADR → 缺乏阶段性审视
2. 与工作流脱节：ADR 独立于 worklog → 难追溯"这个决策是在什么背景下做出的"
3. 过早形式化：单个功能点内的取舍被过早提升为 ADR，但实际影响有限
4. 重复记录：worklog「关键决策」段与 ADR 内容大量重叠

**决策**：

1. `decisions/` 移入 `worklogs/decisions/`：ADR 不再是独立文档流，而是 worklog 体系的子集
2. 仅在收束节点（人触发）时整理决策：日常开发中，技术取舍记 worklog「关键决策」段即可
3. ADR 编号与收束节点对齐：新 ADR 编号紧随上次收束节点的最后一个

**理由**：

- 决策质量更高：经一批功能点验证后再判断"值得永久存档"，比即时判断更准确
- 减少噪音：微决策留 worklog，只有跨批次/影响架构方向的决策才需 ADR
- 可追溯性增强：worklog → decisions 完整还原决策链（why + when + what context）

**操作变更**：

- 日常开发：技术取舍只记录在 worklog「关键决策」段（可选段）
- 收束节点：回顾本批次 worklog → 判断哪些值得提升为 ADR → 写入 `worklogs/decisions/`
- ADR 不再接受"即时创建"，必须是收束节点产出物

## 六、commit-msg 强制引用 worklog（红线 6）

### 6.1 机制

```yaml
# conventions/_meta.yaml（节选）
- id: commit-msg-worklog-ref
  name: commit message must reference worklog file
  description: "06 红线：commit message 必须引用 worklog 文件（汇报不断档）"
  entry: python scripts/check_worklog_ref.py
```

### 6.2 检测逻辑（`scripts/check_worklog_ref.py`）

```python
WORKLOG_PATTERN = re.compile(
    r"worklogs[\\/]+\d{4}-\d{2}-\d{2}[_\-\w]*\.md",
    re.IGNORECASE,
)

# 整段 commit message 至少要有 1 处 worklog 引用
if WORKLOG_PATTERN.search(msg):
    return 0   # 通过
return 1       # 拦截
```

### 6.3 典型 commit 格式

```bash
git commit -m "feat(xxx): 实现功能（详见 worklogs/2026-05-27_规范正文v2重构.md）"
```

### 6.4 豁免

commit message 末尾加 `[skip-worklog]` 标记，可绕过检测。仅用于：

- 收束节点批量提交（一次 commit 含多个 worklog 引用）
- 紧急 hotfix（先修复再补 worklog）
- 文档/元数据纯修改（如 `FILE_GRAPH.md` 决策树调整）

## 七、与汇报体系的关系

| 产出物 | 落点 | 时机 | 受众 |
|--------|------|------|------|
| **功能点级汇报** | 对话内联（不落盘） | 每功能点完成 | 用户 |
| **worklog** | `worklogs/YYYY-MM-DD_*.md` | 每功能点完成 | 后续开发者/AI/审计 |
| **收束报告** | `docs/reports/<版本>-<主题>.md` | 收束节点 | 团队/历史归档 |
| **ADR** | `worklogs/decisions/NNNN-*.md` | 收束节点 | 后续开发者/AI |
| **STATUS 更新** | `STATUS.md` 每功能点一行 | 每功能点完成 | dashboard |

> **关键区分**：worklog ≠ 收束报告 ≠ ADR。worklog 是"过程证据"，收束报告是"阶段性总结"，ADR 是"架构决策"。三者粒度递增，频度递减。

## 十、边界与降级

| 场景 | 降级行为 |
|------|---------|
| commit-msg 钩子失败 | 拒绝 commit；要求补充 worklog 引用（不能用 `--no-verify` 绕过） |
| worklog 缺"做了什么"或"验证结果"段 | 不被工具拦（建议审查时人工检查） |
| 收束节点未到但出现跨批次决策 | 留 worklog「关键决策」段，等下次收束时回溯 |
| 收束节点已到但 worklog 缺漏 | 用 `v<X>-backfill-<范围>.md` 补登（参考 `2026-06-08_v15-backfill-v02-v14-worklogs.md`） |
| `[skip-worklog]` 滥用 | 由 `git log --grep="[skip-worklog]"` 审计；人触发复盘 |
| ADR 路径漂移到 `docs/decisions/` | 不接受；ADR 0006 已固化路径 |
| worklog 文件名超 60 字符 | 不被工具拦（建议审查时人工检查） |

### 明确不做

- **不做**：把 worklog 模板改成 yaml 结构化（牺牲可读性）
- **不做**：worklog 入 git 时压缩（如 `.worklog.json`）—— Markdown 才是协作最友好格式
- **不做**：worklog 数量超过 100 份时自动归档 —— 当前 31 份无此压力
- **不做**：把 ADR 改名/重编号 —— 编号是历史，禁动

## 十一、验收标准

| 维度 | 验收方式 | 通过标准 |
|------|---------|---------|
| 命名合规 | 全部 31 份文件 | 全部匹配 `YYYY-MM-DD_<描述>.md` |
| commit-msg 强引 | `python scripts/check_worklog_ref.py` | 全量 commit 命中 |
| ADR 路径合规 | `ls worklogs/decisions/` | 6 份全部位于此目录 |
| ADR 编号连续 | 编号 0001-0006 | 无跳号 |
| 模板合规 | 人工审阅 31 份 | "做了什么" + "验证结果" 必填段全在 |
| 仪表盘同步 | `collect_l4_stats.py` 统计 worklog 数 | = 31 |
| 收束节点 worklog | V1.5 期间 | ≥ 1 份 `v15-*.md` |
| 断档补登 worklog | V1.5 期间 | = 1 份 `v15-backfill-*.md` |
| 收束报告落盘 | `ls docs/reports/` | 与 worklog 区分（reports/ 不在本模块） |

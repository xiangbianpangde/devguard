01 · 模块定位

worklogs/ —— devguard 的过程证据库

worklog 不是孤立的"日志文件"，而是 AI 协作 + 人类回顾的时间轴。它回答四个核心问题：

"为什么这个决策当时这样做？"

决策记录在 worklog「关键决策」段 → 经收束节点审阅 → 必要时升格为 ADR。可上溯"why + when + what context"。

why + when + context 可上溯

"上周做了什么？"

worklog 日流按 ISO 日期排序 → 配合 STATUS.md 索引 → dashboard 实时可视化。一眼看出本周 / 本月 / 收束周期内产出。

日流 + 索引 dashboard 实时

"commit 都在改什么？"

commit-msg 强制引用 worklog → git log --grep 即可一键回溯。哪一次提交属于哪一份 worklog，零歧义。

commit ↔ worklog 零歧义

"这个架构决策从哪来？"

ADR 升格自 worklog（ADR 0006 决策：仅收束节点产出）。worklog → decisions 完整还原决策链；ADR 编号 = 历史，禁动。

worklog → ADR 编号即历史

它不是什么 · 边界澄清

✗ 不是汇报库

收束报告才落 docs/reports/。worklog 是"开发过程"，不是"对外汇报"。

✗ 不是 changelog

CHANGELOG.md 给最终用户看；worklog 给后续开发者 / AI / 审计看。

✗ 不是会议纪要

worklog 关注代码 / 配置 / 规范变更；会议讨论不在此。

i
关键区分：worklog = 过程证据（细粒度、高频度），收束报告 = 阶段性总结（中粒度、低频度），ADR = 架构决策（粗粒度、最频次低）。三者粒度递增、频度递减，互相不可替代。

03 · 命名规约（强制）

YYYY-MM-DD_<描述>.md —— 5 段规则

# worklog 文件名格式（强制） worklogs/YYYY-MM-DD_<简短描述>.md # 示例 worklogs/2026-05-27_规范正文v2重构.md worklogs/2026-06-08_v15-section3-table-to-charts.md worklogs/2026-06-08_v15-backfill-v02-v14-worklogs.md worklogs/2026-06-09_ai-workflow-v2-rewrite.md
段必填规约
YYYY-MM-DD✓ISO 8601 日期；同日多份用下划线后缀（如 _二轮审查修复）
<描述>✓中文为主、kebab-case 英文为辅；动词 / 对象 / 范围清晰
.md✓UTF-8 / LF 行尾（与全仓库一致）
收束节点前缀可选v<X>-<收束主题>（如 v15-section3-table-to-charts）
断档补登前缀可选v<X>-backfill-<范围>（如 v15-backfill-v02-v14-worklogs）

commit-msg 引用的对应（4 种合法形式）

# 1. 完整路径 git commit -m "feat(xxx): 实现功能（详见 worklogs/2026-05-27_规范正文v2重构.md）" # 2. 相对路径 git commit -m "feat(xxx): 实现功能（详见 ../worklogs/2026-05-27_规范正文v2重构.md）" # 3. 文件名裸引用 git commit -m "feat(xxx): 实现功能（详见 2026-05-27_规范正文v2重构.md）" # 4. Markdown 链接 git commit -m "feat(xxx): 实现功能（[worklog](worklogs/2026-05-27_规范正文v2重构.md)）"

豁免标记

!
commit message 末尾加 [skip-worklog] 标记可绕过检测。仅用于：① 收束节点批量提交（一次 commit 含多个 worklog 引用）② 紧急 hotfix（先修复再补 worklog）③ 文档 / 元数据纯修改（如 FILE_GRAPH.md 决策树调整）。
审计：git log --grep="[skip-worklog]" 列出所有豁免 → 人触发复盘（滥用人触发复盘）。
04 · 内容五段式（来自模板）

必填 2 段 + 可选 3 段 —— 不可缺段

# 工作日志：{简短标题} > 日期: YYYY-MM-DD > 功能点: #{编号} {功能点名称} > 关联 BDD: {docs/specs/NN-xxx.md，无则写 -} ## 做了什么 # 必填 {本轮改了什么、为什么。可分小节列条目} ## 验证结果 # 必填 <!-- 客观证据：测试通过数 / 手动验证 / 截图 / dashboard --> {...} ## 关键决策 # 可选 | 决策 | 原因 | 影响 | |------|------|------| | {决策} | {...} | {...} | ## 遇到的问题 # 可选 - {问题}：{解决方法} ## 给下一位的交接 # 可选 > {下一步该做什么 / 哪里容易踩坑 / 临时约定}

五段式 · Tab 详解

点击切换看每段的必填理由、典型内容、缺失后果

必填① 做了什么
必填② 验证结果
可选③ 关键决策
可选④ 遇到的问题
可选⑤ 给下一位的交接

① 做了什么（必填）

理由：这是 worklog 的核心——本轮改了什么、为什么改。后续追溯时，开发者第一眼看的就是这段。

典型内容：分小节列条目；说明改动的文件 / 配置 / 规范；指出与上次 worklog 的差异。

缺失后果：无工具拦（建议审查时人工检查），但下游"交接"段、"决策"段无法形成完整证据链。

② 验证结果（必填）

理由：避免"自证"陷阱。commit 中说的"实现了功能"必须有客观证据：测试通过数、手动验证、截图、dashboard 截图。

典型内容：pytest 8 passed、curl POST /api/v1/users → 201、dashboard 截图链接、浏览器实拍。

缺失后果：无工具拦（建议审查时人工检查），但违反项目级"质量验证原则"——demo 视频 / 录屏不充当合格证据。

③ 关键决策（可选）

理由：记录"为什么这样做而不那样做"。日常技术取舍（不升格为 ADR）留此即可——比即时写 ADR 更轻量。

典型内容：三列表格（决策 / 原因 / 影响）；影响"跨批次 / 架构方向"才升格为 ADR。

缺失后果：不影响功能点完成度；但"为什么这样做"会在 3 个月后失忆。

④ 遇到的问题（可选）

理由：失败比成功更有教学价值。"我在这卡了 2 小时、原因是 X、解决是 Y"是知识沉淀的核心。

典型内容：- 问题描述：解决方法 列表；附错误信息片段。

缺失后果：踩过的坑会被后人再踩一次。

⑤ 给下一位的交接（可选）

理由：worklog 是"时间序列的"，下一位接手时第一眼看的段落。临时约定 / 容易踩的坑 / 还未完成的部分都列在此。

典型内容：引用 blockquote 突出"下一步该做什么"；列 TODO 项。

缺失后果：下一位开发者 / AI 需重新摸索，浪费起步时间。

▌
与模板严格对齐：五段式骨架与 docs/templates/worklog模板.md 一一对应。必填段（"做了什么" + "验证结果"）不可缺；可选段（决策 / 问题 / 交接）按需增删。不要私自加新段（如加"参考资料"会破坏模板与人工审阅的对应关系）。

ADR 0001 · 规范体系结构 2026-05-27
背景

项目初期（V0.x）规范散落根目录，缺少体系化分层。需要建立"开发规范 + 治理规范"的双层结构。

决策

采用 6 维度开发规范（架构 / 代码 / Git / API / 测试 / 文档）+ 治理规范（安全 / 支持 / 许可证 / 贡献 / 仪表盘 ...）的分层结构。

理由

分层让"开发期规范"与"项目治理规范"职责清晰；6 维度对应开发流程的关键节点。

后果

conventions 雏形建立；后续 0002-0005 在此基础上继续细化。

ADR 0002 · 六维度划分 2026-05-27
背景

6 维度是哪 6 个？需要明确边界，避免"维度数量漂移"。

决策

核心 6 维度 = 架构 / 代码 / Git / API / 测试 / 文档（编号 01-06），治理类（09-17）另起编号。

理由

这 6 个维度对应开发流程的关键节点（设计 → 实现 → 提交 → 接口 → TDD → 文档），是公认的开发实践切分。

后果

01-06 编号定形；后续所有规范都依此编号规则。

ADR 0003 · AI 协作流程独立成文 2026-05-27
背景

原 07-ai-workflow 规范只是"开发规范"中的一条，但 AI 协作流程有 自身的完整生命周期（调研 → 计划 → 迭代 → 收束），与开发规范不同。

决策

AI 协作流程独立成文，单独目录 conventions/ai-workflow_*/，编号 07 留给"传统 AI 协作原则"，独立流程本身占据目录级地位。

理由

AI 协作流程迭代频繁（V1.x → V2.0 → V2.0.1），独立目录便于版本化；与 01-06 规范解耦。

后果

ai-workflow_*/ 创立；7 篇流程文档（01-07）独立维护。

ADR 0004 · 规范移入 conventions 目录 2026-05-27
背景

规范起初散落根目录（/01-architecture.md、/02-coding.md ...），与项目代码混杂，新人无法快速定位。

决策

规范从根目录整体移入 conventions/，根目录只保留 README.md（人类入口） + CLAUDE.md（AI 入口）。

理由

集中存放 + 角色分离（人 / AI 双入口）让导航成本降到最低。

后果

目录结构稳定；CLAUDE.md 与 README.md 成为双入口。

ADR 0005 · 长文件拆文件夹 2026-05-27
背景

ai-workflow 流程最初单文件 ~1200 行，跨 AI 协作全部步骤（调研 → 计划 → 迭代 → 收束），导航、编辑、review 全部不便。

决策

单文件 > 300 行必须拆目录（"长文件"判定标准）；ai-workflow 拆为 7 篇（01-07）。

理由

300 行 = 视觉上"一屏到顶"，编辑 review 不需要滚动；拆目录后每篇聚焦一个流程环节。

后果

ai-workflow 10 篇拆分基础建立；后续 conventions/ 内所有文件都遵守此约束。

ADR 0006 · 仅收束节点产出 ADR 2026-05-28 关键决策
背景

ADR 起初独立成 docs/decisions/，开发过程中随时记录。结果浮现 4 个问题：

① 决策碎片化：随时记录 → 缺乏阶段性审视 ② 与工作流脱节：ADR 独立于 worklog → 难追溯 ③ 过早形式化：单个功能点内的取舍被过早提升为 ADR，但实际影响有限 ④ 重复记录：worklog「关键决策」段与 ADR 内容大量重叠

决策

① decisions/ 移入 worklogs/decisions/（ADR 不再是独立文档流，而是 worklog 体系的子集）
② 仅在收束节点（人触发）时整理决策；日常技术取舍记 worklog「关键决策」段即可
③ ADR 编号与收束节点对齐：新 ADR 编号紧随上次收束节点的最后一个

理由

① 决策质量更高：经一批功能点验证后再判断"值得永久存档" ② 减少噪音：微决策留 worklog，只有跨批次 / 影响架构方向的决策才需 ADR ③ 可追溯性增强：worklog → decisions 完整还原决策链

后果

操作变更：日常开发 → 技术取舍只记录在 worklog「关键决策」段；收束节点 → 回顾本批次 worklog → 升格 ADR → 写入 worklogs/decisions/；ADR 不再接受"即时创建"。
文件变更：docs/decisions/ 目录废弃（迁移至 worklogs/decisions/）；FILE_GRAPH.md 决策树更新。
历史 ADR：0001-0005 路径从 docs/decisions/ 迁至 worklogs/decisions/。

VERDICT · ADR 0006 的核心影响

从"开发中即时写 ADR" → "收束节点统一升格"

这是工作流的一次重要转向：把决策质量（经一批功能点验证）放在决策速度（即时记录）之前。日常技术取舍留 worklog「关键决策」段（轻量），跨批次 / 架构方向的决策才升格 ADR（重量）。

06 · commit-msg 强制引用 worklog（红线 6）

git commit 不引 worklog → 拒绝

本模块的核心红线：commit message 必须引用 worklog 文件，否则拦截。机制：scripts/check_worklog_ref.py 在 commit-msg 阶段正则匹配。

机制 · _meta.yaml 登记

场景降级行为

明确不做

✗ 不做 · worklog 模板改 yaml 结构化

理由：牺牲可读性。Markdown 才是协作最友好格式。

✗ 不做 · worklog 入 git 时压缩

如 .worklog.json。理由：Markdown 才是协作最友好格式。

✗ 不做 · worklog 超过 100 份自动归档

当前 31 份无此压力。理由：避免"过早形式化"。

✗ 不做 · ADR 改名 / 重编号

编号是历史，禁动。理由：路径 / 编号是工作流的"指纹"。

## 一、模块定位

### 它是什么

`meta/` 是 devguard 的**文件治理中枢**。整个仓库的"目录结构、文件归类、引用关系"全部由 `FILE_GRAPH.md` 这**唯一一份**文件权威定义。它不是文档库的成员——它是**描述文档库自身**的元信息。

### 它不是什么

- **不是规范正文** —— 规范在 `conventions/`（M1），本模块说"文件应该放哪"而不说"代码应该怎么写"
- **不是 README** —— `README.md` 是人类使用指南，本模块是**结构层面的**文件归类权威
- **不是工作日志** —— `worklogs/`（M5）记录过程，本模块描述仓库**当下长什么样**
- **不是 ADR** —— ADR 在 `worklogs/decisions/`，那是"为什么这样设计"，本模块是"现在长什么样"

## 二、为什么单文件就够了

### 设计判断

| 备选 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 拆 3 份：tree.md + refs.md + decision.md | 各职能可独立维护 | 决策树要引用目录树节点、引用关系要看完整结构，拆开导致上下文断裂 | ❌ |
| 拆 2 份：tree+refs / decision | 决策树相对独立 | 引用关系与目录树紧耦合，拆开仍需互引 | ❌ |
| **单文件 `FILE_GRAPH.md`** | 三职能在同一上下文、读者一处读完 | 文件长但 ~170 行可控 | ✅ |

**核心理由**：决策树的每个分支的"目标路径"必须出现在 §一 目录树里；引用关系的每个节点也必须出现在 §一 目录树里。**目录树是底座**，其他两章是底座上的衍生视图——拆开就会失去底座。

### 文件规模约束

- 当前：约 170 行
- 触发拆分的阈值：300 行（参考 ADR 0005 "长文拆目录"）——按当前扩展速度，**未来 2 年内不太可能突破**

## 三、目录结构

```
meta/
└── FILE_GRAPH.md    # 文件归类权威（目录树 + 引用关系 + 决策树）
```

> **单文件目录**：不像 `conventions/`（M1）有 17 篇规范、不像 `docs/`（M2）有 plan/reports/research/specs/templates 五个子目录；本模块只有 1 份核心文件。

## 四、核心设计：三大职能（§一 / §二 / §三）

### 4.1 §一 目录树 —— 仓库的"地图"

```
带职责标注的目录树
  ↓
回答"这个文件是干嘛的"
```

设计要点：
- **不显示行数**（行数是动态的，结构才是稳定的）
- **每个节点有职责注释**（"仓库根"、"规范正文"、"模板权威来源"）
- **顶层入口文件单列**（CLAUDE.md / README.md / STATUS.md / dashboard.html）

### 4.2 §二 引用关系图谱 —— 仓库的"调用图"

```
conventions/规范 ←互检→ docs/specs/BDD
        ↓ 落地为示例
   src/<维度>/
        ↓ 实例化
   docs/plan/开发清单 → STATUS.md → dashboard.html
        ↓ 每个功能点
   worklogs/YYYY-MM-DD_*.md
```

设计要点：
- **单向同步链**：conventions → specs → src
- **循环引用只发生在 conventions ↔ specs**（互检机制）
- **关键同步约束**列 4 条（改规范、加功能点、dashboard 改表头、模板复制）

### 4.3 §三 新文件决策树 —— 仓库的"路由器"

```
"要新增一个文件"
    ↓
13 个文件类型分支（见 §五）
    ↓
唯一归位路径
```

设计要点：
- **以问题开头**（"要新增一个文件，它是……"）
- **每个分支给出确切目标路径**（不只是原则，而是 `conventions/NN-*.md`）
- **末端是警告**（"根目录不接受新散文件"）

## 五、决策树覆盖的 13 个文件类型

| # | 文件类型 | 目标路径 | 触发条件 |
|---|---------|---------|---------|
| 1 | 规范正文 | `conventions/NN-<name>_<中文名>.md` | 某个开发维度的"应该怎么做" |
| 2 | BDD 验收标准 | `docs/specs/NN-<name>.md` | 某规范/功能点的可执行验收 |
| 3 | ADR 决策 | `worklogs/decisions/NNNN-<标题>.md` | 架构/技术选型"为什么选 A 不选 B"（仅收束节点） |
| 4 | 项目计划 | `docs/plan/背景.md` + `开发清单.md` | 项目级计划与功能点清单 |
| 5 | 设计文档 | `docs/plan/design/设计文档-<功能名>.md` | 复杂功能（背景/目标/方案/影响/风险） |
| 6 | 进度汇报 | `docs/reports/...` | 功能点完成 / 收束报告 |
| 7 | 调研笔记 | `docs/research/<主题>.md` | 调研产出 |
| 8 | 可复用模板 | `docs/templates/<类型>模板.md` | 新项目可复制的文件模板 |
| 9 | 规范示例代码 | `src/<对应规范维度>/...` | 规范的可运行配置示例 |
| 10 | 功能点日志 | `worklogs/YYYY-MM-DD_<简短描述>.md` | 一次功能点的开发记录 |
| 11 | 工具脚本 | `scripts/<name>.py\|.bat\|.sh` | 构建/启动/部署辅助脚本 |
| 12 | 仓库元信息 | `meta/<name>.md` | 描述仓库自身的文件 |
| 13 | 项目级入口 | 根目录 4 类文件 | **仅 4 类**：CLAUDE.md / README.md / STATUS.md / dashboard.html |

> **覆盖度核验**：经 V0.x 重整后（`worklogs/2026-05-27_文件夹重整与plan-design合并.md`），仓库所有现存文件都能在 13 个分支中找到归宿。

## 六、与 CLAUDE.md 的"互为索引"关系

### 关系图

```
meta/FILE_GRAPH.md  ←─── 互为索引 ───→  CLAUDE.md
       │                                    │
       │ §一 目录树                          │ "目录索引"段
       │ 详细版                              │ 精简版（直接摘抄）
       ↓                                    ↓
       "目录结构 + 引用关系 + 决策树"          "目录结构 + 引用关系 + 核心规则"
```

### 同步纪律

| 触发 | 同步动作 |
|------|---------|
| `FILE_GRAPH.md` §一 目录树改了 | `CLAUDE.md` "目录索引"段必须同步摘抄 |
| `CLAUDE.md` "目录索引"段改了 | `FILE_GRAPH.md` §一 必须同步扩充 |
| 任一边改完未同步另一边 | **CI 阶段 `compliance` 应阻断**（暂未自动化，靠人工纪律） |

### 为什么是"互为索引"而不是"双向同步"

- **CLAUDE.md** 给 AI 速读用，精简
- **FILE_GRAPH.md** 给所有读者细读用，详尽
- 两份文档的**职能不同**（AI 上下文 vs 文件归类权威），所以保留"双份正文"
- 但内容上必须保持一致——"互为索引"表示两边互为可信源，修改时同步即可

## 七、维护规则

### 何时更新 `FILE_GRAPH.md`

| 触发场景 | 必做动作 |
|---------|---------|
| 新增任何文件/目录 | §一 目录树 + §三 决策树核验 |
| 移动任何文件/目录 | §一 目录树 + §二 引用关系 |
| 改了 `conventions/` 规范 | §二 引用关系（约束 1）→ 同步 specs |
| 新增功能点 | §二 引用关系（约束 2）→ 同步 STATUS + 开发清单 |
| 新增一类文件类型 | §三 决策树增 1 个分支 |
| `CLAUDE.md` 改了目录段 | §一 目录树同步扩充 |

### 维护原则

- **不在图谱里维护细节**（如具体行数、临时状态）——细节漂移会拖垮维护
- **不在图谱里维护历史**（如"V0.x 增加了什么"）——历史在 worklog
- **只在图谱里维护"现在长什么样"**（目录结构 + 引用关系 + 归位规则）

## 八、边界与降级

| 场景 | 降级行为 |
|------|---------|
| 决策树没有覆盖的新文件类型 | **禁止**新增，先在 §三 决策树增 1 个分支再创建文件 |
| `FILE_GRAPH.md` 与 `CLAUDE.md` 不一致 | 以 `FILE_GRAPH.md` 为准（它是权威），同步 `CLAUDE.md` |
| `FILE_GRAPH.md` 与实际仓库不一致 | 以实际仓库为准（实际才是真），同步图谱 |
| ADR 误放进 `meta/` | 移到 `worklogs/decisions/`（ADR 是决策记录，不属于元信息） |
| 设计文档误放进 `meta/` | 移到 `docs/plan/design/`（设计文档属于"项目计划"） |

### 明确不做

- **不做**：把 §一 目录树拆出去成单独 `TREE.md`（失去上下文）
- **不做**：在 §三 决策树里维护"具体放哪个子目录"（如 `conventions/01-` 还是 `conventions/02-`）——那是规范的编号
- **不做**：在 §二 引用关系里画"功能点 ↔ worklog"的具体线条（粒度太细，维护成本高）
- **不做**：把 `docs/reports/` 拆成 `reports/per-feature/` 子目录（违反 ADR 0005 拆目录阈值 300 行）

## 九、验收标准

| 维度 | 验收方式 | 通过标准 |
|------|---------|---------|
| 决策树覆盖率 | 枚举所有现存文件，逐一对照 §三 13 个分支 | 100% 命中（无文件找不到归位） |
| 目录树准确性 | `tree -L 2` 对照 §一 | 节点完全一致 |
| 引用关系正确性 | 抽查 3 条引用链（如改规范→改 specs） | 全部成立 |
| 与 CLAUDE.md 一致性 | 人工 diff "目录索引"段 vs §一 | 仅详略差异，无结构差异 |
| 决策树分支数 | 静态计数 | = 13 |
| 单文件长度 | `wc -l FILE_GRAPH.md` | ≤ 300 行（ADR 0005 阈值） |

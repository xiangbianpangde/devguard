## 设计.md 章节导航

### §一 模块定位
- **看什么**：元信息 vs 规范 vs 文档库 vs worklog vs ADR 的**职能正交**
- **想什么**：为什么 FILE_GRAPH 不在 `docs/` 下而在 `meta/` 下？——因为 `docs/` 是"项目的文档"，`meta/` 是"仓库自身的描述"

### §二 为什么单文件
- **看什么**：3 个备选方案对比表
- **核心理解**：**目录树是底座，引用关系 + 决策树是衍生视图**——拆开就失去底座
- **易踩坑**：不要因为"§一 §二 §三 内容多"就拆 3 份——三大职能在同一上下文才有价值

### §三 目录结构
- **看什么**：单文件目录 `meta/FILE_GRAPH.md`
- **想什么**：未来会不会扩成多文件？——按 ADR 0005 阈值 300 行，目前 ~170 行，2 年内不太可能突破

### §四 三大职能
- **看什么**：§一 目录树、§二 引用关系、§三 决策树
- **核心理解**：三章不是平行章节，而是**层次关系**——目录树是事实层，引用关系是约束层，决策树是应用层

### §五 13 个文件类型
- **看什么**：13 个分支的"目标路径 + 触发条件"表
- **易踩坑**：第 13 分支"项目级入口"末端有 ⚠️ 警告——根目录只接受 4 类文件，其他一律拒绝

### §六 互为索引
- **看什么**：与 `CLAUDE.md` 的关系
- **核心理解**：**互为索引 ≠ 双向同步**——两边职能不同（AI 上下文 vs 文件归类权威），但内容上必须一致
- **易踩坑**：改了 `FILE_GRAPH.md` §一 但漏改 `CLAUDE.md` "目录索引"段——会导致双轨漂移

### §七 维护规则
- **看什么**：6 种触发场景的必做动作
- **核心理解**：FILE_GRAPH 只维护"现在长什么样"——细节在文件里，历史在 worklog，决策在 ADR

### §八~§十 边界 + 验收 + 未来
- **看什么**：降级行为、明确不做的事、6 维验收标准、未来扩展
- **核心理解**：FILE_GRAPH 是仓库的"宪法"——违反它的"根目录不接受新散文件"等于动摇治理根基

## 实现计划.md 章节导航

### §二~§七 五阶段
- **看什么**：V0.x 启步（新建）→ V0.x 重整（scripts + design 合并）→ V0.x 稳定（v2.0 不动结构）→ V1.x 治理（9 篇规范不动 FILE_GRAPH）→ V2.0 流程（流程 v2.0 不动 FILE_GRAPH）
- **想什么**：为什么 V0.x 之后 3 次大改都不动 FILE_GRAPH 结构？
  - 答：FILE_GRAPH 的"决策树 13 分支"在 V0.x 重整后已**定型**，新增的 9 治理规范 + 3 AI 流程都仍在已有分支覆盖范围内（规范正文 / 元信息 / 流程文档）
- **核心理解**：FILE_GRAPH 描述的是"仓库结构"——只要不增删目录，就不需要更新；V0.x 之后仓库结构稳定，所以 FILE_GRAPH 也稳定

### §九 红线关系
- **看什么**：本模块累计触发的红线类型
- **易踩坑**：本模块的"漏改 CLAUDE.md"红线触发了 ~2 次——互为索引纪律需要靠 review 兜底

## BDD 验收清单

实施时如要新增/修改本模块，必须满足：

- [ ] **BDD 1（决策树覆盖率）**：枚举所有现存文件，逐一对照 §三 13 个分支，100% 命中
- [ ] **BDD 2（目录树准确性）**：`tree -L 2` 对照 §一，节点完全一致
- [ ] **BDD 3（引用关系正确性）**：抽查 3 条引用链（如改规范→改 specs），全部成立
- [ ] **BDD 4（与 CLAUDE.md 一致性）**：人工 diff "目录索引"段 vs §一，仅详略差异、无结构差异
- [ ] **BDD 5（决策树分支数）**：静态计数 = 13
- [ ] **BDD 6（单文件长度）**：`wc -l FILE_GRAPH.md` ≤ 300 行
- [ ] **BDD 7（结构稳定）**：V0.x 重整后 FILE_GRAPH 结构未变（仅在 V0.x 启步 + V0.x 重整时改过）
- [ ] **BDD 8（同步纪律）**：6 种触发场景下必做动作（见设计.md §七）全部执行

## 思考题

1. **为什么 FILE_GRAPH 不在 `docs/` 下而在 `meta/` 下？** —— 因为 `docs/` 是"项目的文档资产"（M2），`meta/` 是"仓库自身的元信息"（M6）；如果 FILE_GRAPH 放进 `docs/`，那新增 FILE_GRAPH 之前要先查 FILE_GRAPH（鸡生蛋问题）。
2. **决策树 13 分支的颗粒度怎么定？** —— 以"现存文件类型"为颗粒度，不多不少；新增一类时再扩（V0.x 重整时从 10+ 扩到 13，是因为新增了 design / scripts / 模板索引三类）。
3. **互为索引的同步纪律怎么自动化？** —— 理论上可以做 pre-commit 钩子（`scripts/check_file_graph.py` 跑 §一 与实际目录树 diff + 与 CLAUDE.md 段 diff），但当前靠人工 review 兜底。
4. **FILE_GRAPH 与 `docs/plan/design/INDEX.md` 的关系？** —— INDEX.md 描述"8 个模块设计文档"的总图（跨模块），FILE_GRAPH 描述"仓库文件结构"（单模块）；两者都属"仓库元信息"层级，**但 INDEX.md 因跨模块放于 `docs/` 而非 `meta/`**，是设计取舍。
5. **未来 FILE_GRAPH 会不会扩成多文件？** —— 按当前扩展速度 + ADR 0005 阈值 300 行，2 年内不太可能；如果未来仓库结构变成"插件化"（多套规范共存），可能需要拆。

## 易踩坑速查

| 症状 | 真因 | 修复 |
|------|------|------|
| 新文件放到了错误目录 | 没查决策树 | 强制纪律：新增前必查 §三 |
| 改 `FILE_GRAPH.md` 后 `CLAUDE.md` 没同步 | 互为索引纪律未落实 | review 时加一个 diff 检查项 |
| 决策树找不到对应分支（新增了一类文件） | 决策树未扩展 | 禁止新增，先在 §三 加分支 |
| 根目录出现了散文件（如临时 .md） | 违反决策树末端警告 | 移到决策树对应分支的目录 |
| FILE_GRAPH 与实际仓库不一致 | 改了实际仓库但忘改 FILE_GRAPH | 强制纪律：移文件后立即更新图谱 |
| 把 ADR 放进 `meta/` | 误把"决策"当"元信息" | 移到 `worklogs/decisions/`（ADR 是决策记录） |
| 把设计文档放进 `meta/` | 误把"项目计划"当"元信息" | 移到 `docs/plan/design/` |

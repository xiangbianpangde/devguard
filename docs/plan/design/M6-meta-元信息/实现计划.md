## 二、阶段 V0.x 启步：FILE_GRAPH.md 新建（2026-05-26）

### 目标

> 详见 `worklogs/2026-05-26_文件图谱与仪表盘优化.md`

把"文件归类"这件事从散落的 CLAUDE.md / README.md / 内部口口相传，固化成一份**唯一权威**的文档。

### 实现路径

```
仓库内散落的目录结构认知
    ↓
新增 meta/FILE_GRAPH.md（三章：目录树 + 引用关系 + 决策树）
    ↓
CLAUDE.md 增加"新增文件前先查 meta/FILE_GRAPH.md"硬规则
    ↓
同步 docs/design/README.md（当时 design 还在 docs/ 下，#28 才合并）
```

### 改动

| 模块 | 改什么 | 状态 |
|------|--------|:---:|
| `meta/FILE_GRAPH.md` | 新建（三章 + 关键约束 4 条） | ✅ |
| `CLAUDE.md` | 增加 FILE_GRAPH 引用 + 目录树补 `meta/` `docs/design/` `scripts/` | ✅ |
| `docs/design/README.md` | 新建（用途 + 五段式设计文档模板） | ✅ |
| `.gitignore` | 补 `.benchmarks/` | ✅ |

### 决策依据

- **新增时机选在 V0.x 启步**：此时仓库结构已稳定（5 大模块成型），是固化"文件归类"的合适窗口
- **三章结构**而非单章"目录树"：根因是 CLAUDE.md 与 README.md 中"目录结构"段都已经有引用关系和决策树的事实需求——既然分散的认知需要整合，索性一份文件覆盖全部
- **决策树 13 分支**（初版 10+ 个，V0.x 重整后增补）：以"现存文件类型"为颗粒度，不多不少

## 三、阶段 V0.x 重整：scripts/ 合并 + 决策树扩展（2026-05-27）

### 目标

> 详见 `worklogs/2026-05-27_文件夹重整与plan-design合并.md`

执行功能点 #27 文件夹重整 + #28 plan/design 合并。`meta/FILE_GRAPH.md` 需同步：
- 目录树补 `scripts/` 节点（`start_server.py` + `打开仪表盘.bat` 从根目录移入）
- 决策树补"工具/辅助脚本 → `scripts/`"分支
- 决策树补"复杂功能设计文档 → `docs/plan/design/`"分支（design 从 `docs/design/` 合并过来）

### 实现路径

```
#27 文件夹重整
  ├─ start_server.py + 打开仪表盘.bat → scripts/
  └─ meta/FILE_GRAPH.md 同步补 scripts/ 节点
       ↓
#28 plan/design 合并
  ├─ plan/ → docs/plan/
  ├─ docs/design/ → docs/plan/design/
  └─ meta/FILE_GRAPH.md 同步修路径 + 决策树补 design 分支
```

### 改动

| 模块 | 改什么 | 状态 |
|------|--------|:---:|
| `meta/FILE_GRAPH.md` | 目录树 + 决策树扩充（+scripts/ +design/） | ✅ |
| `docs/templates/README.md` | +5 模板登记 | ✅ |
| `CLAUDE.md` | 目录结构更新 | ✅ |
| `README.md` | dashboard 路径修正 | ✅ |

### 决策依据

- **决策树 13 分支定型**（V0.x 重整后）：每多一类文件就多一个分支，到 2026-05-27 仓库已稳定的文件类型覆盖完毕
- **同步策略**：每次目录重整必同步 FILE_GRAPH；FILE_GRAPH 与实际仓库的不一致会**严重**破坏它的"权威性"，所以宁可加班同步也不能漏

## 六、阶段 V1.x 治理：9 篇治理规范不动 FILE_GRAPH（2026-06-07）

### 目标

`worklogs/2026-06-07` 期间，把 9 篇治理规范（仪表盘 / 模板 / README / CODEOWNERS / CHANGELOG / SECURITY / SUPPORT / LICENSE / CONTRIBUTING）入 `_meta.yaml`——

### 关键判断

| 评估项 | 结论 |
|--------|------|
| 9 篇治理规范是否需要更新 FILE_GRAPH | 否（落地产物在 `docs/templates/` `dashboard.html` `LICENSE` 等已存在的路径，未新增目录） |
| 是否需要给 FILE_GRAPH 加"治理规范"分支 | 否（治理规范的"该存在性即满足"机制在 `_meta.yaml` 维护，FILE_GRAPH 不重复登记） |

### 决策依据

- **FILE_GRAPH 不重复 M1 的真源** —— `_meta.yaml` 已经在 M1 章节维护，FILE_GRAPH 描述的是"文件放哪"而不是"规范怎么管"
- **职能正交** —— M1 说"代码应该怎么写"，M6 说"文件应该放哪"；M1 治理规范的存在性由 M1 自己保证，M6 不需要为 M1 维护清单

## 一、背景：现在能做什么，缺什么

### 现在能做的（已有基建）

AgentHub 的 IM 聊天核心链路已经跑通：用户能创建会话、发消息、收 Agent 流式回复、看代码高亮和 Diff。左侧面板有会话列表（私聊 + 群聊分 Tab），右侧是聊天窗口，消息气泡支持 Pin、复制代码、Diff 内联预览、网页预览卡片。

### 缺什么（四个功能点）

| 功能点 | 现在 | 目标 |
|--------|------|------|
| **私聊入口** | 新用户打开左侧面板 → 空白页 + "去 AI 队友里发起"纯文字，无可点击入口 | 启动时自动加载已有会话 + 空态显示「发起私聊」按钮 |
| **会话归档** | 几十个会话堆在一起，没有归档能力 | 右键 → 归档 → 移入「已归档」折叠分区 |
| **消息操作验证** | 按钮代码都写了但没在真浏览器里跑过完整链路 | Playwright 截图：发消息 → 回复/复制/Pin/Diff → 逐个断言 |
| **对话式建 Agent** | 只能 3 步表单向导（选模板→配 CLI→填信息） | 输入自然语言描述 → LLM 抽取 → 预览 → 一键创建 |

## 二、用户故事

### 功能点 1：私聊入口修复

**用户故事**：小陈第一次打开 AgentHub，左侧面板空空如也，只显示"去 AI 队友里发起"几个字。他不知道"AI 队友"在哪，也不知道怎么开始第一次对话。

**根本原因**：左侧面板的私聊列表只显示用户在当前会话中手动创建的会话。后端数据库里已经存在的 Session（比如之前创建的、或者别人分享的）从不加载到左侧面板。空态时也没有任何可点击的按钮。

**交互流程**：

```
用户打开 AgentHub
    │
    ▼
系统启动时自动调用后端 → 拿到所有属于当前用户的私聊 Session
    │
    ▼
把这些 Session 加载到左侧面板的会话列表里
    │
    ├── 有会话 → 正常显示列表，点击任意一条进聊天
    │
    └── 没有会话 → 显示「发起私聊」按钮
                      │
                      ▼
                  点击 → 弹出选择 Agent 的对话框
                      │
                      ▼
                  选一个 Agent → 创建会话 → 进入聊天
```

**为什么按钮"不响应"是误判**：之前有人反馈建议按钮点了没反应。调查发现按钮代码本身没问题——它只有在已经有活跃会话时才工作。真正的坑是：新用户根本没有活跃会话，按钮根本不会出现。问题不在按钮逻辑，在会话加载。

---

### 功能点 2：会话归档

**用户故事**：小李用 AgentHub 两周了，左侧面板堆了 50+ 个会话。有些是上个月的项目早就不用了，但和活跃会话混在一起。他想把不用的归档掉，但发现根本没有归档功能。

**交互流程**：

```
用户右键某个会话
    │
    ▼
弹出菜单 → 点击「归档」
    │
    ▼
该会话从主列表消失
    │
    ▼
左侧面板底部出现「已归档（N）」折叠分区
    │
    ▼
点击展开 → 看到所有已归档的会话
    │
    ▼
右键 → 「取消归档」→ 回到主列表
```

**数据流向**：归档不是纯前端行为——它需要持久化到数据库，否则刷新页面归档状态就丢了。这是一个从前端到数据库的完整链路：前端 toggle → API PATCH → 后端 Service → 数据库更新。

**搜索说明**：搜索功能前端已经有了（输入关键词即时过滤），但目前只搜索本地 store 里的会话名和 Agent 名。后端其实也有按标题搜索的接口，只是前端没接。本期先保持本地搜索——对大多数用户够用。如果会话量特别大再改走后端。

---

### 功能点 3：消息操作端到端验证

**用户故事**：开发团队已经实现了消息气泡的全部操作按钮（回复、复制代码、Pin、Diff 预览、网页预览），也有单元测试。但从来没在真实浏览器里，连着真实后端和真实 Agent 跑过一条完整链路——用户能不能正常回复消息？复制代码按钮真的能复制吗？Diff 渲染对不对？

**交互流程**（验证而非新功能）：

```
Playwright 自动执行：
    起后端 + 前端 → 创建会话 → 发一条包含代码块 + diff + URL 的消息
        │
        ▼
    Agent 返回消息后，在消息气泡上逐一验证：
        ├── 点「回复」→ 输入框出现引用
        ├── 点「复制代码」→ 剪贴板有内容
        ├── 点「Pin」→ 消息被固定
        ├── Diff 区域正确渲染绿色（新增行）和红色（删除行）
        ├── 网页预览卡片能展开全屏
        └── 「重新生成」按钮是禁用态（后端还没做这个功能）
    全流程截图存档
```

### 功能点 4：对话式创建 Agent

**用户故事**：小王想建一个"前端代码审查 Agent"，但他不想填 3 步表单。他想像聊天一样输入"帮我建一个 Agent，专门审查 React 代码，关注性能和安全"，系统自动理解他的需求并生成 Agent。

**交互流程**：

```
用户在创建 Agent 页面选「对话式创建」（新增入口）
    │
    ▼
进入聊天式界面（和普通聊天一样的体验）
    │
    ▼
用户说："帮我建一个前端代码审查 Agent，关注 React 性能和安全"
    │
    ▼
系统调 LLM 抽取 → 返回结构化草稿：
    { name: "前端代码审查助手",
      role: "代码审查",
      system_prompt: "你是资深前端审查专家，重点关注 React 性能优化和…",
      skills: ["React", "性能分析", "安全审计"] }
    │
    ▼
前端展示草稿卡片："这是根据你的描述生成的 Agent 配置，确认创建吗？"
    │
    ├── 确认 → 调 POST /api/agents 创建 → 跳转到新 Agent 详情页
    │
    └── 不满意 → 继续聊天调整："再加一个 TypeScript 类型检查"

```

**和传统方式对比**：

| | 表单向导 | 对话式创建 |
|------|---------|-----------|
| 操作 | 3 步：选模板 → 配 CLI → 填信息 | 一句话描述 → 确认 |
| 门槛 | 需要知道 system_prompt / skills / role 的含义 | 用自然语言描述需求 |
| 迭代 | 返回去改表单字段 | 继续聊天调整 |
| 时间 | ~3-5 分钟 | ~30 秒 |

## 三、架构决策

### 决策 1：私聊死路修 Session 回灌而不是改按钮

**选择**：在 store 里加一个 `hydrateFromSessions` 方法，App 启动时调一次后端 list 接口，把所有已有 Session 加载到前端。

**原因**：按钮逻辑没问题——它只在有活跃会话时才工作。真坑是"没有会话可用"。回灌解决的是数据源头问题。

**代价**：启动时多一次 API 调用；回灌的 Session 和本地新建的 Session 需要以 agent_id 去重。

---

### 决策 2：归档走全链路 DB 持久化而不是纯前端

**选择**：在 Session 实体上加 `archived` 字段，沿 domain → model → migration → command → service → API → 前端 完整链路实现。

**原因**：归档是持久行为——用户期望刷新页面后归档状态还在。纯前端做（比如 localStorage）会在清缓存后丢失。

**代价**：需要一条 alembic migration（涉及多 worktree stamp 冲突风险）；改动面大（~9 个文件）。

---

### 决策 3：E2E 不改源码只补测试

**选择**：消息操作 E2E 只新建一个 Playwright spec 文件，不动任何源码。

**原因**：MessageBubble 的所有操作按钮代码已经完整（回复/Pin/复制/Diff/网页预览），缺的只是端到端证据。加功能 = 加风险；补测试 = 降低风险。

**代价**：依赖真实后端和 Agent 能跑起来——如果环境挂了 E2E 跑不了，但这不是代码问题。

---

### 决策 4：对话式创建走 LLM 抽取而不是规则模板

**选择**：用户自然语言 → 调 LLM 抽取结构化 JSON → 预览 → 确认创建。

**原因**：规则模板（比如关键词匹配"前端"→role="前端开发"）太僵硬，用户稍微换种说法就匹配不上。LLM 抽取灵活但依赖 LLM 可用性。

**代价**：需要 LLM 凭证（复用现有协调者凭证）；需要 PR-01 评审新端点。

## 四、数据流

### 功能点 1：Session 回灌

```
App 启动
    │
    ▼
前端调 GET /api/sessions?type=private
    │
    ▼
后端返回 SessionOut[]（含 agent_id / title / updated_at）
    │
    ▼
chatStore.hydrateFromSessions(sessions)
    遍历每个 session，映射成 conversation 对象
    以 agent_id 为 key 存入 store，避免和本地新建的重复
    同时填充 sessionIds 映射表（session_id → agent_id）
    │
    ▼
左侧面板重新渲染 → 会话列表可见
```

### 功能点 2：归档

```
用户点击归档按钮
    │
    ▼
前端调 PATCH /api/sessions/{id}  { archived: true }
    │
    ▼
后端 session_service.update() → repository 更新 DB
    │
    ▼
前端 chatStore.setConversationArchived(id, true)
    │
    ▼
左侧面板从主列表移除该会话 → 移入「已归档」分区
```

### 功能点 4：对话式创建 Agent

```
用户输入自然语言描述
    │
    ▼
前端 POST /api/agents/draft-from-chat  { description: "…" }
    │
    ▼
后端调 LLM（复用协调者凭证）→ 抽取 JSON
    │
    ▼
返回 { name, role, system_prompt, skills }
    │
    ▼
前端渲染草稿卡片 → 用户确认
    │
    ▼
调 POST /api/agents → 创建 Agent → 跳转详情页
```

## 六、边界与降级

| 场景 | 降级行为 |
|------|---------|
| Session 回灌和本地新建重复 | 以 agent_id 去重，本地优先 |
| 归档 migration 遇到 alembic stamp 冲突 | 手动修复 stamp 再 upgrade |
| 功能点 4 的 LLM 不可用 | mock LLM 返回固定草稿，前端链路先跑通 |
| 功能点 4 的 PR-01 未通过 | 对话式创建不实现，回退到仅有表单向导 |
| E2E 环境挂了 | 跳过 E2E 但保留 spec 文件，环境恢复后再跑 |

### 明确不做

- **不做**「重新生成」消息的真实现（后端无此端点，按钮暂 disabled）
- **不做**搜索强制改走后端（本地搜索够用）
- **不做** LLM 凭证体系新建（功能点 4 复用现有协调者凭证）

## 七、验收标准

| 功能点 | 验收方式 | 通过标准 |
|--------|---------|---------|
| 私聊入口修复 | 浏览器手动测试 + vitest | 新用户打开 → 看到「发起私聊」按钮 → 点击 → 选 Agent → 进聊天 |
| 会话归档 | 浏览器手动测试 + pytest | 右键归档 → 从主列表消失 → 「已归档」分区出现 → 刷新页面状态还在 |
| 消息操作 E2E | **Playwright 浏览器截图** | 发消息 → 回复/复制/Pin/Diff/网页预览 → 每个操作截图通过 |
| 对话式创建 Agent | pytest(mock LLM) + 浏览器截图 | 输入描述 → LLM 抽取 → 草稿预览 → 确认创建 → 跳转详情 |

## 八、与其他模块的关系

- **依赖**：已有的 Session CRUD、WebSocket 通道、Agent 管理、协调者 LLM 凭证
- **被依赖**：私聊入口修复后，M2（任务编排）的群聊列表也能受益于回灌逻辑
- **并行**：四项功能点互不阻塞（功能点 4 的阻塞不影响 P0/P1）
- **无关**：与 M3（产物预览）、M4（部署桌面）无契约冲突

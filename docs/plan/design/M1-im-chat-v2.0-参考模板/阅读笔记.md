## 设计.md 章节导航

### §一 背景
- **看什么**：四列"现在 vs 目标"对比
- **想什么**：为什么「私聊死路」优先级最高？—— 它影响的是新用户的第一印象，门都进不来

### §二 用户故事
- **看什么**：四个完整的交互流程，每个都有流程图
- **核心理解**：功能点 1 的真因 = 后端 Session 没回灌前端 store，不是按钮有问题
- **易踩坑**：功能点 2 的归档不是纯前端操作——必须持久化到 DB，走 5 层全链路

### §三 架构决策
- **看什么**：4 个决策的选择/原因/代价
- **核心理解**：决策 1（回灌 vs 改按钮）是整个模块最关键的设计判断——修源头而非修症状
- **易踩坑**：决策 4（LLM 抽取 vs 规则模板）的代价是 LLM 依赖，排最后是有原因的

### §四 数据流
- **看什么**：三条链路图
- **想什么**：归档链路最长（前端→API→Service→Repository→DB→返回→前端更新），哪个环节漏了都会断

### §五 组件关系
- **看什么**：LeftPanel / ChatView / CreateAgentModal 三块各自的改动
- **想什么**：ChatView 在本模块完全不动——功能点 1 修的是 LeftPanel + chatStore

### §六 边界与降级
- **看什么**：5 个降级场景
- **想什么**：功能点 4 有两条降级路径——LLM 不可用和 PR-01 不过，两条都能安全降级

## 思考题

1. **为什么私聊死路之前被误判为"按钮不响应"？** —— 因为测试者只在"已经有活跃会话"的环境下测，没从新用户视角打开。这说明：**测试要覆盖零状态场景**。
2. **归档要不要做成"软删除"？** —— 当前设计是 `archived` 布尔字段。如果未来需要"永久删除会话"，要不要复用同一套？还是要分开？
3. **对话式创建如果 LLM 抽取不准怎么办？** —— 多轮对话调整的设计已预留，但 LLM 抽取的质量边界在哪里？什么情况下应该 fallback 到表单向导？
